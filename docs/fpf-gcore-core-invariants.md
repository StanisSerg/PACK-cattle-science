# G.Core — Core Invariants Part G: фундамент стабильного Pack

## 1. Зачем это читать
Представьте, что вы собрали лучшие в мире исследования по переходному периоду коров в один Pack. Через полгода выходит новая статья, и вам нужно понять: пересматривать ли выводы? Менять ли рекомендации? А что, если вы переименовали раздел — и сломали все ссылки? Или добавили штраф за низкое качество исследования, но он случайно повлиял не только на доверие, но и на сами факты?

G.Core — это **скелет**, на котором держится вся Part G FPF. Он не говорит, как лечить кетоз. Он говорит: «Если вы строите SoTA Pack, вот правила, без которых он развалится при первом обновлении». Это инварианты, а не методы. Нарушить их можно только молча — и платить потом неделями аудита.

Если вы планируете создавать, поддерживать или аудировать Pack для PACK-cattle-science — этот паттерн обязателен к прочтению.

## 2. История одной ошибки
**Ферма «Молочный берег», 2023 год.** Зоотехник Михаил собрал «Протокол кормления сухостойных v1.0» — сравнил 8 исследований, выбрал 3 лучших, сделал рекомендации. Pack лег на полку.

В 2024 году вышло новое исследование из Висконсина. Михаил добавил его в Pack — но забыл проверить, что единицы измерения энергии в нём (NEL vs. ME) отличаются от остальных. Система автоматически «свернула» разнородные данные в одно число. Рекомендация изменилась. Коровы получили меньше энергии.

В 2025 году Михаил уволился. Новый зоотехник не понял, почему в Pack используется старая версия CN-Spec. Он обновил ссылку — и половина сравнений стала нелегальной, потому что новая версия изменила правила comparability. Pack не сломался явно. Он просто начал врать молча.

**Что пошло не так:**
1. Пересечение контекстов (NEL vs. ME) было неявным — никто не зафиксировал CrossingBundle.
2. Частичный порядок (набор исследований разного качества) «свернули» в скаляр (одно число «средней надёжности»).
3. Штраф за сомнительное исследование повлиял на F/G-инварианты, хотя должен был идти только в R_eff.
4. Обновление ссылки на CN-Spec не вызвало RSCR-триггер — потому что не было каталога типовых причин.

G.Core создан, чтобы такого не происходило.

## 3. Core Invariants Part G: фундамент стабильного Pack — полное описание
### 3.1 Пересечения только через Bridge (Crossing Visibility)
**Что это:** Любое движение знания из одного контекста в другой должно быть видимым и зафиксированным.

**Аналогия:** Перевозка быков через областную границу. Ветеринарное свидетельство, карантин, отметка в паспорте — всё это видимо. Вы не можете просто «перегнать быка» — должна быть бумага. В FPF эта «бумага» — CrossingBundle.

**На ферме:** Вы сравниваете исследование из Нидерландов (пастбищное содержание) с вашей фермой (стойловое). Это crossing. Без explicit Bridge вы делаете вид, что условия одинаковые. CrossingBundle фиксирует: «при переводе учтены потери X, Y, Z».

**Инвариант G.Core:** Если CrossingBundle отсутствует или некорректен — downstream consumers MUST abstain (воздержаться). Никаких молчаливых переходов.

### 3.2 Три состояния защиты: pass | degrade | abstain
**Что это:** Решение о пригодности данных имеет ровно три состояния. «Не знаю» не превращается автоматически в «годится».

**Аналогия:** Ветеринарный осмотр перед запуском в стадо.
- **pass** — здоров, запускаем.
- **degrade** — подозрение, пускаем в карантинный сектор с ограничениями.
- **abstain** — недостаточно данных, не пускаем, но и не лечим.

Ошибка: «Я не знаю, больна ли корова» → «Пусть идёт, раз не подтверждено». G.Core запрещает такое: unknown никогда не coerces to pass.

**На ферме:** Исследование без декларации конфликта интересов — это не «нормальное, просто без пометки». Это unknown. И оно не может получить pass без явного решения политики.

### 3.3 Штрафы идут только в R_eff (Penalty Routing)
**Что это:** Когда данные штрафуются за качество, наказание влияет только на доверие (R-полоса), но не меняет сами факты (F) и не нарушает структурные правила (G).

**Аналогия:** На аукционе быков конкретный бык получил штраф за плохое родословное (R_eff падает). Но это не меняет его вес (F) и не отменяет правила взвешивания (G).

**На ферме:** Исследование с малой выборкой получает штраф за статистическую мощь. Это уменьшает вклад в итоговое доверие к рекомендации — но не искажает сырые цифры выхода молока. Если бы штраф менял сырые данные, Pack стал бы ложью.

### 3.4 Множества остаются множествами (Set-Return Semantics)
**Что это:** Частичный порядок (когда A лучше B по одному критерию, но хуже по другому) нельзя тихо «свернуть» в одно число.

**Аналогия:** У вас три корма:
- Корм A: лучше по цене, хуже по переваримости.
- Корм B: лучше по переваримости, дороже.
- Корм C: посередине по обоим.

Нельзя просто сказать «средний рейтинг: B = 8, A = 7, C = 7.5» — и выбрать B. Это silent scalarisation. Пользователь должен видеть **множество лучших вариантов** (Pareto-front), а не один «победитель».

**На ферме:** Выбирая протокол лечения метритов, вы не должны получать один «лучший» — вы должны получать **shortlist** с пояснением: «A лучше по скорости, B лучше по стоимости, C — компромисс». G.Core требует: набор кандидатов возвращается как множество.

### 3.5 Типизированные RSCR-триггеры (Typed RSCR Triggers)
**Что это:** Причины, по которым Pack нуждается в пересмотре, записываются не произвольным текстом, а из строгого каталога.

**Аналогия:** В ветеринарном журнале не пишут «что-то изменилось». Пишут: «вакцинация» (код VAC), «осеменение» (код INS), «клинический случай» (код CLI). Коды известны всем, понятны машине, не двусмысленны.

**На ферме:** Вышла новая редакция CN-Spec — это `RSCRTriggerKindId.EditionPinChange`. Изменилась политика штрафов — это `RSCRTriggerKindId.PenaltyPolicyEdit`. Появилось новое исследование с сомнительной методологией — это `RSCRTriggerKindId.EvidenceSurfaceEdit`. Без такой типизации вы не знаете, какие Packs проверять при обновлении.

### 3.6 Одна управляющая дефиниция на Default (Default Governing Definition Index)
**Что это:** Любое значение по умолчанию имеет ровно одно каноническое определение. Нельзя иметь два разных «стандарта по умолчанию» для одного и того же.

**Аналогия:** На ферме есть «стандартная доза пропиленгликоля». Не может быть двух разных «стандартных доз» в разных протоколах без явного указания, какая из них управляющая. Если одна изменилась — другая автоматически становится deprecated или получает новый ID.

**На ферме:** `DefaultId.GammaFoldForR_eff` — это способ сворачивать доверие при наличии штрафов. У него ровно одна управляющая дефиниция (`CC-G5.4`). Если кто-то в локальном Pack попытается «переопределить умолчание» — G.Core блокирует, требуя явного override с proof_refs.

## 4. Почему смешивать / игнорировать — значит рисковать
| Anti-pattern | Проявление на ферме | Почему опасно |
|---|---|---|
| **Shadow Spec** | В локальном Pack переписали правила comparability из CN-Spec «для удобства» | Локальная версия расходится с канонической; при аудите невозможно понять, по каким правилам сравнивали |
| **Silent Crossing** | Исследование из пастбищной системы применили к стойловой без декларации потерь | Рекомендации становятся нелегальными; результаты не воспроизводятся |
| **Hidden Fourth Status** | Система позволяет «условный pass» в дополнение к pass/degrade/abstain | Три-state guard размывается; unknown тихо превращается в решение |
| **Scalarisation of Partial Order** | Из 5 кормов с несравнимыми trade-off выбрали одного «победителя» по взвешенной сумме | Пользователь не видит компромиссов; выбор необоснован |
| **Penalty Leakage** | Штраф за качество исследования изменил сырые данные (F-полосу) | Pack врёт о фактах; доверие падает, но незаметно |
| **Missing Trigger Catalogue** | При обновлении CN-Spec никто не знает, какие Packs затронуты | Устаревшие Packs продолжают использоваться как актуальные |
| **Divergent Defaults** | В двух разных Packs «стандартная доза» означает разное | Медицинская ошибка при переходе между протоколами |

## 5. Как это выглядит на ферме: полный сценарий
**Задача:** Создать Pack «Протоколы профилактики субклинического кетоза (SCK)» для фермы с 500 голов.

**Шаг 1. Scope anchor (M1)**
- `describedEntity := ⟨SCK_prevention, ReferencePlane=world⟩`
- `CGSpecRef.edition = 2024.3`
- `CNSpecRef.edition = 2023.7`
- `ReferenceMap`: термины (NEFA, BHB, пропиленгликоль, монензин) привязаны к UTS-идентификаторам.

**Шаг 2. Harvesting (M2)**
- Собираем SoTA_Set: 12 исследований из 4 стран.
- CrossingBundle для каждого исследования: пастбищное vs. стойловое, разные породы, разные стадии лактации.
- Если CrossingBundle невозможен (данные совершенно несовместимы) — abstain, исследование не входит в ComparatorSet.

**Шаг 3. Variant pool (M3)**
- 5 протоколов-кандидатов: пропиленгликоль, монензин, холин, гликерол, комбинация.
- Каждый кандидат имеет `EmitterPolicyRef` и `provenance anchors`.
- Tri-state guard: два исследования получили `degrade` (малая выборка), одно — `abstain` (нет данных по Holstein в стойле).

**Шаг 4. Shortlist (M4)**
- Set-return: в shortlist попадают 3 кандидата (не 1 «победитель»).
- Penalties применены только к R_eff: доверие к «монензину» снижено из-за малой выборки, но сами данные о снижении BHB не искажены.

**Шаг 5. Library + Refresh readiness (M5-M6)**
- `CGFrameLibraryId` публикуется в UTS с Name Cards.
- `RefreshReadinessCardId` регистрирует: «следить за `RSCRTriggerKindId.EditionPinChange` в CGSpecRef и `RSCRTriggerKindId.EvidenceSurfaceEdit` в SoTA_Set».
- При выходе новой версии CG-Spec автоматически известно, что этот Pack нужно пересмотреть.

## 6. Практическое применение: с чего начать
**Шаг 1. Зафиксируй scope anchor.**
Для одного CG-Frame запиши: что описываем (describedEntity), в каком мире (ReferencePlane), какие спеки управляют (CGSpecRef/CNSpecRef edition). Без этого всё остальное — песок.

**Шаг 2. Создай CrossingBundle для каждого «чужого» исследования.**
Если в Pack попадает исследование не с твоей фермы — запиши: что отличается (порода, климат, кормление, управление), и какие потери при переносе. Если не можешь — не переноси.

**Шаг 3. Внедри три-state guard в оценку исследований.**
Для каждого исследования в Pack: pass / degrade / abstain. «Не знаю» = abstain. Запрещено превращать abstain в pass «для полноты».

**Шаг 4. Проверь penalty routing.**
Если исследование штрафуется — убедись, что штраф пошёл только в R_eff (доверие). Сырые цифры (F) и структурные правила (G) не изменились.

**Шаг 5. Замени «рейтинги» на shortlists.**
Если у тебя есть 4 корма с разными trade-off — не выдавай «лучший». Выдавай shortlist с объяснением, почему каждый в нём.

**Шаг 6. Заведи каталог RSCR-триггеров.**
Для каждого Pack запиши: какие события (типизированные!) должны вызывать пересмотр. Привязывай к `RSCRTriggerKindId`, а не к тексту «обновить при необходимости».

## 7. Проверь себя
| Вопрос | Если ответ «не знаю» — проблема |
|---|---|
| Можете ли вы за 5 минут назвать edition pins всех спеков, на которые опирается ваш главный Pack? | Shadow spec / отсутствие governance |
| Есть ли в вашем Pack исследования без явного CrossingBundle? | Silent crossing — нелегальные сравнения |
| Используете ли вы где-либо «условный pass» помимо pass/degrade/abstain? | Hidden fourth status — guard размыт |
| При штрафе за качество исследования меняются ли сырые данные? | Penalty leakage — F-полоса скомпрометирована |
| Выдаёт ли ваша система один «лучший» вариант там, где trade-off несравнимы? | Silent scalarisation — пользователь обманут |
| Есть ли каталог типизированных причин для пересмотра Pack? | Отсутствие RSCR-дисциплины |
| Каждое ли значение по умолчанию имеет одну каноническую дефиницию? | Divergent defaults — риск медицинской ошибки |
