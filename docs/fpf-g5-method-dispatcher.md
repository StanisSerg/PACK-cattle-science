# G.5 — Multi-Method Dispatcher & MethodFamily Registry

## 1. Зачем это читать

Если вы когда-нибудь спрашивали: *«Какой метод профилактики кетоза самый лучший?»* — вы задали неправильный вопрос. FPF говорит: **нет одного лучшего метода**. Есть набор допустимых методов, и диспетчер должен вернуть *короткий список* (`Shortlist`) или *ранжированный короткий список* (`RankedShortlist`), а не навязать один «победитель».

Этот паттерн учит, как управлять семействами методов, которые конкурируют за одну и ту же задачу, не смешивая их в одну кучу и не прячая альтернативы за «единственно правильным» ответом.

---

## 2. История одной ошибки

Ферма «Зелёная долина» столкнулась с вспышкой субклинического гипокальциемия. Консультант провёл анализ и объявил: *«Лучшее решение — DCAD (диетическая катион-анионная разница). Внедряйте».*

Ферма внедрила. Через год: кальций улучшился, но кетоз вырос. Почему? Потому что DCAD хороша для кальция, но сдвигает метаболизм в сторону кетогенеза. Для этой фермы, где кетоз был скрытой проблемой, DCAD оказалась не «лучшей», а **частично доминируемой** — она побеждала по одному критерию и проигрывала по другому.

Если бы диспетчер вернул `Shortlist` из пяти методов (DCAD, пропиленгликоль, мониторинг, болюсы, генетический отбор) с указанием, кто по какому критерию доминирует, ферма могла бы выбрать **сочетание** или хотя бы понять компромисс. Вместо этого она получила один «победитель» и скрытые потери.

## 3. Core concepts

### 3.1 MethodFamily Registry — реестр, а не рейтинг

`MethodFamily Registry` — это не таблица «метод vs балл». Это **аудируемый реестр** семейств методов, каждое из которых зарегистрировано как отдельная, версионированная запись со своим контекстом и традицией.

**На ферме:** пять семейств профилактики кетоза — это пять записей в реестре. У каждой есть:
- `MethodFamilyId` (стабильный идентификатор)
- `EligibilityStandardRef` (когда метод допустим)
- `AssuranceProfileRef` (какие доказательства требуются)
- `LegalityBindings` (какие нормативные ограничения применимы)
- `EvidencePins` (ссылки на `PathId` из G.6)

Реестр **не сравнивает** методы. Он делает их **диспетчируемыми**.

### 3.2 TaskSignature — сигнатура задачи, не описание метода

`TaskSignature` — это минимальная типизированная запись о задаче, которую нужно решить. Например: *«Профилактика кетоза в переходный период у высокопродуктивных коров, 2+ лактация, ферма без полноценного ветеринара»*.

**Ключевой момент:** `TaskSignature` не говорит, каким методом решать. Она говорит, **что** решать и **в каких условиях**. Диспетчер сопоставляет сигнатуру с реестром и выбирает допустимые семейства.

### 3.3 Selector — ядро выбора, не ядро догмы

Селектор потребляет `TaskSignature` + записи реестра + закреплённые спецификации. Он применяет фильтры пригодности (tri-state: допустим / деградирован / воздержаться) и вычисляет **частичный порядок**.

**Что селектор возвращает:**
- `Shortlist` — неупорядоченный набор допустимых кандидатов
- `RankedShortlist` — упорядоченный набор, когда порядок значим
- `SpecialistHandoff` — гетерогенный пакет для разных специализаций
- `abstain` — ни один метод не прошёл порог
- `escalation` — требуется эскалация

**Что селектор НЕ возвращает:** одного скалярного победителя, если частичный порядок не позволяет его назначить.

### 3.4 Set-return semantics — возврат множества, не скаляра

Это сердце G.5. Если два метода **несравнимы** по Парето (один лучше по кальцию, другой по кетозу), селектор обязан вернуть **оба** в `Shortlist`, а не выбрать одного по скрытому весу.

**Аналогия:** вы выбираете машину. Одна быстрее, другая экономичнее. Если вы скажете «лучшая — быстрая», вы скроете потребность в экономичности. `Shortlist` говорит: «вот две несравнимые альтернативы, выбирайте осознанно».

**На ферме:** DCAD и пропиленгликоль — несравнимы по умолчанию. DCAD лучше по кальцию, пропиленгликоль — по кетозу. Вернуть одного «победителя» — значит солгать.

### 3.5 Composition & fallbacks — композиция, не монополия

G.5 позволяет публиковать **шаблоны композиции**: предобработка → решатель → верификатор; каскады; мета-селекторы. Но семантика конкретных методов остаётся в MethodFamily, а G.5 несёт только шаблон композиции.

**На ферме:** шаблон «мониторинг (BHB) → если >1.2, то пропиленгликоль → если нет эффекта через 48ч, то ветеринар» — это композиция трёх MethodFamily. G.5 управляет шаблоном, не переопределяя сами методы.

### 3.6 Publication & telemetry — публикация результата

Результат селектора публикуется с явными пинами:
- `DRR` (Decision Rationale Record) — почему так решено
- `SCR` (Support/Confidence Record) — на чём основано
- `ShortlistId` — публичная идентичность результата
- Телеметрические пины для обновления (RSCR)

## 4. Антипаттерны

| Антипаттерн | Почему опасно | Как выглядит на ферме |
|---|---|---|
| **«Один победитель» (Scalar Winner)** | Скрывает частичный порядок, компромиссы и риски. Принимающий решение не видит альтернатив. | «DCAD — лучший метод, внедряем DCAD». Кетоз растёт, но никто не знал, что есть альтернатива. |
| **«Скрытый компаратор» (Silent Comparator)** | Веса критериев не опубликованы, их нельзя проверить. | Селектор выбирал по «экономике», но вес стоимости корма был в 3 раза выше веса здоровья коровы — без объявления. |
| **«Догматическое ядро» (Algorithmic Dogma)** | Алгоритм выбора зашит в ядро и не может быть проверен или заменён. | Программа всегда выбирает метод с минимальной стоимостью, даже если стадо гибнет. |
| **«Пробник без политики» (Exploration Leak)** | Исследование альтернатив становится скрытым четвёртым статусом, а не явной политикой. | «Мы иногда пробуем новое» — без регистрации, без аудита, без учёта риска. |
| **«Упрятывание в C.11/C.19» (Hidden Publication)** | Результат выбора прячется внутри локального выбора или пула, и неясно, что это публикация. | Решение принято на совете, но в системе оно выглядит как «просто запись», не как `Shortlist`. |

## 5. Пример на ферме: выбор метода профилактики кетоза

**Задача (TaskSignature):**
- Сущность: коровы переходного периода, 2+ лактация
- Условия: высокая продуктивность (>45 л/день), нет полноценного ветлаборатория
- Цель: снизить частоту клинического и субклинического кетоза

**Реестр MethodFamily (5 записей):**

| ID | Семейство | Eligibility | Ключевой критерий |
|---|---|---|---|
| MF-001 | DCAD | Требует анализ корма | Кальций ↑, но кетоз ~ |
| MF-002 | Пропиленгликоль | Нужен доступ к препарату | Кетоз ↓, кальций ~ |
| MF-003 | Мониторинг BHB | Нужен тестер или лаборатория | Раннее обнаружение |
| MF-004 | Болюсы Ca/Mg | Нужен ветеринар для введения | Быстрый Ca, но временный |
| MF-005 | Генетический отбор | Горизонт 5+ лет | Долгосрочное снижение риска |

**Селектор в действии:**

1. **Eligibility gating:** MF-003 отсеивается (нет лаборатории). MF-005 отсеивается (горизонт не подходит). Остаются MF-001, MF-002, MF-004.
2. **Assurance gating:** MF-004 требует ветеринара — роль не назначена → `degrade` (допустимо при условии назначения ветеринара).
3. **Partial order:** MF-001 доминирует по кальцию. MF-002 доминирует по кетозу. MF-004 быстр, но временен. Ни один не доминирует полностью.
4. **Результат:** `RankedShortlist = [MF-002 (по кетозу), MF-001 (по кальцию), MF-004 (условно)]` с `DRR`, объясняющим, почему MF-002 ранг 1 для данной TaskSignature.

**Что произошло бы при «одном победителе»:**
- Если бы селектор вернул только MF-001, ферма потеряла бы контроль над кетозом.
- Если бы только MF-002, ферма рисковала бы гипокальциемией.

**Правильный ответ G.5:** «Вот три несравнимых метода. Ваш приоритет — кетоз? Берите MF-002. Приоритет — кальций? Берите MF-001. Нужен быстрый эффект сегодня и ветеринар завтра? MF-004 при условии».

## 6. Практика

**Шаг 1. Опишите TaskSignature.**
Возьмите одну задачу (например, профилактику мастита). Запишите: цель, ограничения, ресурсы, контекст. Не пишите метод — только задачу.

**Шаг 2. Перечислите MethodFamily.**
Найдите все методы, которые теоретически могут решить эту задачу. Не отбрасывайте «странные» — они пойдут в реестр и будут отфильтрованы селектором, а не вами заранее.

**Шаг 3. Зарегистрируйте в реестре.**
Для каждого метода запишите: `MethodFamilyId`, когда он допустим (`Eligibility`), какие доказательства нужны (`Assurance`), какие ограничения (`Legality`). Если метод требует пересечения контекста — добавьте `CrossingAllowance`.

**Шаг 4. Запустите селектор честно.**
Примените eligibility/assurance gating. Если остались несравнимые методы — не форсируйте скалярный победитель. Верните `Shortlist` или `RankedShortlist` с `DRR`.

**Шаг 5. Опубликуйте результат.**
Результат должен содержать: вид исхода (`Shortlist`/`RankedShortlist`/`SpecialistHandoff`/`abstain`), список допустимых членов, статус упорядочивания, базисные пины (`PathId`, `UTSRowId`).

## 7. Проверь себя

| Вопрос | Если ответ «нет» — проблема |
|---|---|
| Можете ли вы назвать 3–5 допустимых метода для вашей текущей задачи, не называя «лучший»? | Scalar Winner |
| Публикует ли ваш селектор `Shortlist`, или он скрывает выбор внутри отчёта? | Hidden Publication |
| Известны ли веса критериев выбора всем заинтересованным? | Silent Comparator |
| Есть ли у вас записи, почему конкретный метод был исключён? | Missing DRR |
| Можете ли вы заменить алгоритм селектора без переписывания протоколов методов? | Algorithmic Dogma |

## 8. Связи

- **G.Core** — несёт универсальные инварианты Part-G: tri-state guard, penalties→R_eff, P2W split, типизированные RSCR-триггеры.
- **G.2 (SoTA Synthesis)** — поставщик семейств методов для регистрации.
- **G.3/G.4 (CAL)** — типизация слотов/шкал и acceptance clauses, которые использует селектор для gating.
- **G.6 (Evidence Graph)** — источник `PathId/PathSliceId` для evidence pins в registry.
- **C.11 (Local Choice)** — G.5 начинается там, где результат становится публикацией выбранного набора, а не локальным выбором.
- **C.19 (Pool Policy)** — постура пула; G.5 публикует её, не переопределяя.
- **C.23 (SoS-LOG)** — правила tri-state решений, которые селектор обязан соблюдать.
