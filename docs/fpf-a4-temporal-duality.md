# A.4 — Temporal Duality: проектирование ≠ работа

## 1. Зачем это читать
Если вы путаете чертёж с построенным объектом, план с фактом, протокол с инъекцией — вы смешиваете два временных мира. Чертёж насоса не качает воду. Протокол лечения не лечит корову. План обработки — не сама обработка.

A.4 делает это различение явным и даёт механизм бесконечного улучшения.

**FPF-тезис:** *«Чертежи мечтают; экземпляры говорят. Эволюция — разговор между ними.»*

## 2. История одной ошибки
Зоотехник изменил рацион прямо в загоне, чтобы срочно поднять удой. Изменение сработало, но design-time протокол остался старым. Через месяц новый сотрудник действовал по протоколу и вернул старый рацион, потеряв эффект. Разделение между проектированием (Tᴰ) и исполнением (Tᴿ) было нарушено.

## 3. Temporal Duality: проектирование ≠ работа — полное описание

### 3.1 Два временных scope
| Scope | Символ | Что это | Примеры |
|---|---|---|---|
| **Design-Time** | Tᴰ | Период, когда холон **может быть структурно изменён** внешним Transformer | Чертеж, CAD, SOP, скрипт, спецификация |
| **Run-Time** | Tᴿ | Период, когда холон **исполняет свои операционные методы** | Работающий насос, лечение коровы, эксплуатируемая ферма |

**Инварианты:**
```text
Tᴰ ∩ Tᴿ = ∅           (никогда не пересекаются)
Tᴰ ∪ Tᴿ = worldline   (вместе покрывают всё существование)
version(n+1) создаётся только в Tᴰₙ
```

### 3.2 Open-Ended Evolution Principle
Холон может бесконечно проходить цикл:

```text
(H₀ in Tᴿ₀) → observe → Δspec in Tᴰ₁ → build → H₁ in Tᴿ₁ → …
```

- Наблюдение — это тоже transformation.
- Измерение — это тоже transformation.
- Любой переход между Tᴰ и Tᴿ совершает внешний Transformer.

**На ферме:**
- Tᴰ: разработка нового протокола сухостоя.
- Tᴿ: применение протокола на ферме.
- Наблюдение: данные показывают, что протокол даёт больше кетозов.
- Возврат в Tᴰ: пересмотр протокола.
- Новая версия H₁ снова переходит в Tᴿ.

### 3.3 Почему нельзя смешивать
Если Tᴰ и Tᴿ пересекаются:
- «as-built» расходится с «as-designed» — незаметно.
- версии перезаписывают друг друга — provenance рвётся.
- измерение считается особым, нематериальным актом — пропадает provenance.

### 3.4 Примеры
#### 3.4.1 Насос
| Phase | Что происходит |
|---|---|
| Tᴰ | 3D-модель насоса, G-code, симуляция напряжений |
| Tᴿ | Насос качает охлаждающую жидкость |
| Tᴿ → Tᴰ | Датчики показывают кавитацию; отчёт создаётся системой мониторинга |
| Tᴰ → Tᴿ | Инженеры делают v3; 3D-принтер печатает новый корпус |

#### 3.4.2 Протокол лечения
| Phase | Что происходит |
|---|---|
| Tᴰ | MethodDescription: CS.METHOD.001 «Лечение кетоза» |
| Tᴿ | Ветеринар вводит пропиленгликоль конкретной корове |
| Tᴿ → Tᴰ | Данные по эффективности собраны; анализ показывает, что порог BHB нужно скорректировать |
| Tᴰ → Tᴿ | Обновлённая версия протокола утверждена и внедрена |

## 4. Почему смешивать / игнорировать — значит рисковать
Если пренебречь A.4, типичные риски на ферме:

| Что происходит | Почему это опасно |
|---|---|
| Решения принимаются на основе смешанных или устаревших данных | Невозможно отследить, что именно повлияло на результат |
| Роли и методы не разделены | Один и тот же человек или документ отвечает за несовместимые обязательства |
| Перенос знаний между контекстами без явного bridge | Рекомендации из одной традиции применяют к другой без учёта потерь |
| Отсутствие контрольного списка | Ошибки обнаруживаются слишком поздно, когда они уже стоят денег |

**Каскад:** нечёткое утверждение → неправильная интерпретация → некорректное решение → экономический убыток и снижение доверия к системе.

## 5. Как это выглядит на ферме: правильное применение
**Задача:** развести design-time протокол и run-time изменения для рациона.

**Было:**
> «Внесли корректировку в поле, записали в блокнот.»

**Стало:**
> Tᴰ: `RationProtocol_v4` с approved ingredient list и nutrient constraints.
> Tᴿ: `Work-RationAdjustment-0619` по полевому наблюдению.
> Feedback loop: результат полевого изменения передаётся в комитет по рационам.
> Change request: если изменение повторяется, оно оформляется как patch к `RationProtocol_v4`.
> Version pin: каждый Work ссылается на конкретную версию протокола.

Теперь полевые находки улучшают протокол, а не подрывают его.

## 6. Практическое применение: с чего начать
**Шаг 1.** Разметьте все документы: какие относятся к design-time (Tᴰ), какие — к run-time (Tᴿ).

**Шаг 2.** Запретите править Tᴰ-артефакты в поле без change request.

**Шаг 3.** Введите version pin: каждый Work ссылается на версию протокола.

**Шаг 4.** Создайте feedback loop для переноса удачных Tᴿ-изменений в Tᴰ.

**Шаг 5.** Раз в месяц проверяйте drift между Tᴰ и Tᴿ.
