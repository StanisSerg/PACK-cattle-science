# A.1.1 — Bounded Context: один термин — разные значения

## 1. Зачем это читать
Если вы думаете, что «кетоз» означает одно и то же для ветеринара, экономиста и разработчика отчётов — вы ошибаетесь. Если «протокол» в вашей ферме и у консультанта понимается по-разному, а вы этого не называете — вы строите мосты через пропасть, не замечая её.

A.1.1 учит: **смысл термина локален**. Он имеет силу только внутри объявленного bounded context. Перенос смысла между контекстами требует явного bridge.

**FPF-тезис:** *«Одинаковое написание ≠ одинаковое значение. Контекст решает смысл.»*

## 2. История одной ошибки
На ферме «Заря» слово «кетоз» использовалось в трёх разных смыслах: ветеринар под ним понимал клиническую картину, бухгалтер — статью затрат, а оператор отчёта — порог по BHB в молоке. Когда зоотехник получил выгрузку с надписью «кетоз: 12 %», он принял её за долю больных коров и изменил рацион. На самом деле это была доля затрат на профилактику в другом контексте. Удой упал, а обсуждение зашло в тупик, потому что каждый говорил о своём «кетозе».

## 3. Bounded Context: один термин — разные значения — полное описание

### 3.1 Что такое BoundedContext
`U.BoundedContext` — это холон, который хранит одну семантическую рамку:

- `contextIdentity` — имя контекста,
- `contextBoundary` — где он действует,
- `localVocabulary` — локальные значения терминов,
- `localInvariantSet` — локальные инварианты,
- `localRoleTaxonomy` — локальные роли,
- `bridgeRelationSet` — связи с другими контекстами.

**Контекст — не организация, не отдел, не документ, не домен.** Это рамка смысла.

### 3.2 Хорошие и плохие имена контекстов
| Хорошо | Плохо |
|---|---|
| `Farm.Ostromichi.RationProtocol.2026` | «Животноводство» |
| `VetClinical.Ketosis.Diagnostic` | «Здоровье коров» |
| `NASEM_2021.NutrientRequirements.Dairy` | «Наука» |
| `PackCS.Method.001.Ketosis` | «Метод» |

Бродкие ярлыки (healthcare, physics, software, agriculture) — это domain family, а не bounded context. Они ориентируют, но не решают смысл.

### 3.3 Bridge — не merge
Когда термин переходит из одного контекста в другой, нужно объявить bridge:

```text
BridgeRelation:
  sourceContextRef:
  targetContextRef:
  sourceValueRef:
  targetValueRef:
  relationKind:
  direction:
  fit:
  loss:
  scope:
```

**Пример:** `ticket` в авиакомпании — travel authorization; `ticket` в IT — work item. Без bridge'а это разные вещи.

**На ферме:** `кетоз` в клиническом контексте — диагноз с признаками; в экономическом контексте — статья потерь. Bridge должен явно сказать, как связаны диагноз и стоимость.

### 3.4 Контекст — не контейнер
BoundedContext локализует смысл, но **не содержит** все объекты, которые его используют. Система, документ, работа могут быть *интерпретированы в* контексте, *управляемы* им, *описаны под* ним — но не являться его частями.

### 3.5 Правильные и неправильные фразы
| Вместо | Пиши |
|---|---|
| «В нашей ферме кетоз = BHB > 1,2» | «В контексте `Farm.Ostromichi.Ketosis.Diagnostic` порог кетоза: BHB > 1,2 ммоль/л» |
| «Оператор должен проверить» | «Система в роли `OperatorRole:Farm.Ostromichi.Operations` должна выполнить Work» |
| «NASEM говорит, что нужно 16% CP» | «В контексте `NASEM_2021.Dairy` claim о CP применим к declared population; bridge к нашей ферме требует CL» |

## 4. Почему смешивать / игнорировать — значит рисковать
1. **Semantic drift.** Одно слово используется в разных значениях, но никто этого не называет.
2. **Leakage локальных правил.** Политика одного контекста применяется в другом без bridge.
3. **Конфликт там, где его нет.** Два контекста кажутся противоречивыми, хотя каждый внутренне согласован.
4. **Роль без контекста.** `Owner`, `operator`, `reviewer` используются как глобальные.
5. **Домен вместо контекста.** «Ветеринария» выступает аргументом, где нужен конкретный стандарт.

## 5. Как это выглядит на ферме: правильное применение
**Задача:** развести значения одного термина по разным bounded contexts.

**Было:**
> «Кетоз в отчёте 12 %.»

**Стало:**
> `Farm.Zarya.Vet.Clinical.Ketosis` — клинический диагноз по симптомам.
> `Farm.Zarya.Account.Cost.KetosisPrevention` — статья затрат.
> `Farm.Zarya.QA.BHBThreshold.KetosisRisk` — риск по порогу BHB ≥ 1.2 mmol/L.

Для каждого контекста введён отдельный Characteristic Card; при переносе цифры между контекстами требуется явный bridge с declared CL.

## 6. Практическое применение: с чего начать
**Шаг 1.** Соберите 5–7 терминов, которые на вашей ферме звучат по-разному у ветеринара, зоотехника и бухгалтера.

**Шаг 2.** Для каждого термина выпишите: в каком контексте он используется, какая шкала/критерий, кто авторитет.

**Шаг 3.** Присвойте каждому контексту уникальный идентификатор вида `Farm.<Unit>.<Domain>.<Term>`.

**Шаг 4.** Запретите перенос цифры из одного контекста в другой без bridge и declared CL.

**Шаг 5.** Проверьте все отчёты: если термин встречается без контекста — добавьте метку.

| Вопрос | Если ответ «да» — проблема |

| `CC-A1.1-2`: у контекста есть boundary, vocabulary, invariant set, role taxonomy. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1.1-3`: role assignments указывают ровно один bounded context. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1.1-4`: cross-context use выражен через bridge с fit, loss, scope. | Контрольный критерий не выполнен — возможна структурная ошибка. |
