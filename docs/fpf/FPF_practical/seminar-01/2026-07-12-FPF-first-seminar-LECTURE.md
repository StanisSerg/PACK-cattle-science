## 1. Что такое FPF на практике

FPF — это **не пошаговая инструкция**, а **язык паттернов** для мышления о проектах.

Он связывает три вещи:

- **Реальность** — объекты в физическом мире (корова, группа, рацион).
- **Знания** — утверждения, модели, описания (BHB ≥ 1.2 — кетоз).
- **Работу** — действия, которые изменяют реальность (скрининг, эксперимент, capture).

> Главная цель FPF — получить **первый полезный результат**, после которого становится возможным конкретный **следующий ход**.

## 2. Три уровня: FPF, DPF, LPF

| Уровень | Что это | Кто создаёт | Пример |
|---|---|---|---|
| **FPF** | Первые принципы, общая онтология | Авторы FPF | `EntityOfConcern`, `U.Work`, `C22.2` |
| **DPF** | Предметные паттерны | Эксперты области | `CS.METHOD.007` — диагностика проблем кормления |
| **LPF** | Местные правила конкретного предприятия | Организаторы/разработчики | «На нашей ферме pH мочи нормируем 6.0–6.5» |

- Ты **сам разрабатываешь паттерны DPF/LPF**.
- FPF даёт **язык и форму**, в которой эти паттерны должны быть написаны.
- DRR (Design Rationale Record) — полезен при разработке DPF/LPF, но **не зашит в FPF** и не требуется при обычном применении.

## 3. Карточка README vs паттерн vs мантра

| | Карточка README | Паттерн | Мантра |
|---|---|---|---|
| **Где** | В README | В FPF/DPF/LPF | В карточке README |
| **Задача** | Помочь узнать свою ситуацию | Дать способ действия | Показать один понятный проход |
| **Содержит** | Ситуация, результат, паттерн | Problem, Forces, Solution, Consequences | Показательную развёртку CGUS |
| **Обязательна** | Нет | Да, если используешь паттерн | Нет |

**Важно:** одна карточка → один ведущий паттерн, но один паттерн может использоваться в разных карточках.

## 4. Мантра, CGUS и иллюзия цепочки

### Мантра

- **Мантра** — это **показательная развёртка**, псевдо-последовательность для удержания внимания.
- Она **не является планом работы**.
- Помогает мозгу, который привык думать «сначала-второе-третье», перейти к декларативному мышлению.

### CGUS

- **CGUS** = Constraint-Governed Unfolding Structure.
- Это структура **типизированных позиций, отношений и ограничений**.
- Паттерны связаны не цепочкой, а через позиции и ограничения.

### Иллюзия цепочки

Паттерны **никогда не являются цепочкой**. Мы видим цепочку в мантре, но на самом деле каждый паттерн выбирается **по ситуации**. В любой момент можно ответвиться или вернуться назад.

> Линейный порядок строк мантры — это порядок **объяснения**, а не порядок **работы**.

## 5. Kind и эпистема

### Kind

**Kind** — это «вид» объекта: понятие с интенсионалом, экстенсионалом и допустимыми отношениями в ограниченном контексте.

Примеры kinds:

| Kind | Пример |
|---|---|
| `U.System` | Корова, группа, стадо |
| `U.Method` | Алгоритм диагностики кетоза |
| `U.MethodDescription` | Файл `CS.METHOD.007.md` |
| `U.Work` | Проведённый скрининг |
| `U.WorkPlan` | План отбора проб завтра |
| `U.Episteme` | «BHB ≥ 1.2 — кетоз» |
| `ProblemCard@Context` | Карточка проблемы |
| `ChoiceResult` | Выбранный метод корректировки рациона |

### Эпистема

**Эпистема (`U.Episteme`)** — это **один конкретный kind**. Она обозначает знание: утверждения, модели, теории.

> **Kind** — любая категория. **Эпистема** — категория знаний.

### Пример

Размытое слово «отчёт»:

- если это инструкция → `U.MethodDescription`;
- если это запись о проведённой работе → `U.Work`;
- если это аналитические выводы → `U.Episteme`.

## 6. Архитектура, описание, математика и АрхГейт

### Архитектура

**Архитектура** — это **интересующие нас структуры, найденные прямо в реальном мире**.

- Не путать с **описанием** (чертёж, документ).
- Не путать с **математикой** (модель, уравнения).

| | Архитектура | Описание | Математика |
|---|---|---|---|
| **Что** | Реальные структуры | Знание о структурах | Формальная модель |
| **Пример** | Три кормосмесителя на ферме | Схема кормления | Оптимизация рациона |
| **Kind** | `U.System` + `Structure` | `U.Episteme` | `U.Episteme` |

### Архитектура метода

Архитектура есть не только у систем, но и у **методов, работ, дисциплин**. Например, `CS.METHOD.007` имеет архитектуру из 8 шагов.

### АрхГейт

- **АрхГейт** — это не архитектура, а **метод оценки архитектурных решений**.
- Оценивает варианты по 7 характеристикам ЭМОГССБ.
- Можно построить 4 варианта архитектуры метода и сравнить их через АрхГейт.

## 7. Entity of Concern и bounded context

### Entity of Concern (EOC)

**EOC** — это «о чём именно мы сейчас рассуждаем». Это не весь проект, а конкретный объект текущего вопроса.

### Bounded context

**Bounded context** уточняет, в какой ситуации, версии и временном окне действует выбор.

### Пример

Слово «кетоз» в разных bounded context:

| Контекст | Значение |
|---|---|
| Ветеринарный | BHB ≥ 1.2 ммоль/л |
| Экономический | Событие потерь |
| Рациональный | Негативный энергетический баланс |

## 8. Акаузальность

**Акаузально** — значит, нет заранее заданного порядка шагов.

- Ты описываешь ситуацию.
- Агент с помощью FPF предлагает подходящую карточку и паттерн.
- Ты **не задаёшь заранее** порядок применения паттернов.
- Но ты **проверяешь**, подходит ли предложенный паттерн, и читаешь его полный текст.

> FPF не думает сам. Это агент использует FPF, а человек принимает решение.

## 9. Минимальная онтология FPF

Минимальный набор понятий, достаточный для разбора любой проектной ситуации:

| Понятие | Вопрос | Пример |
|---|---|---|
| Entity of Concern | О чём рассуждаем? | pH мочи коровы |
| Holon | Целое с частями? | Корова → группа → стадо |
| Episteme | Что знаем? | «BHB ≥ 1.2 — кетоз» |
| Method | Как работаем? | CS.METHOD.007 |
| WorkPlan | Что планируем? | Отобрать пробы завтра |
| Work | Что сделано? | Скрининг проведён |
| Structure | Как устроено? | Состав рациона |
| Architecture | Какие структуры важны? | Три кормосмесителя |
| Characteristic | По чему оцениваем? | Продуктивность, pH |
| Candidate | Какие варианты? | Уменьшить сенаж / добавить бикарбонат |
| Decision | Что выбрали? | Вариант А |
| Role | Кто действует? | Технолог по кормлению |

Если ты можешь назвать EOC и kind — ты уже на полпути к подходящему паттерну.

## 10. Паттерн C22.2 — карточка проблемы

### Когда применяется

Есть слабый сигнал или уже обсуждается решение, но **постановка проблемы ещё не принята**.

### Что даёт

`ProblemCard@Context` — карточка проблемы, достаточная для поиска и сравнения методов.

### Структура карточки

| Поле | Пример: pH мочи |
|---|---|
| Сигнал | pH мочи снизился до 6.8 |
| Bounded context | Группа А, ферма МТК Ленинский, июль 2026 |
| Проблема | pH вне целевого диапазона |
| Признак улучшения | pH 6.0–6.5 с сохранением продуктивности |
| Следующий ход | Подобрать метод корректировки рациона |

### Связь с bounded context и следующим ходом

```
Bounded context задаёт границу →
C22.2 формирует ProblemCard внутри границы →
ProblemCard открывает следующий ход
```

## 12. Список вопросов и ответов

1. **DRR зашит в FPF?** — Нет, это практика авторинга DPF/LPF.
2. **15 карточек README — это базовые принципы?** — Нет, это типовые входы в паттерны.
3. **При разработке DPF я сам создаю паттерны?** — Да, FPF даёт форму, ты даёшь содержание.
4. **Мантра — это паттерн?** — Нет, мантра в карточке, паттерн — отдельно.
5. **Паттерны — цепочки?** — Нет, они связаны через CGUS, не линейно.
6. **Что такое CGUS?** — Структура позиций, отношений и ограничений.
7. **Kind и эпистема — разное?** — Да, kind — категория, эпистема — вид знания.
8. **Архитектура = описание?** — Нет, архитектура — реальные структуры.
9. **АрхГейт = архитектура?** — Нет, АрхГейт — оценка архитектурных решений.
10. **Что такое EOC?** — Объект текущего рассуждения вbounded context.
11. **Акаузально — значит FPF сам решает?** — Нет, агент предлагает, человек выбирает.
12. **Можно попросить агента описать карточки для выбора?** — Да, см. шаблон.
13. **Можно попросить построить CGUS?** — Да, но лучше для конкретной ситуации.
14. **Минимальная онтология — это минимум для разбора проблемы?** — Да.
15. **C22.2 — что делает?** — Превращает слабый сигнал в карточку проблемы.
