# Пример автоматической обработки отчётов: CASE-002

> **Назначение:** показать, как система контроля (вариант B) может обрабатывать данные фермы автоматически и генерировать оповещения для пилота.  
> **Основа:** данные CASE-002 (КТ Зенченко, МТК Ленинский, Ж/К №2).  
> **Статус:** пример для пилота, не production-скрипт.

---

## 1. Что можно автоматизировать на основе CASE-002

В кейсе ключевые события и тренды выявлялись вручную при анализе отчётов. Система контроля может делать то же самое автоматически:

- падение продуктивности на корову;
- падение продуктивности по группам;
- выход физической структуры TMR за целевые границы;
- несоответствие версий рецептов комбикорма;
- отклонение фактического DMI от расчётного;
- резкие изменения дойного поголовья.

---

## 2. Источники данных и их формат

| Источник | Формат данных | Частота | Пример поля |
|----------|---------------|---------|-------------|
| **Ежедневный отчёт по ферме** | CSV / Excel / API доильной системы | Ежедневно | `date`, `farm_id`, `avg_yield_l`, `milk_sales_kl`, `milking_cows` |
| **Отчёт по группам** | CSV / Excel | Еженедельно | `date`, `group_id`, `dim`, `yield_l`, `headcount` |
| **Результаты пенсильванских сит** | CSV / Excel | Еженедельно | `date`, `group_id`, `sieve_1_pct`, `sieve_2_pct`, `sieve_3_pct`, `sieve_4_pct` |
| **Журнал загрузки бункера** | CSV / 1С / API | На каждую загрузку | `timestamp`, `recipe_id`, `loaded_kg`, `recipe_version` |
| **Расчётный рацион AMTS** | PDF / Excel / API | При изменении | `group_id`, `target_dmi_kg`, `recipe_version` |
| **События фермы** | Журнал | По событию | `date`, `event_type`, `description` |

## 3. Правила-триггеры (автоматическая обработка)

### 3.1. Падение продуктивности на корову

```
IF (avg_yield_7d - avg_yield_1d) / avg_yield_7d > 0.05
   AND avg_yield_7d > 0
THEN alert(
  level = "warning",
  message = "Удой на корову снизился более чем на 5% за сутки",
  context = {farm_id, date, avg_yield_7d, avg_yield_1d}
)
```

**Пример из CASE-002:**
> 20.06.2026 — открытие новой траншеи сенажа + жара до 36 °C. Система зафиксировала бы краткосрочное падение и оповестила пилота.  
> 06–07.07.2026 — вакцинация + отбор крови. Система зафиксировала бы временное снижение и отметила его как внешнее событие.

### 3.2. Падение продуктивности по группе

```
FOR each group_id:
  IF (yield_current_7d - yield_previous_7d) / yield_previous_7d < -0.10
     AND previous_7d > 0
  THEN alert(
    level = "critical",
    message = "Продуктивность группы снизилась более чем на 10% за неделю",
    context = {group_id, date, yield_previous_7d, yield_current_7d}
  )
```

**Пример из CASE-002:**
> Группа 19 — единственная с отрицательной динамикой (−1,0 л/сут). Система автоматически выделила бы эту группу и предложила пилоту разбор.

### 3.3. Физическая структура TMR выходит за границы

```
IF sieve_1_pct > 5.0
THEN alert(
  level = "warning",
  message = "Доля крупных частиц TMR (сито 1) превышает 5% — риск сортировки",
  context = {group_id, date, sieve_1_pct}
)

IF sieve_2_pct < 40.0
THEN alert(
  level = "warning",
  message = "Целевая фракция TMR (сито 2) ниже 40%",
  context = {group_id, date, sieve_2_pct}
)
```

**Пример из CASE-002:**
> До интервенции: сито 1 = 6.4%, сито 2 = 43.4%. Система оповестила бы о высокой доле крупных частиц.  
> После интервенции (08.07.2026): сито 1 = 2.19%, сито 2 = 47.23%. Система зафиксировала бы нормализацию.

### 3.4. Несоответствие версии рецепта комбикорма

```
IF loaded_recipe_version != current_active_recipe_version
THEN alert(
  level = "critical",
  message = "Загружен устаревший или неправильный рецепт комбикорма",
  context = {timestamp, mixer_id, loaded_recipe_version, current_active_recipe_version}
)
```

**Пример из CASE-002:**
> 06.07.2026 выяснилось, что рецепты 5 и 6 перепутаны из-за устаревших рецептов апреля. Система сравнения версий предотвратила бы это заранее.

### 3.5. Отклонение фактического DMI от целевого

```
IF abs(actual_dmi - target_dmi) / target_dmi > 0.10
THEN alert(
  level = "warning",
  message = "Фактический DMI отклоняется от целевого более чем на 10%",
  context = {group_id, date, actual_dmi, target_dmi, deviation_pct}
)
```

**Пример из CASE-002:**
> Группы 2–8: target DMI ~23.9 кг СВ, фактический DMI варьировался от 100 до 143 кг по группам (в таблице «Потребление рациона» — это суммарное потребление, а не на голову). Для автоматизации нужно приводить к DMI на голову.

### 3.6. Резкое изменение дойного поголовья

```
IF abs(milking_cows_today - milking_cows_7d_avg) / milking_cows_7d_avg > 0.05
THEN alert(
  level = "warning",
  message = "Дойное поголовье изменилось более чем на 5% за неделю",
  context = {farm_id, date, milking_cows_today, milking_cows_7d_avg}
)
```

**Пример из CASE-002:**
> В конце июня дойное поголовье на Ж/К №2 снизилось до ~850 голов с пика ~882. Система оповестила бы и предложила проверить причину.

## 4. Пример ежедневной сводки для пилота

```text
Сводка за 2026-07-08 (Ж/К №2, КТ Зенченко)
============================================

🟢 Продуктивность на корову: 32.8 л/сут (стабильно vs 7d avg)
🟢 Реализация молока: 27.9 тыс. л/сут (стабильно)
🟡 Физическая структура TMR группа 2: сито 1 = 2.19% (норма), сито 2 = 47.23% (норма)
🔴 Группа 19: продуктивность 19.9 л/сут (−1.0 л/сут за 2 недели) → требуется разбор
🟢 Дойное поголовье: 851 гол. (стабильно)
🟢 Загрузки бункера: версии рецептов соответствуют активным

Рекомендуемое действие:
- Разобрать группу 19 (здоровье, рацион, генетика, точность доения).
- Планировать следующий отбор пенсильванских сит для групп 1, 7, 8.
```

## 5. Минимальный комплект для пилота B

Для запуска системы контроля на этой ферме достаточно:

1. **Ежедневный отчёт по продуктивности** (из доильной системы).
2. **Еженедельный отчёт по группам** (DIM, удой, поголовье).
3. **Еженедельные результаты пенсильванских сит** (группа 2 как контрольная).
4. **Журнал загрузки бункера** с версией рецепта.
5. **Журнал событий** (вакцинация, отбор крови, смена сенажа, погода).

## 6. Связь с планом пилота

Этот пример показывает, как может выглядеть автоматическая обработка в рамках пилота варианта B. Он не требует обучения технолога глубокой интерпретации метода — только регулярную фиксацию исходных данных и реакцию пилота на сгенерированные оповещения.

## 7. Что доработать в следующий раз

> Пометка для следующей сессии WP-109.

1. **Адаптировать триггеры под конкретный формат данных** фермы (Excel, 1С, API доильной системы).
2. **Уточнить пороги** на основе внутренних норм фермы, а не универсальных примеров (5%, 10%, 40%).
3. **Добавить обработку ложных срабатываний** — как отличить внешние события (вакцинация, жара, смена сенажа) от системных проблем.
4. **Протестировать один триггер** на исторических данных CASE-002: проверить, что система оповестила бы о смешении рецептов, падении группы 19 и изменении физической структуры TMR.
5. **Связать с CS.METHOD.007** — превратить триггеры в шаги типовой проблемы «падение продуктивности».
6. **Продумать канал оповещений** пилота (мессенджер, email, дашборд) и частоту сводок.
