# C.1 — Sys-CAL

## 1. Зачем это читать

Если вы когда-нибудь видели, как на ферме говорят: **«У нас хорошая система управления»** — без уточнения, что именно делает эта система, какие у неё границы, входы, выходы и надёжность — вы встретили C.1 «в диком виде».

> **FPF-тезис:** *«Система должна быть охарактеризована не по названию, а по ролям, функциям, интерфейсам и доверительным свойствам. Характеризация — это не маркетинговое описание, а измеримый профиль, который можно проверить, сравнить и передать другому агенту.»*

C.1 — Sys-CAL (System Characterization & Acceptance Lens). Это базовый паттерн FPF Part C: перед тем как принимать систему в эксплуатацию, её нужно описать через набор характеристик (CHR), ограничений (CAL) и уровней доверия.

**Фермерский пример:**

> Ферма рассматривает покупку системы мониторинга жвачки. Продавец говорит: «Система улучшает здоровье коров». Но что это значит? Какие сенсоры? Как часто измеряет? Какая точность? Как интегрируется с доильным роботом? Каковы ограничения по температуре и влажности? Без C.1 ферма покупает «кота в мешке».

---

## 2. История одной ошибки

Представьте ферму, которая внедряет **систему раннего оповещения о кетозе**:

> «У нас теперь есть система раннего оповещения. Будем снижать кетоз.»

Через год приходит аудитор. Он спрашивает:

- «Что входит в систему?» — Ответ: «Датчики, программа, ветеринар» (но нет explicit role map).
- «Какие измерения она производит?» — Ответ: «BHB, вроде» (но BHB — кровь/молоко/дыхание? Какой метод?).
- «Какие у неё ограничения?» — Ответ: «Работает всегда» (но не указан температурный диапазон, latency, missingness).
- «Какой у неё уровень доверия?» — Ответ: «Продавец сказал 95%» (но R, R_eff, CL не разделены).

Аудитор уходит ни с чем. Потому что система не была охарактеризована. C.1 предлагает: роли, функции, интерфейсы, характеристики, acceptance criteria.

---

## 3. System Characterization — полное описание

### 3.1 Роли и функции

**Система = набор ролей, каждая со своими функциями.**

**На ферме для системы мониторинга жвачки:**

| Роль | Функция | Актор |
|---|---|---|
| Сенсор жвачки | Измерять количество и интенсивность жвачки | T (tool) |
| Базовая станция | Собирать данные, передавать на сервер | T |
| Аналитический модуль | Выявлять отклонения от baseline | A (agent) |
| Интерфейс оповещения | Показывать алерты зоотехнику | T |
| Ветеринар | Принимать решение по алерту | H (human) |

**Ключевой признак:** Нельзя говорить «система делает X», не уточняя, какая роль это делает и кто/что за неё отвечает.

### 3.2 Интерфейсы

**Входы и выходы системы должны быть явными.**

**На ферме:**
- **Входы:** данные сенсоров, ID коровы, время, температура окружающей среды.
- **Выходы:** score риска нарушения жвачки, alert, trend-график.
- **Внешние связи:** интеграция с ПО управления стадом, API для ветеринарного журнала.

**Ключевой признак:** Неописанный интерфейс — это скрытое предположение, которое приведёт к сбою.

### 3.3 Характеристики (CHR)

**На ферме:**
- **Accuracy:** sensitivity, specificity при обнаружении ацидоза.
- **Latency:** время от события до алерта.
- **Availability:** доля времени, когда система онлайн.
- **Maintainability:** интервал калибровки, замены батарей.
- **Scalability:** максимальное число коров.

**Ключевой признак:** Каждая характеристика должна иметь тип шкалы (ratio, ordinal и т.д.) и метод измерения.

### 3.4 Ограничения и acceptance criteria (CAL)

**На ферме:**
- **Hard constraints:** работа при −20…+40 °C; совместимость с существующими болусами.
- **Soft constraints:** стоимость владения ≤ 500 тыс. руб/год; latency ≤ 15 мин.
- **Acceptance criteria:** sensitivity ≥ 80% при false alarm rate ≤ 20% на пилоте из 100 коров.

**Ключевой признак:** Acceptance criteria — это не пожелания, а проверяемые условия, при которых система принимается в эксплуатацию.

### 3.5 Доверительные свойства

**На ферме:**
- **R (reliability):** измеренная на обучающем наборе.
- **CL (congruence level):** насколько обучающий контекст близок к ферме.
- **R_eff:** effective reliability = R × Φ(CL).
- **Provenance:** кто разработал, на каких данных, какая версия.

**Ключевой признак:** Доверие — тоже характеристика, которую нужно измерять и записывать.

## 4. Каскад типичных ошибок

Возьмём снова:

> *«У нас теперь есть система раннего оповещения о кетозе.»*

**Разложение по C.1:**

| Часть утверждения | Что это в FPF | Почему ошибка |
|---|---|---|
| «система» | **Неразложенная роль** | Не ясно, что входит в систему |
| «раннего оповещения» | **Функция без интерфейса** | Нет определения «рано», нет latency |
| «о кетозе» | **Неточная цель** | Кетоз по BHB крови? молока? поведению? |
| «теперь есть» | **Нет acceptance criteria** | Не проверено, принята ли система |

**Что плохого в смешивании:**

1. **Невозможно проверить.** «Улучшает здоровье» — не измеримо.
2. **Невозможно интегрировать.** Неизвестные интерфейсы ломают workflow.
3. **Невозможно доверять.** Нет R, CL, provenance.

## 5. Как это выглядит на ферме: правильное применение

Перепишем внедрение системы мониторинга.

**Было (смешанное):**
> «У нас теперь есть система раннего оповещения о кетозе.»

**Стало (разложенное):**

**System profile:**
```
SystemID: rumination_monitor_v2
Roles:
  - sensor_neck: measure rumination minutes/2h (T)
  - gateway_barn: collect and forward data (T)
  - anomaly_detector: flag deviation from cow baseline (A)
  - alert_ui: show alerts to manager (T)
  - veterinarian: decide on alert (H)
Interfaces:
  In: {sensor_id, timestamp, rumination_min, env_temp}
  Out: {cow_id, alert_level, deviation_score, trend_plot}
  External: integration with herd_management_api_v3
CHR:
  - latency: ≤ 15 min (ratio)
  - availability: ≥ 99% (ratio)
  - sensitivity@subclinical_ketosis: ≥ 80% (ratio)
  - false_alarm_rate: ≤ 20% (ratio)
CAL:
  - hard: temp_range −20..+40 °C, IP65
  - soft: capex ≤ 800K RUB, annual_support ≤ 150K RUB
Acceptance criteria:
  - Pilot on 100 cows, 90 days
  - sensitivity ≥ 80%, FAR ≤ 20% vs vet-confirmed BHB
  - availability ≥ 99%
Trust:
  - R_base: 0,85 (vendor report)
  - CL (vendor_context → our_farm): 3
  - R_eff: 0,85 × 0,8 = 0,68
```

**Что это даёт:**
- Все роли и интерфейсы видны.
- Acceptance criteria проверяемы.
- Доверие выражено через R_eff.
- Система готова к интеграции и аудиту.

## 6. Практическое применение: с чего начать

**Шаг 1.** Для любой новой системы составьте role map: кто/что делает что.

**Шаг 2.** Опишите входы, выходы, внешние связи.

**Шаг 3.** Выпишите 4–7 CHR с типами шкал и методами измерения.

**Шаг 4.** Задайте hard/soft constraints и acceptance criteria.

**Шаг 5.** Запишите R_base, CL, R_eff и provenance.

**Шаг 6.** Не принимайте систему в эксплуатацию, пока acceptance criteria не выполнены.

---

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| Говорите ли «система» без распределения ролей? | Неразложенная система |
| Есть ли функции без входов/выходов? | Неявные интерфейсы |
| Используете ли маркетинговые оценки? | Отсутствие CHR |
| Принимаете ли систему без acceptance criteria? | Нет CAL |
| Есть ли только R без CL и R_eff? | Неполная trust-profile |
