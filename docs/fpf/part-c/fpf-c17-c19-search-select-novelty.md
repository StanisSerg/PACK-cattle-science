# C.17—C.19 — Search, Select, Novelty

> **Цель capture:** объяснить, почему в FPF поиск решений — это не нахождение единственного победителя, а иллюминация пространства кандидатов, и как связаны измерение креативности, открытый поиск и губернатор explore/exploit.

---

## 1. Зачем это читать

Если на ферме из двадцати вариантов рациона выбирают «оптимальный» по одной формуле, а потом удивляются, что коровы «не пошли» — это C.17–C.19 «в диком виде». Эти три паттерна вместе описывают ответственный поиск: как измерять новизну и ценность, как генерировать и хранить разнообразных кандидатов, и как управлять балансом исследования (explore) и использования (exploit).

> **FPF-тезис:** *«Поиск решений — это иллюминация пространства: генерация разнообразного множества кандидатов и осознанный выбор по многим критериям. Новизна и разнообразие — telemetry, а не доминирование, если политика не сказала иначе.»*

**Фермерский пример:**

> Ферма ищет новый рацион для переходного периода. Зоотехник предлагает три варианта и выбирает один по «средней оценке»: цена, протеин, энергия. Через месяц выясняется, что победивший вариант хорош для большинства коров, но плох для первотёлок и для жарких дней. Если бы ферма использовала C.17–C.19, она сохранила бы фронт решений: дешёвый базовый, дорогой с высокой энергией, устойчивый к жаре — и выбрала бы осознанно под контекст.

---

## 2. История одной ошибки

Ферма решила выбрать систему доения. Инженер сравнил три робота: «Робот A — 32 кг молока в день, B — 31 кг, C — 30 кг. Выбираем A». Через год выяснилось, что скрытые критерии — цена, ремонтопригодность, стресс коров, доступность локального сервиса — не были учтены. Отвергнутые варианты удалили, хотя они могли пригодиться при смене контекста. ConstraintFit был afterthought. Скалярный выбор по одной оси обошёлся дороже, чем разница в удое.

## 3. Search, Select, Novelty — полное описание

### 3.1 Определение

C.17–C.19 объединяют три паттерна. **C.17 Creativity-CHR** описывает, как измерять новизну, ценность, неожиданность, соответствие ограничениям и разнообразие. **C.18 NQD-CAL** описывает открытый поиск: генерацию кандидатов, архив, фронт и telemetry. **C.19 E/E-LOG** описывает губернатор explore/exploit: что делать с живым пулом кандидатов дальше.

### 3.2 Почему это важно

Скрытая скаляризация многокритериальной задачи в одно число приводит к ложной оптимальности, потере знаний и невозможности объяснить выбор. C.17–C.19 требуют явного пространства характеристик, declared dominance set и policy-governed choice.

### 3.3 Creativity-CHR

**Определение.** Creativity-CHR предоставляет измерительный набор для креативных исходов: `Novelty@context`, `Use-Value` (или `ValueGain`), `Surprise`, `ConstraintFit`, `Diversity_P` (на уровне портфеля), `Originality`, `AttributionIntegrity` и `ResourceEfficiency`.

**Пояснение.** Этот паттерн не говорит, как выбирать; он только предоставляет оси для измерения. Выбор остаётся за C.18/C.19 и G.5.

**Пример из животноводства.** Для рациона переходного периода:
- Novelty@context — насколько рацион отличается от текущего базового (0–1).
- Use-Value — улучшение BHB или снижение риска ацидоза.
- ConstraintFit — доля must-ограничений, которые выполнены (бюджет, минеральные нормы, доступность сырья).
- Diversity_P — на уровне портфеля из четырёх рационов покрываются жаркая/холодная погода, первотёлки/многотёлки.

**Ключевой признак.** Creativity-CHR экспортирует измерения, а не операторы выбора.

### 3.4 NQD-CAL

**Определение.** NQD-CAL (Novelty-Quality-Diversity Calculus) — это открытый поисковый исчисление: генерация кандидатов, обновление архива, illumination telemetry и вычисление фронта.

**Пояснение.** Основные понятия: `DescriptorMap` превращает кандидата в точку в `CharacteristicSpace`; `CandidateSet` — множество гипотез с векторами {Q, N, D, S}; `Archive` или `Niche` — ячейки в пространстве характеристик; `Front` — недоминируемое множество по declared dominance set; `IlluminationSummary` — report-only telemetry.

**Пример из животноводства.** Для выбора рациона DescriptorMap может быть {NEL/СВ, %NDF, DCAD, цена}. Генератор создаёт 50 рационов, archive хранит элиты по ячейкам, selectFront вычисляет Pareto-front по цене и BHB. IlluminationSummary сообщает, насколько заполнено пространство, но не решает, какой рацион выбрать.

**Ключевой признак.** Pipeline: Eligibility (ConstraintFit = pass) → Dominance → Tie-breakers. Illumination — report-only, пока policy не разрешит иначе.

### 3.5 E/E-LOG

**Определение.** E/E-LOG (Explore-Exploit Governor) — это паттерн управления живым пулом кандидатов: решать, когда расширять поиск (widen), сохранять фронт (keep frontier), сужаться до подмножества (narrow), закрывать линию (sunset) или перенаправлять (reroute).

**Пояснение.** E/E-LOG не делает локальный выбор (это C.11) и не публикует selector-facing shortlist (это G.5). Он управляет пулом. Ключевые элементы: `EmitterPolicy` (регим, explore_share, temperature, backstop_confidence), `Lens` (hard eligibility, soft aggregation, trust policy) и `Pool-policy result` с change trigger.

**Пример из животноводства.** После первого фронта рационов ферма решает: `keep frontier` и провести пилот двух вариантов; change trigger — получение backstop confidence L1 через три месяца. Пока policy не разрешит, фронт остаётся живым.

**Ключевой признак.** E/E-LOG возвращает policy result, а не одного победителя.

### 3.6 Нет скалярного победителя по умолчанию

**Определение.** По умолчанию FPF требует `ParetoOnly`: если фронт показывает компромисс между критериями, нельзя просто сложить взвешенную сумму без явной policy.

**Пояснение.** Если один рацион дешёвый, другой энергетически плотный, третий устойчив к жаре — все три могут быть на фронте. Выбор одного — это решение `DecisionSubject` с явным lens id и policy, который фиксирует trade-offs.

**Пример из животноводства.** Фронт по цене и BHB показывает компромисс: дешёвый рацион уступает по BHB, дорогой — по цене. Без объявленной policy нельзя сказать, какой «лучше»; оба остаются на фронте.

**Ключевой признак.** «Лучшее» — это не одно число, а осознанный выбор из front с явными trade-offs.

## 4. Почему смешивать / игнорировать — значит рисковать

Рассмотрим типичное смешанное утверждение:

> *«Робот A даёт 32 кг, робот B — 31 кг, робот C — 30 кг. Выбираем A.»*

**Разложение по C.17–C.19:**

| Часть утверждения | Что это в FPF | Почему важно разделять |
|---|---|---|
| «32 кг, 31 кг, 30 кг» | Use-Value по одной оси | Игнорируются другие критерии |
| «Выбираем A» | скалярный выбор без lens/policy id | Нет явного DecisionSubject и policy |
| удаление B и C | потеря Diversity_P / archive | Отвергнутые варианты не сохранены |
| «главный KPI» | скрытая скаляризация | Нет объявленного dominance set |
| отсутствие ограничений | ConstraintFit не как gate | Ремонтопригодность, сервис, стресс — afterthought |

**Основные риски смешивания:**

1. **Ложная оптимальность.** Победитель по одной оси может проиграть по жизненному циклу.
2. **Потеря знаний.** Варианты B и C могли пригодиться при смене контекста.
3. **Невозможность объяснить выбор.** Аудитор не видит trade-offs и policy.

## 5. Как это выглядит на ферме: правильное применение

**Ситуация:** ферма выбирает систему доения.

**Было (смешанное / нечёткое):**

> «Робот A — 32 кг, B — 31 кг, C — 30 кг. Выбираем A.»

**Стало (разложенное / ясное):**

**Context:** `MilkingSystemSelection_2026`

**CreativitySpace:**
> Q1: DailyYield [кг/день, ratio, ↑]
> Q2: Capex [руб., ratio, ↓]
> Q3: ServiceAvailability [часы простоя/год, ratio, ↓]
> Q4: CowStressIndex [ordinal, ↓]
> N: Novelty@context [0–1, ↑]
> ConstraintFit: {must: local service center < 100 km; must: compatible with existing software}

**CandidateSet:**

| Робот | Q1 | Q2 | Q3 | Q4 | N | ConstraintFit |
|---|---|---|---|---|---|---|
| A | 32 | 12M | 80 | 3 | 0,2 | pass |
| B | 31 | 9M | 40 | 2 | 0,4 | pass |
| C | 30 | 8M | 30 | 2 | 0,6 | pass |
| D | 28 | 7M | 25 | 1 | 0,8 | fail (нет сервиса) |

**Dominance:** A не доминирует B (хуже capex и stress). B и C — на фронте.

**E/E-LOG pool-policy result:**
> livePool = {A, B, C}
> governingLens = `multi_objective_dairy_lens_v2`
> currentTreatment = `keep_frontier`
> changeTrigger = пилотная установка B и C на 3 месяца даёт backstop_confidence L1
> whyNotLocalChoice = trade-offs по capex/stress/yield ещё не разрешены

**Результат:**
- Ферма не спешит выбрать «победителя» по одной оси.
- Варианты A, B, C остаются живыми; D отсеян по ConstraintFit.
- Решение будет принято после пилота с явным lens id и policy.

## 6. Практическое применение: с чего начать

**Шаг 1.** Определите проблему поиска: рацион, оборудование, протокол, лекарственная схема.

**Шаг 2.** Объявите CreativitySpace: 2–4 quality-оси + Novelty + ConstraintFit.

**Шаг 3.** Сгенерируйте CandidateSet. Не обязательно автоматически — можно вручную, но с явными векторами.

**Шаг 4.** Примените ConstraintFit как hard gate.

**Шаг 5.** Вычислите Pareto-front по declared dominance set.

**Шаг 6.** Запишите E/E-LOG policy: keep frontier / narrow / widen / sunset / reroute.

**Шаг 7.** Если выбираете одно решение, укажите lens id, DecisionSubject и trade-offs.

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| Есть ли «оптимальный» вариант без объявленных критериев? | Скрытая скаляризация |
| Сравниваете ли разнородные шкалы взвешенной суммой? | Незаконная агрегация |
| Сохраняете ли отвергнутые варианты как archive? | Потеря Diversity_P |
| Входит ли Illumination в dominance без policy id? | Telemetry → dominance creep |
| Есть ли explicit pool-policy result с change trigger? | Незавершённый E/E-LOG |

## 9. Что запомнить

1. Поиск — это иллюминация пространства, а не гонка за одним победителем.
2. Creativity-CHR измеряет, NQD-CAL генерирует и хранит, E/E-LOG управляет пулом.
3. ConstraintFit — hard gate, а не afterthought.
4. ParetoOnly по умолчанию; scalar winner требует явной policy.
5. Отвергнутые варианты — это knowledge, а не мусор.
