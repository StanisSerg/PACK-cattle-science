# C.17–C.19 — Search, Select, Novelty

## 1. Зачем это читать

Если вы когда-нибудь видели на ферме, как **из 20 вариантов рациона выбирают «оптимальный» по одной формуле**, а потом удивляются, что коровы «не пошли» — вы встретили C.17–C.19 «в диком виде».

> **FPF-тезис:** *«Поиск решений — это не нахождение единственного победителя, а иллюминация пространства: генерация разнообразного множества кандидатов и осознанный выбор по многим критериям. Новизна и разнообразие — telemetry, а не доминирование, если политика не сказала иначе.»*

Этот capture объединяет три паттерна:
- **C.17 Creativity-CHR** — как измерять новизну, ценность, неожиданность и соответствие ограничениям.
- **C.18 NQD-CAL** — как генерировать и хранить разнообразных кандидатов (archive, front, illumination).
- **C.19 E/E-LOG** — как управлять балансом исследования (explore) и использования (exploit) над живым пулом кандидатов.

**Фермерский пример:**

> Ферма ищет новый рацион для переходного периода. Зоотехник предлагает 3 варианта и выбирает один по «средней оценке»: цена, протеин, энергия. Через месяц выясняется, что победивший вариант хорош для большинства, но плох для первотёлок и для жарких дней. Если бы ферма использовала C.17–C.19, она сохранила бы фронт решений: дешёвый базовый, дорогой с высокой энергией, устойчивый к жаре — и выбрала бы осознанно под контекст.

## 2. История одной ошибки

Представьте ферму, где решают **выбрать систему доения**:

> «Робот A даёт 32 кг молока в день, робот B — 31 кг, робот C — 30 кг. Выбираем A.»

Через год приходит аудитор. Он спрашивает:

- «Почему только молоко?» — Ответ: «Это главный KPI» (но были **скрытые критерии**: цена, ремонтопригодность, стресс коров, поддержка локального сервиса).
- «Где записаны отвергнутые варианты?» — Ответ: «Удалили, они же хуже» (но они могли быть **разнообразными stepping stones** для будущего).
- «Как учитывались ограничения?» — Ответ: «Потом посмотрели» (но ConstraintFit должен быть **gate**, не afterthought).

Аудитор уходит ни с чем. Потому что выбор **скаляризовал разнородные качества**, потерял разнообразие и не зафиксировал policy. C.17–C.19 предлагают: возвращать front/archive, измерять по многим осям, управлять explore/explicit.

## 3. Search, Select, Novelty — полное описание

### 3.1 C.17 Creativity-CHR — измерение креативных исходов

**Вопрос:** Что значит «хорошее новое решение»?

**На ферме CreativitySpace может включать:**
1. **Novelty@context** — насколько решение отличается от ReferenceBase (0–1).
   - *Пример:* новый рацион с DCAD 350 mEq/kg, если база — 200–250.
2. **Use-Value (ValueGain)** — улучшение по declared objective.
   - *Пример:* +5% энергетического статуса в переходный период.
3. **Surprise** — неожиданность под GenerativePrior (bits/nats).
   - *Пример:* рацион, который модель не предсказывала, но оказался удачным.
4. **ConstraintFit** — доля must-constraints, которые выполнены (0–1).
   - *Пример:* рацион укладывается в бюджет, минеральные нормы, ограничения по сырью.
5. **Diversity_P** — на уровне портфеля: насколько разнообразны выбранные решения.
   - *Пример:* фронт из 4 рационов покрывает жаркую/холодную погоду, первотёлок/многотёлок.
6. **AttributionIntegrity** — признание источников, лицензии.
7. **EffortCost** — ресурсы на разработку (advisory).

**Ключевой признак:** Creativity-CHR **не экспортирует Γ-операторы и не говорит, как выбирать**. Он только предоставляет измерительный набор. Выбор — в C.18/C.19/G.5.

### 3.2 C.18 NQD-CAL — открытый поиск и архив

**Вопрос:** Как генерировать и сохранять разнообразных кандидатов?

**На ферме:**
- **DescriptorMap** — превращает кандидата в точку в CharacteristicSpace. Для рациона это может быть {NEL/СВ, %NDF, DCAD, цена}.
- **CandidateSet** — множество гипотез (рационов) с векторами {Q, N, D, S}.
- **Archive / Niche** — ячейки в пространстве характеристик, где хранятся элиты.
- **IlluminationSummary** — telemetry: coverage, QD-score, occupancy. **Не доминирует по умолчанию.**
- **Front** — недоминируемое множество по declared dominance set.

**Операторы Γ_nqd:**
- `generate(...)` — создаёт CandidateSet с учётом бюджета, EmitterPolicy, DescriptorMap.
- `updateArchive(...)` — обновляет архив по InsertionPolicy.
- `illuminate(...)` — возвращает telemetry.
- `selectFront(...)` — вычисляет Pareto-front по заданным характеристикам.

**Ключевой признак:** Pipeline: Eligibility (ConstraintFit=pass) → Dominance → Tie-breakers (Novelty, ΔDiversity, Surprise). Illumination — report-only, пока policy не разрешит иначе.

### 3.3 C.19 E/E-LOG — губернатор explore/exploit

**Вопрос:** Что делать с живым пулом кандидатов дальше?

**На ферме:**
- **EmitterPolicy** — именованная, версионированная политика: regimeKey (UCB, Thompson, ...), explore_share, temperature, backstop_confidence, cell_capacity K, dedup_threshold.
- **Lens** — объявляет hard eligibility, soft aggregation, trust policy. Любая скаляризация **должна быть записана по lens id**.
- **Pool-policy result** — явный результат: `widen`, `keep frontier`, `narrow to subset`, `sunset line`, `reroute`.
- **Change trigger** — событие, которое изменит treatment.

**Ключевой признак:** E/E-LOG **не делает локальный выбор** (это C.11) и **не публикует selector-facing shortlist** (это G.5). Он управляет пулом.

### 3.4 Золотое правило: нет скалярного победителя по умолчанию

**На ферме:**
- Если фронт решений по цене и BHB показывает компромисс, **нельзя** просто сложить взвешенную сумму без явной политики.
- Если один рацион дешёвый, другой энергетически плотный, третий устойчив к жаре — все три могут быть на фронте.
- Выбор одного — это **DecisionSubject** с явным lens id и policy.

**Ключевой признак:** «Лучшее» — это не одно число, а осознанный выбор из front с явными trade-offs.

## 4. Каскад типичных ошибок

Возьмём снова:

> *«Робот A даёт 32 кг, робот B — 31 кг, робот C — 30 кг. Выбираем A.»*

**Разложение по C.17–C.19:**

| Часть утверждения | Что это в FPF | Почему ошибка |
|---|---|---|
| «32 кг, 31 кг, 30 кг» | **Use-Value по одной оси** | Игнорируются другие критерии |
| «Выбираем A» | **Скалярный выбор без lens/policy id** | Нет явного DecisionSubject и policy |
| Удаление B и C | **Потеря Diversity_P / archive** | Отвергнутые варианты не сохранены как exploration archive |
| «Главный KPI» | **Скрытая скаляризация** | Нет объявленного dominance set |
| Отсутствие ограничений | **ConstraintFit не как gate** | Ремонтопригодность, сервис, стресс — afterthought |

**Что плохого в смешивании:**

1. **Ложная оптимальность.** Победитель по одной оси может проиграть по жизненному циклу.
2. **Потеря знаний.** Варианты B и C могли пригодиться при смене контекста.
3. **Невозможность объяснить выбор.** Аудитор не видит trade-offs и policy.

## 5. Как это выглядит на ферме: правильное применение

Перепишем выбор системы доения.

**Было (смешанное):**
> «Робот A — 32 кг, B — 31 кг, C — 30 кг. Выбираем A.»

**Стало (разложенное):**

**Context:** `MilkingSystemSelection_2026`

**CreativitySpace:**
- Q1: DailyYield [кг/день, ratio, ↑]
- Q2: Capex [руб, ratio, ↓]
- Q3: ServiceAvailability [часы простоя/год, ratio, ↓]
- Q4: CowStressIndex [ordinal, ↓]
- N: Novelty@context [0–1, ↑]
- ConstraintFit: {must: local service center <100 km, must: compatible with existing software}

**CandidateSet:**
| Робот | Q1 | Q2 | Q3 | Q4 | N | ConstraintFit |
|---|---|---|---|---|---|---|
| A | 32 | 12M | 80 | 3 | 0,2 | pass |
| B | 31 | 9M | 40 | 2 | 0,4 | pass |
| C | 30 | 8M | 30 | 2 | 0,6 | pass |
| D | 28 | 7M | 25 | 1 | 0,8 | fail (нет сервиса) |

**Dominance:** A не доминирует B (хуже capex и stress). B и C — на фронте.

**E/E-LOG pool-policy result:**
```
livePool = {A, B, C}
governingLens = multi_objective_dairy_lens_v2
currentTreatment = keep_frontier
changeTrigger = пилотная установка B и C на 3 месяца даёт backstop_confidence L1
whyNotLocalChoice = trade-offs по capex/stress/yield ещё не разрешены
```

**Что это даёт:**
- Ферма не спешит выбрать «победителя» по одной оси.
- Варианты A, B, C остаются живыми; D отсеян по ConstraintFit.
- Решение будет принято после пилота с явным lens id и policy.

## 6. Практическое применение: с чего начать

**Шаг 1.** Определите проблему поиска: рацион, оборудование, протокол, лекарственная схема.

**Шаг 2.** Объявите CreativitySpace: 2–4 quality-оси + Novelty + ConstraintFit.

**Шаг 3.** Сгенерируйте CandidateSet. Не обязательно автоматически — можно вручную, но с явными векторами.

**Шаг 4.** Примените ConstraintFit как hard gate.

**Шаг 5.** Вычислите Pareto-front по declared dominance set.

**Шаг 6.** Запишите E/E-LOG policy: keep frontier / narrow / widen / sunset / reroute.

**Шаг 7.** Если выбираете одно решение, укажите lens id, DecisionSubject и trade-offs.

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| Есть ли «оптимальный» вариант без объявленных критериев? | Скрытая скаляризация |
| Сравниваете ли вы разнородные шкалы взвешенной суммой? | Незаконная агрегация |
| Сохраняете ли вы отвергнутые варианты как archive? | Потеря Diversity_P |
| Входит ли Illumination в dominance без policy id? | Telemetry → dominance creep |
| Есть ли explicit pool-policy result с change trigger? | Незавершённый E/E-LOG |
