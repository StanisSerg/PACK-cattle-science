# C.26 — Cross-Context Transport

## 1. Зачем это читать

Если вы когда-нибудь видели, как на ферме **копируют рацион или протокол с соседнего хозяйства**, не спросив: «А у нас такие же коровы, корм, климат, персонал?» — вы встретили C.26 «в диком виде».

> **FPF-тезис:** *«Решение из одного контекста нельзя переносить в другой напрямую. Нужен явный Bridge, оценка Congruence Level (CL), пересчёт Scope и Reliability (R_eff), и учёт Bridge-only reuse: не всё, что работает в одном месте, можно адаптировать.»*

C.26 в реестре называется **Cross-Context Transport**. В спецификации FPF этот паттерн часто обсуждается через призму «Quantum-Like Modeling Lens» (C.2.2 / F.9): идея о том, что знание «работает» только относительно контекста измерения/применения. Здесь мы используем понятный фермерский язык: перенос решений между фермами.

**Фермерский пример:**

> Ферма Б решила скопировать рацион переходного периода с фермы А, где он «хорошо себя показал». Но у фермы А другой климат (холоднее зимой), другой состав силоса (кукурузный vs травяной), другой генотип коров и другой уровень персонала. Через месяц у фермы Б вырос BHB. Почему? Потому что рацион перенесли без Bridge и без адаптации.

---

## 2. История одной ошибки

Представьте ферму, которая покупает **программный модуль прогноза кетоза**:

> «У фермы А этот модель сработала. Поставим и к нам.»

Через год приходит аудитор. Он спрашивает:

- «Есть ли Bridge между фермой А и вашей?» — Ответ: «Нет, это же одна порода» (но одной породы мало).
- «Какой Congruence Level?» — Ответ: «Не считали» (CL влияет на R_eff).
- «Какие признаки модели использует?» — Ответ: «Те же, что и у фермы А» (но DataShape может отличаться).
- «Какой ScopeSlice(G) для вашей фермы?» — Ответ: «Все коровы» (слишком широко).
- «Есть ли Bridge-only reuse: что нельзя перенести?» — Ответ: «Не думали» (некоторые элементы не адаптируются).

Аудитор уходит ни с чем. Потому что перенос был «копипастой». C.26 предлагает: Bridge, CL, R_eff, ScopeSlice(G)' = translate(Bridge, G).

## 3. Cross-Context Transport — полное описание

### 3.1 Почему «Quantum-Like Modeling Lens»?

В FPF C.2.2 и F.9 используется аналогия с квантовой механикой: **знание не существует «в чистом виде», оно определяется контекстом измерения/применения**. Как частица ведёт себя как волна или корпускула в зависимости от эксперимента, так и решение «работает» или «не работает» в зависимости от контекста.

**На ферме:**
- Рацион «работает» на ферме А с кукурузным силосом.
- Тот же рацион «не работает» на ферме Б с травяным силосом.
- Это не значит, что рацион «плохой» или «хороший» сам по себе. Его эффективность — свойство контекста.

Поэтому в реестре этот паттерн назван **Cross-Context Transport**: речь о том, как безопасно переносить решения между контекстами, понимая, что «перевозка» меняет смысл.

### 3.2 Bridge

**Bridge** — это явное описание соответствия между двумя контекстами.

**На ферме:**
```
Bridge: FarmA → FarmB
- climate_zone: temperate_continental → temperate_continental (same)
- silage_type: corn → grass (diff)
- breed: Holstein → Holstein (same)
- avg_milk_yield: 38 kg → 32 kg (diff)
- staff_training: high → medium (diff)
```

**Ключевой признак:** Bridge не делает решения идентичными. Он делает различия явными.

### 3.3 Congruence Level (CL)

**CL** — степень соответствия двух контекстов по критическим измерениям.

**На ферме:**
- **CL=4 (near-clone):** одна порода, один климат, один тип корма, схожий yield, один персонал.
- **CL=3 (calibrated sibling):** различия есть, но Bridge позволяет пересчитать параметры.
- **CL=2 (adaptable cousin):** существенные различия; нужна адаптация и пилот.
- **CL=1 (distant analogue):** только высокоуровневые принципы переносимы.
- **CL=0 (incommensurable):** перенос незаконен.

**Ключевой признак:** CL определяется по declared dimensions, не по интуиции.

### 3.4 R_eff = effective reliability

**R_eff = R_base × Φ(CL)**

**На ферме:**
- Модель на ферме А имеет R_base = 0,87.
- Bridge FarmA→FarmB имеет CL=2.
- Φ(CL=2) = 0,6.
- R_eff = 0,87 × 0,6 = 0,52.

**Ключевой признак:** Нельзя использовать R_base в новом контексте без пересчёта.

### 3.5 ScopeSlice(G)' = translate(Bridge, G)

**На ферме:**
- Scope на ферме А: коровы ≥3-й лактации, кукурузный силос, yield ≥35 кг.
- Bridge показывает: на ферме Б нет кукурузного силоса, средний yield ниже.
- Scope' на ферме Б: коровы ≥3-й лактации, травяной силос, yield ≥28 кг.

**Ключевой признак:** Scope сужается или пересчитывается, но не расширяется без новых данных.

### 3.6 Bridge-only reuse

**Некоторые элементы нельзя адаптировать.** Их можно только использовать как есть или отказаться.

**На ферме:**
- Принцип «контроль DCAD в переходный период» — переносим (Bridge=principle).
- Конкретное значение DCAD=350 mEq/kg — требует адаптации (Bridge=parameter).
- Поставщик силоса с фермы А — не переносим (Bridge=none).

**Ключевой признак:** Для каждого элемента решения указывается Bridge type: none / parameter / principle / full.

## 4. Почему смешивать / игнорировать — значит рисковать

Возьмём снова:

> *«У фермы А этот модель сработала. Поставим и к нам.»*

**Разложение по C.26:**

| Часть утверждения | Что это в FPF | Почему ошибка |
|---|---|---|
| «У фермы А» | **Source context** | Нет Bridge к target context |
| «сработала» | **R_base** | Не пересчитано в R_eff |
| «Поставим и к нам» | **Direct transfer** | Нет translate(Bridge, G) |
| «одна порода» | **Ad-hoc CL** | CL должна быть по declared dimensions |

**Что плохого в смешивании:**

1. **Ложная уверенность.** R_base кажется высоким, но R_eff может быть низким.
2. **Неправильный Scope.** Модель применяется к коровам, для которых она не обучена.
3. **Потеря адаптивных элементов.** Конкретные параметры не пересчитаны.

## 5. Как это выглядит на ферме: правильное применение

Перепишем перенос модели прогноза кетоза.

**Было (смешанное):**
> «У фермы А этот модель сработала. Поставим и к нам.»

**Стало (разложенное):**

**Bridge FarmA → FarmB_v1:**
| Dimension | FarmA | FarmB | Match |
|---|---|---|---|
| breed | Holstein | Holstein | yes |
| climate | continental | continental | yes |
| silage | corn | grass | no |
| avg_yield | 38 kg | 32 kg | partial |
| DIM_at_pred | 0–21 | 0–21 | yes |
| features | BHB, BCS, yield, DIM | same | yes |
| data_quality | high | medium | partial |

**CL assessment:**
> Critical dimensions: breed, DIM, features — match. Silage, yield, data_quality — partial. Declared CL = 2 (adaptable cousin).

**R_eff calculation:**
> R_base = 0,87 (FarmA). Φ(CL=2) = 0,65. R_eff = 0,87 × 0,65 = 0,57.

**ScopeSlice(G)':**
> translate(Bridge, G): коровы FarmB, 0–21 DIM, доступные признаки BHB/BCS/yield/DIM, исключая сильные отклонения по силосу (требуется калибровка).

**Bridge-only reuse:**
- Model architecture — principle (переносим).
- Feature weights — parameter (требуют fine-tuning).
- Threshold BHB ≥1,2 — parameter (требует калибровки).
- Vendor contract — none (не переносим).

**Policy:**
> Deploy as sandbox for 3 months; collect FarmB labels; fine-tune; re-evaluate CL and R_eff quarterly.

**Что это даёт:**
- Перенос не «копипаста», а осознанная адаптация.
- R_eff честно отражает сниженную надёжность.
- Scope' ограничивает применение до безопасной зоны.
- Bridge-only reuse защищает от переноса неадаптируемых элементов.

## 6. Практическое применение: с чего начать

**Шаг 1.** Для любого «заимствованного» решения постройте Bridge: source → target.

**Шаг 2.** Определите declared dimensions и вычислите CL.

**Шаг 3.** Пересчитайте R_eff = R_base × Φ(CL).

**Шаг 4.** Пересчитайте ScopeSlice(G)' = translate(Bridge, G).

**Шаг 5.** Для каждого элемента решения укажите Bridge type: none / parameter / principle / full.

**Шаг 6.** Установите policy: sandbox → calibration → production.

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| Копируете ли решения без явного Bridge? | Прямой перенос |
| Используете ли R_base в новом контексте? | Непересчитанная надёжность |
| Определяете ли CL интуитивно? | Ad-hoc congruence |
| Расширяете ли Scope без данных? | Scope inflation |
| Переносите ли параметры без адаптации? | Bridge-only reuse нарушен |
