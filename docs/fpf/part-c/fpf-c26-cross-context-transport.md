# C.26 — Cross-Context Transport: перенос решений между фермами без потери смысла

> **Цель capture:** объяснить, почему решение из одного контекста нельзя переносить в другой напрямую, и как в FPF организован безопасный перенос через Bridge, Congruence Level и пересчёт Scope и Reliability.

---

## 1. Зачем это читать

Если на ферме копируют рацион или протокол с соседнего хозяйства, не спросив: «А у нас такие же коровы, корм, климат, персонал?» — это C.26 «в диком виде». В FPF знание не существует «в чистом виде»: его применимость определяется контекстом. Cross-Context Transport учит явно описывать соответствие между контекстами, пересчитывать надёжность и сужать scope.

> **FPF-тезис:** *«Решение из одного контекста нельзя переносить в другой напрямую. Нужен явный Bridge, оценка Congruence Level (CL), пересчёт Scope и Reliability (R_eff), и учёт Bridge-only reuse: не всё, что работает в одном месте, можно адаптировать.»*

**Фермерский пример:**

> Ферма Б решила скопировать рацион переходного периода с фермы А, где он «хорошо себя показал». Но у фермы А другой климат (холоднее зимой), другой состав силоса (кукурузный vs травяной), другой генотип коров и другой уровень персонала. Через месяц у фермы Б вырос BHB. Почему? Потому что рацион перенесли без Bridge и без адаптации.

---

## 2. История одной ошибки

Ферма купила программный модуль прогноза кетоза у вендора. Аргумент: «У фермы А эта модель сработала, поставим и к нам». При внедрении не был построен Bridge между контекстом вендора и собственной фермой, не вычислен CL, не пересчитан R_eff. Модель использовала признаки, которые на ферме измерялись другим методом, а порог BHB был взят из исследования на другой породе. Через год модель показывала много ложных срабатываний. Оказалось, что «сработала» означало `R_base = 0,87` в чужом контексте, а `R_eff` в целевом контексте был значительно ниже.

## 3. Cross-Context Transport — полное описание

### 3.1 Определение

C.26 в спецификации FPF называется Quantum-Like Modeling Lens: знание «работает» только относительно контекста применения. Для практики это означает: перенос решения между фермами требует явного Bridge, declared dimensions, Congruence Level, пересчёта Scope и R_eff, а также Bridge-only reuse для каждого элемента решения.

### 3.2 Почему это важно

Прямое копирование создаёт ложную уверенность, неправильный scope и потерю адаптивных элементов. Bridge делает различия явными, CL честно штрафует надёжность, а Bridge-only reuse защищает от переноса того, что адаптировать нельзя.

### 3.3 Bridge

**Определение.** Bridge — это явное описание соответствия между source context и target context по declared dimensions.

**Пояснение.** Bridge не делает контексты идентичными. Он делает различия видимыми и позволяет пересчитать Scope и R. Каждая dimension оценивается как same, diff или partial.

**Пример из животноводства.**

```text
Bridge: FarmA → FarmB
- climate_zone: temperate_continental → temperate_continental (same)
- silage_type: corn → grass (diff)
- breed: Holstein → Holstein (same)
- avg_milk_yield: 38 kg → 32 kg (partial)
- staff_training: high → medium (diff)
```

**Ключевой признак.** Bridge содержит declared dimensions и оценку совпадения для каждой.

### 3.4 Congruence Level (CL)

**Определение.** CL — это степень соответствия двух контекстов по критическим измерениям. FPF использует шкалу 0…4: 4 — near-clone, 3 — calibrated sibling, 2 — adaptable cousin, 1 — distant analogue, 0 — incommensurable.

**Пояснение.** CL определяется по declared dimensions, а не по интуиции. Одинаковая порода не гарантирует высокий CL, если отличаются рацион, климат и данные.

**Пример из животноводства.** Модель на ферме А и ферме Б совпадают по породе и DIM, но различаются по типу силоса, среднему удою и качеству данных. Объявленный CL = 2 (adaptable cousin).

**Ключевой признак.** CL не выводится из одного признака — он требует многостороннего сравнения.

### 3.5 R_eff = effective reliability

**Определение.** `R_eff = R_base × Φ(CL)`. Эффективная надёжность в целевом контексте всегда меньше или равна `R_base`.

**Пояснение.** Нельзя использовать `R_base` в новом контексте без пересчёта. Φ(CL) — явная функция штрафа, зависящая от Bridge и policy.

**Пример из животноводства.** Модель на ферме А имеет `R_base = 0,87`. Bridge FarmA→FarmB имеет CL = 2, `Φ(CL=2) = 0,6`. Тогда `R_eff = 0,87 × 0,6 = 0,52`.

**Ключевой признак.** Если в новом контексте используется `R_base` — это overread.

### 3.6 ScopeSlice(G)' = translate(Bridge, G)

**Определение.** Scope в целевом контексте получается трансляцией исходного Scope через Bridge. Scope может сужаться или пересчитываться, но не расширяется без новых данных.

**Пояснение.** Если в целевом контексте нет какого-то сырья или признака, соответствующая часть G исключается. Расширение G требует нового evidence.

**Пример из животноводства.** Scope на ферме А: коровы ≥ 3-й лактации, кукурузный силос, удой ≥ 35 кг. Bridge показывает: на ферме Б нет кукурузного силоса, средний удой ниже. Scope' на ферме Б: коровы ≥ 3-й лактации, травяной силос, удой ≥ 28 кг, исключая сильные отклонения по силосу.

**Ключевой признак.** Scope' всегда сужается или пересчитывается, но не расширяется без данных.

### 3.7 Bridge-only reuse

**Определение.** Некоторые элементы решения нельзя адаптировать. Их можно использовать только как есть (principle/full) или отказаться (none). Для каждого элемента указывается Bridge type.

**Пояснение.** Принцип «контроль DCAD в переходный период» переносим как principle. Конкретное значение DCAD = 350 mEq/kg требует адаптации — parameter. Поставщик силоса с фермы А не переносится — none.

**Пример из животноводства.** Для модели прогноза кетоза: архитектура модели — principle; веса признаков — parameter; порог BHB — parameter; вендорский договор — none.

**Ключевой признак.** Для каждого элемента решения указано: none / parameter / principle / full.

## 4. Почему смешивать / игнорировать — значит рисковать

Рассмотрим типичное смешанное утверждение:

> *«У фермы А эта модель сработала. Поставим и к нам.»*

**Разложение по C.26:**

| Часть утверждения | Что это в FPF | Почему важно разделять |
|---|---|---|
| «У фермы А» | source context | Нет Bridge к target context |
| «сработала» | R_base | Не пересчитано в R_eff |
| «Поставим и к нам» | direct transfer | Нет translate(Bridge, G) |
| «одна порода» | ad-hoc CL | CL должна быть по declared dimensions |

**Основные риски смешивания:**

1. **Ложная уверенность.** R_base кажется высоким, но R_eff может быть низким.
2. **Неправильный Scope.** Модель применяется к коровам, для которых она не обучена.
3. **Потеря адаптивных элементов.** Конкретные параметры не пересчитаны.

## 5. Как это выглядит на ферме: правильное применение

**Ситуация:** ферма Б хочет перенести модель прогноза кетоза с фермы А.

**Было (смешанное / нечёткое):**

> «У фермы А эта модель сработала. Поставим и к нам.»

**Стало (разложенное / ясное):**

**Bridge FarmA → FarmB_v1:**

| Dimension | FarmA | FarmB | Match |
|---|---|---|---|
| breed | Holstein | Holstein | yes |
| climate | continental | continental | yes |
| silage | corn | grass | no |
| avg_yield | 38 kg | 32 kg | partial |
| DIM_at_pred | 0–21 | 0–21 | yes |
| features | BHB, BCS, yield, DIM | same | yes |
| data_quality | high | medium | partial |

**CL assessment:**
> Critical dimensions: breed, DIM, features — match. Silage, yield, data_quality — partial. Declared CL = 2 (adaptable cousin).

**R_eff calculation:**
> R_base = 0,87 (FarmA). Φ(CL=2) = 0,65. R_eff = 0,87 × 0,65 = 0,57.

**ScopeSlice(G)':**
> translate(Bridge, G): коровы FarmB, 0–21 DIM, доступные признаки BHB/BCS/yield/DIM, исключая сильные отклонения по силосу (требуется калибровка).

**Bridge-only reuse:**
> - Model architecture — principle (переносим).
> - Feature weights — parameter (требуют fine-tuning).
> - Threshold BHB ≥ 1,2 — parameter (требует калибровки).
> - Vendor contract — none (не переносим).

**Policy:**
> Deploy as sandbox for 3 months; collect FarmB labels; fine-tune; re-evaluate CL and R_eff quarterly.

**Результат:**
- Перенос не «копипаста», а осознанная адаптация.
- R_eff честно отражает сниженную надёжность.
- Scope' ограничивает применение до безопасной зоны.
- Bridge-only reuse защищает от переноса неадаптируемых элементов.

## 6. Практическое применение: с чего начать

**Шаг 1.** Для любого «заимствованного» решения постройте Bridge: source → target.

**Шаг 2.** Определите declared dimensions и вычислите CL.

**Шаг 3.** Пересчитайте R_eff = R_base × Φ(CL).

**Шаг 4.** Пересчитайте ScopeSlice(G)' = translate(Bridge, G).

**Шаг 5.** Для каждого элемента решения укажите Bridge type: none / parameter / principle / full.

**Шаг 6.** Установите policy: sandbox → calibration → production.

## 9. Что запомнить

1. Решение не существует вне контекста.
2. Перенос требует Bridge, CL, R_eff и translate(Bridge, G).
3. R_eff всегда ≤ R_base.
4. Scope' сужается, не расширяется без данных.
5. Не всё переносится: указывайте Bridge type для каждого элемента.
