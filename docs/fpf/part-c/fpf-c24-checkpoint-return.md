# C.24 — CheckpointReturn

## 1. Зачем это читать

Если вы когда-нибудь видели, как на ферме **робот даёт рекомендацию, ветеринар её переписывает от руки, а зоотехник потом вносит в Excel третью версию** — вы встретили C.24 «в диком виде».

> **FPF-тезис:** *«Когда человек и агент совместно используют инструменты, любой шаг должен иметь чёткую роль (H/A/T), чёткую передачу и проверку обратимости. Нельзя, чтобы агент действовал «через» человека или человек «дописывал» решение агента без явного CAL-протокола.»*

C.24 — Agent-Tools CAL. Это архитектурный паттерн для совместной работы человека (H), агента/модели (A) и инструмента (T) в одном pipeline. Ключевое требование: **явная схема ролей, permission, handoff и rollback**.

**Фермерский пример:**

> На ферме установлен доильный робот с системой поддержки принятия решений. Робот предлагает отделить корову 47 из-за подозрения на мастит. Оператор подтверждает. Ветеринар осматривает и решает, что это ложное срабатывание. Но в системе остаётся отметка «отделена», потому что не было явного rollback-канала. C.24 предписывает: каждое действие агента должно быть reversable и иметь чёткого accountable party.

---

## 2. История одной ошибки

Представьте ферму, где агент-диетолог **генерирует рацион**:

> «Агент предложил рацион. Мы его немного подправили и ввели в кормосмеситель.»

Через месяц приходит аудитор. Он спрашивает:

- «Кто автор финального рациона?» — Ответ: «Ну, мы все» (но у final action должен быть единый accountable party).
- «Был ли шаг агента обратим?» — Ответ: «Нет, мы сразу перезаписали» (нарушение reversibility).
- «Есть ли журнал handoff?» — Ответ: «Только финальная версия» (нет audit trail).
- «Агент действовал от своего имени или как proxy человека?» — Ответ: «Не поняли вопрос» (proxy confusion).

Аудитор уходит ни с чем. Потому что H/A/T работали в одной куче. C.24 предлагает: роли, permissions, handoffs, rollback.

## 3. Agent-Tools-CAL — полное описание

### 3.1 Три актора

**H — Human / accountable operator**
- Может принимать binding decisions ( approve / reject / escalate ).
- Может отменить действие агента.
- Несёт ответственность за final action.

**A — Agent / model / algorithm**
- Генерирует candidate outputs.
- Не выполняет binding actions в production без H-approve (если policy так установлено).
- Должен self-report confidence, source, uncertainty.

**T — Tool / execution system**
- Выполняет действия по одобренной команде.
- Проверяет pre-conditions.
- Логирует исполнение и обратимость.

**На ферме:**
- **H** — ветеринар, зоотехник, менеджер.
- **A** — ML-модель, DSS, рекомендательная система.
- **T** — доильный робот, кормосмеситель, программа учёта, сенсоры.

### 3.2 Три типа handoff

**H→A:**
- Человек делегирует задачу агенту с явным scope.
- Пример: «Агент, предложи 3 варианта рациона для первотёлок с BHB <0,8 ммоль/л».

**A→H:**
- Агент возвращает candidate outputs с уверенностью, источниками и ограничениями.
- Пример: «Рекомендация: увеличить DCAD до 350. Confidence: 0,72. Источник: модель M_v3, калибровка Q2-2026».

**H→T:**
- Человек одобряет исполнение инструментом.
- Пример: «Подтверждаю загрузку рациона R-17 в кормосмеситель».

**A→T напрямую запрещён**, если policy не разрешает autonomous execution.

### 3.3 Permission matrix

| Действие | H | A | T |
|---|---|---|---|
| Генерация вариантов | — | ✓ | — |
| Одобрение binding action | ✓ | — | — |
| Исполнение команды | — | — | ✓ |
| Отмена/rollback | ✓ | по policy | ✓ |
| Изменение policy | ✓ | — | — |

### 3.4 Reversibility и rollback

**На ферме:**
- Если кормосмеситель загрузил неправильный рацион, должна быть процедура отката: пустая загрузка, ручная коррекция, маркировка партии.
- Если робот отделил корову ошибочно, должна быть кнопка «вернуть в группу» с записью причины.
- Любое действие агента, которое нельзя отменить, должно быть либо sandbox, либо с H-approve.

### 3.5 Proxy confusion — агент не человек

**Ошибка:**
> «Врач сказал роботу, а робот сделал.»

**Правильно:**
> «Врач одобрил рекомендацию агента; инструмент исполнил одобренную команду.»

Агент не «действует от имени врача». Агент предлагает, человек одобряет, инструмент исполняет.

### 3.6 Audit trail

Каждый handoff должен журналировать:
- timestamp;
- actor (H/A/T) и role;
- action type (propose / approve / execute / reject / rollback);
- payload summary;
- policy id / version;
- confidence / uncertainty (если A);
- pre-conditions check (если T).

## 4. Каскад типичных ошибок

Возьмём снова:

> *«Агент предложил рацион. Мы его немного подправили и ввели в кормосмеситель.»*

**Разложение по C.24:**

| Часть утверждения | Что это в FPF | Почему ошибка |
|---|---|---|
| «Агент предложил» | **A→H handoff** | Хорошо, но неизвестно, что передано |
| «мы немного подправили» | **H overrides A без явного diff** | Нет версионирования, нет rollback |
| «ввели в кормосмеситель» | **H→T handoff без чёткого approve** | Нет policy id, нет pre-conditions check |
| «мы все» авторы | **Нет accountable party** | Final action должен иметь единого H-owner |

**Что плохого в смешивании:**

1. **Нет ответственности.** Если рацион окажется плохим, непонятно, кто принял решение.
2. **Нет отката.** «Немного подправили» не восстановить.
3. **Нет аудита.** Повторить инцидент и научиться невозможно.

## 5. Как это выглядит на ферме: правильное применение

Перепишем сценарий с рационом.

**Было (смешанное):**
> «Агент предложил рацион. Мы его немного подправили и ввели в кормосмеситель.»

**Стало (разложенное):**

**H→A:**
> Зоотехник делегирует: «Агент, сгенерируй 3 кандидата рациона для группы «Переходный период, HerdA» с ограничениями: цена ≤ 450 руб/голова, NEL ≥ 1,55, DCAD 300–400.»

**A→H:**
> Агент возвращает:
> - R-17a: цена 440, NEL 1,58, DCAD 320, confidence 0,82, source: model_R3_train_2026Q2.
> - R-17b: цена 445, NEL 1,60, DCAD 360, confidence 0,75, source: model_R3_train_2026Q2.
> - R-17c: цена 430, NEL 1,56, DCAD 340, confidence 0,68, source: heuristic_baseline.
> - Limitation note: нет данных по жаркому сезону.

**H review:**
> Зоотехник выбирает R-17b, меняет DCAD с 360 на 350 с причиной: «у фермы исторически меньше мочевого синдрома при 350». Создаётся diff: `R-17b → R-17b-H1`.

**H→T:**
> Зоотехник подтверждает загрузку R-17b-H1 в кормосмеситель C-2.
> Pre-condition check: кормосмеситель свободен, ингредиенты на складе.
> T исполняет, логирует batch_id, timestamp, H-owner.

**Rollback:**
> Если через 3 дня BHB растёт, H может активировать `rollback_to(R-17b)` или вернуться к предыдущему рациону R-16.

**Что это даёт:**
- Ясная цепочка H→A→H→T.
- Версионирование и diff.
- Ответственность и откат.
- Полный audit trail.

## 6. Практическое применение: с чего начать

**Шаг 1.** Для каждого workflow определите: где H, где A, где T.

**Шаг 2.** Напишите permission matrix: кто может что.

**Шаг 3.** Определите, какие handoffs разрешены. Запретите A→T без H-approve, если нет autonomous policy.

**Шаг 4.** Добавьте reversibility: для каждого binding action — процедура rollback.

**Шаг 5.** Ведите audit trail для каждого handoff.

**Шаг 6.** Периодически проверяйте: нет ли proxy confusion («агент сделал, потому что врач сказал»).

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| Агент может напрямую влиять на инструмент? | A→T без H-approve |
| Решение принимается «коллективно» без H-owner? | Нет accountable party |
| Есть ли diff при правках человеком? | Потеря версионирования |
| Можно ли откатить действие агента? | Отсутствие reversibility |
| Ведётся ли журнал handoff? | Нет audit trail |
