# F.18 — Interface Semantics Discipline

## 1. Зачем это читать

Если вы когда-нибудь слышали на ферме фразу *«API даёт нам данные по молоку»* или *«сервис ветеринарного учёта обещает контроль здоровья»* — вы уже встретились с проблемой, которую решает F.18.

> **FPF-тезис:** *«API, интерфейсы и сервисы — не просто точки входа. Это семантические границы: каждый вызов что-то означает, создаёт обязательства и требует предпосылок.»*

В современном молочном скотоводстве ферма всё чаще состоит из связанных систем:
- доильный робот передаёт данные в учётную систему;
- лаборатория загружает результаты через API;
- ветеринарный сервис шлёт уведомления;
- кормовой консультант забирает данные через интеграцию.

Каждый такой «интерфейс» — это не техническая деталь. Это **семантическая граница**, где одна система обещает что-то другой. F.18 учит читать эти границы: что именно обещано, кто несёт ответственность, какие данные нужны для входа и какой результат ожидается на выходе.

**Фермерский пример:**

> Ферма подключила облачный сервис мониторинга mastitis. Интерфейс возвращает «риск высокий» для коровы 1847. Что это значит? Обещает ли сервис диагноз? Требует ли он подтверждения лабораторией? Кто отвечает, если риск окажется ложным? Без семантической дисциплины интерфейс превращается в чёрный ящик, который принимают за истину.

## 2. История одной ошибки

Представьте ферму, которая интегрировала кормовой калькулятор с лабораторией анализа кормов.

- **Интерфейс:** «POST /feed-analysis → возвращает состав корма».
- **Кормовой консультант** считает: «Это точные данные для рациона».
- **Лаборатория** считает: «Это предварительные данные с погрешностью ±10%».
- **Зоотехник** считает: «Это разрешение на кормление».
- **Аудитор** спрашивает: «Кто подтвердил, что корм соответствует SOP?»

Через месяц приходит инцидент:

- «API сказал, что Mg = 0.42%» — Ответ: «API вернул значение измерения» (но это было **Observation**, не **Standard**).
- «Почему рацион не скорректировали?» — Ответ: «Консультант думал, что данные точные» (но **CL/Loss** Bridge не был объявлен).
- «Кто отвечает за ошибку?» — Ответ: «Сервис» (но **PromiseContent** и **Commitment** не были разделены).
- «Какие предпословки нужны для вызова?» — Ответ: «Не знаем, просто вызываем» (нет **accessSpec** и **admissibility conditions**).

F.18 предлагает разложить «интерфейс» на семантические компоненты: PromiseContent, accessSpec, Commitment, Work, Evidence.

## 3. Interface Semantics — полное описание

В FPF интерфейс/сервис/API — это не монолит. Это **набор семантических граней**, каждая из которых отвечает на свой вопрос. Разложение следует за A.6.8 (Service Polysemy Unpacking) и A.6.C (Contract Unpacking), которые делают «сервисную» речь проверяемой.

### 3.1 PromiseContent — что обещано

**Вопрос:** *Что именно потребитель получит?*

**Пример:** «Сервис анализа молока обещает измерение соматических клеток с погрешностью ±5% в течение 24 ч после отбора пробы». Это **PromiseContent** — потребительское обещание + критерии приёмки.

**На ферме:**
- лаборатория обещает результаты анализа корма;
- ветеринарный сервис обещает уведомления о риске кетоза;
- доильный робот обещает данные удоя и проводимости;
- кормовой консультант обещает рекомендацию по рациону.

**Ключевой признак:** PromiseContent — это **внешнее обещание**, а не система, API или работа. Его можно нарушить или выполнить.

### 3.2 accessSpec — как обратиться

**Вопрос:** *Какие предпосылки и правила доступа?*

**Пример:** «Для вызова API нужна проба молока, отобранная по SOP-M-05, в ёмкости с идентификатором коровы и временем отбора». Это **accessSpec** — спецификация доступа.

**На ферме:**
- формат и качество входных данных;
- идентификация коровы/группы;
- временное окно, в котором данные актуальны;
- права и роли вызывающей стороны.

**Ключевой признак:** accessSpec — это MethodDescription для входа. Без соблюдения предпосылок вызов незаконен.

### 3.3 Commitment — кто отвечает

**Вопрос:** *Кто и чем обязан?*

**Пример:** «Лаборатория А (провайдер) обязуется вернуть результаты в 24 ч; ферма (потребитель) обязуется предоставить пробу по SOP». Это **Commitment** — деонтическая связь.

**На ферме:**
- лаборатория отвечает за точность анализа;
- ветеринар отвечает за трактовку риска;
- ферма отвечает за правильный отбор проб;
- поставщик ПО отвечает за доступность сервиса.

**Ключевой признак:** Commitment привязан к **роли** и может быть нарушен. «Сервис обещает» — метафора; в FPF обещает конкретная роль/организация.

### 3.4 Work — что делается

**Вопрос:** *Какая работа выполняется для исполнения обещания?*

**Пример:** «2026-06-18, лаборатория А провела анализ пробы №1847, потрачено 2 ч, реагенты X». Это **Work** — фактическое выполнение.

**На ферме:**
- лабораторный анализ пробы;
- расчёт рациона консультантом;
- обработка данных доильным роботом;
- ветеринарный осмотр по уведомлению сервиса.

**Ключевой признак:** Work — датированное событие. PromiseContent не выполняется сам по себе; его выполняет Work под Commitment.

### 3.5 Evidence — как проверить

**Вопрос:** *Какие доказательства подтверждают исполнение?*

**Пример:** «Результат анализа с идентификатором пробы, методом измерения, погрешностью и подписью оператора». Это **Evidence**.

**На ферме:**
- лабораторный протокол с калибровкой прибора;
- журнал Work с датой и исполнителем;
- измерение pH мочи как evidence для DCAD-работы;
- данные доильного робота как evidence для продуктивности.

**Ключевой признак:** Evidence привязано к Work и используется для adjudication — решения, выполнено ли PromiseContent.

## 4. Почему смешивать / игнорировать — значит рисковать

Возьмём обычное фермерское утверждение:

> *«Наш API молочной лаборатории гарантирует точный анализ.»*

**Разложение по F.18:**

| Часть утверждения | Что это в FPF | Почему рискованно |
|---|---|---|
| «API» | **access point / system facet** | API — не субъект, он не может «гарантировать» |
| «гарантирует» | **Commitment** | Но не назван committer (роль/организация) |
| «точный анализ» | **PromiseContent + acceptance criteria** | Нет шкалы точности и метода измерения |
| (недостаёт) | **Work + Evidence** | Нет требования предоставить датированную работу |

**Что плохого в смешивании:**

1. **Interface-as-agent.** «API обещает» — категориальная ошибка; обещают роли/организации.
2. **API = Service.** Точка доступа подменяет потребительское обещание.
3. **Read = Evidence.** Простой вызов API считается доказательством истины.
4. **No admissibility check.** Забывают проверить, были ли соблюдены предпословки входных данных.
5. **Responsibility gap.** При инциденте никто не может назвать committer.

## 5. Как это выглядит на ферме: правильное применение

Возьмём сервис: **облачный мониторинг риска мастита**.

**Было (нечёткое):**
> «Сервис mastitis даёт нам риск для каждой коровы. Мы действуем по его рекомендациям.»

**Стало (разложенное по F.18):**

**PromiseContent:**
> F.18.PC-001: Сервис предоставляет индикатор риска мастита в ближайшие 48 ч (scale: Low / Medium / High) на основе проводимости и удоя, но **не** диагноз.

**accessSpec:**
> F.18.AS-001: Входные данные — проводимость и удой за последние 3 дойки, идентификация коровы, калибровка прибора не старше 30 дней.

**Commitment:**
> F.18.CM-001: Провайдер «DairySense Oy» обязуется обновлять индикатор в течение 1 ч после каждой дойки. Ферма обязуется не использовать индикатор как единственное основание для антибиотикотерапии.

**Work:**
> F.18.W-001: 2026-06-18 14:23, сервис рассчитал индикатор для коровы 1847 на основе данных робота.

**Evidence:**
> F.18.EV-001: Исходные данные проводимости/удоя + алгоритм версии 2.1 + ссылка на валидационное исследование.

**Что это даёт:**
- Ветеринар знает: индикатор — не диагноз, а триггер для осмотра.
- Ферма знает: обязательства провайдера ограничены скоростью обновления.
- Аудитор видит: PromiseContent, accessSpec, Commitment, Work, Evidence разделены.
- Интегратор знает: какие данные нужны и в каком виде.

## 6. Практическое применение: с чего начать

**Шаг 1.** Выберите один внешний интерфейс на ферме (API лаборатории, сервис ветучёта, доильный робот, кормовой калькулятор).

**Шаг 2.** Ответьте на пять вопросов:
- **PromiseContent:** что именно обещано и с какими критериями?
- **accessSpec:** какие входные данные и предпосылки нужны?
- **Commitment:** кто и чем обязан?
- **Work:** какая работа выполняется для исполнения?
- **Evidence:** как проверить, что обещание выполнено?

**Шаг 3.** Для каждого вызова интерфейса проверьте:
- назван ли committer?
- заявлена ли шкала/точность результата?
- указаны ли предпосылки входных данных?
- есть ли ссылка на Work и Evidence?

**Шаг 4.** Перепишите SLA/API-документацию в терминах F.18: не «API возвращает X», а «при соблюдении accessSpec сервис возвращает PromiseContent, за который отвечает Commitment, подтверждаемое Work и Evidence».

**Шаг 5.** Проверьте cross-references: PromiseContent → Commitment → Work → Evidence → Bridge (если данные переносятся в другой контекст).

---

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| «API/sервис обещает» без указания организации/роли? | Interface-as-agent |
| Точка доступа (endpoint) называется «сервисом»? | API = Service |
| Результат API принимается как доказательство без валидации? | Read = Evidence |
| Нет описания, какие входные данные нужны? | Missing accessSpec |
| При инциденте непонятно, кто несёт ответственность? | Commitment gap |
