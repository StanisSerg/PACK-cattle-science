# F.18 — Interface Semantics Discipline: читай интерфейс как семантическую границу

> **Цель capture:** объяснить, как в FPF разбирают интерфейсы, API и сервисы на семантические компоненты, чтобы обещания, предпосылки, ответственность, работа и доказательства не смешивались.

---

## 1. Зачем это читать

Если на ферме говорят «API даёт нам данные по молоку» или «сервис ветеринарного учёта обещает контроль здоровья», — это уже интерфейсная семантика. В современном скотоводстве ферма состоит из связанных систем: доильные роботы, лаборатории, ветеринарные сервисы, кормовые калькуляторы. Каждый «интерфейс» — это не техническая деталь, а семантическая граница, где одна система что-то обещает другой. F.18 учит читать эту границу: что именно обещано, кто несёт ответственность, какие данные нужны для входа и какой результат ожидается на выходе.

> **FPF-тезис:** *«API, интерфейсы и сервисы — не просто точки входа. Это семантические границы: каждый вызов что-то означает, создаёт обязательства и требует предпосылок.»*

**Фермерский пример:**

> Ферма подключила облачный сервис мониторинга мастита. Интерфейс возвращает «риск высокий» для коровы 1847. Что это значит? Обещает ли сервис диагноз? Требует ли он подтверждения лабораторией? Кто отвечает, если риск окажется ложным? Без семантической дисциплины интерфейс превращается в чёрный ящик, который принимают за истину.

---

## 2. История одной ошибки

Ферма интегрировала кормовой калькулятор с лабораторией анализа кормов.

- **Интерфейс:** `POST /feed-analysis → возвращает состав корма`.
- **Кормовой консультант** считал: «Это точные данные для рациона».
- **Лаборатория** считала: «Это предварительные данные с погрешностью ±10 %».
- **Зоотехник** считал: «Это разрешение на кормление».
- **Аудитор** спросил: «Кто подтвердил, что корм соответствует SOP?»

Через месяц произошёл инцидент: API вернул Mg = 0,42 %, но это было Observation, а не Standard. Консультант принял значение как точное, не проверив Bridge/CL. При ошибке никто не мог назвать committer, потому что PromiseContent, accessSpec, Commitment, Work и Evidence не были разделены.

F.18 предлагает разложить «интерфейс» на пять семантических компонентов: `PromiseContent`, `accessSpec`, `Commitment`, `Work`, `Evidence`.

## 3. Interface Semantics — полное описание

### 3.1 Определение

**Interface Semantics Discipline (F.18)** — это паттерн разложения сервисных, API- и договорных утверждений на пять категорий: обещание (`PromiseContent`), предпосылки доступа (`accessSpec`), деонтическую связь (`Commitment`), фактическую работу (`Work`) и доказательство исполнения (`Evidence`).

### 3.2 Почему это важно

Интерфейс без семантической дисциплины порождает категориальные ошибки: «API обещает» (interface-as-agent), «точка доступа = сервис» (API = Service), «вызов API = доказательство истины» (Read = Evidence). Разложение делает каждое утверждение проверяемым и привязывает ответственность к роли.

### 3.3 PromiseContent — что обещано

**Определение.** `PromiseContent` — это потребительское обещание плюс критерии приёмки: что именно получит потребитель при соблюдении предпосылок.

**Пояснение.** PromiseContent — внешнее обещание, а не система, API или работа. Его можно выполнить или нарушить. В него входят шкала, точность, временное окно и явные исключения.

**Пример из животноводства.**

> `PC-MAST-001`: Сервис предоставляет индикатор риска мастита в ближайшие 48 ч (scale: Low / Medium / High) на основе проводимости и удоя, но **не** диагноз. Погрешность индикатора: sensitivity/specificity заявлены в валидационном исследовании v2.1.

**Ключевой признак.** PromiseContent содержит «что», «с какими критериями» и «что не обещано».

### 3.4 accessSpec — как обратиться

**Определение.** `accessSpec` — это спецификация доступа: какие входные данные, предпосылки, права и роли нужны, чтобы вызов был законным.

**Пояснение.** accessSpec — это MethodDescription для входа. Без соблюдения предпосылок вызов неadmissible. Включает формат данных, идентификацию, временное окно актуальности и калибровку.

**Пример из животноводства.**

> `AS-MAST-001`: Входные данные — проводимость и удой за последние 3 дойки, идентификация коровы, калибровка прибора не старше 30 дней, права `FarmA.DataFeedRole`.

**Ключевой признак.** accessSpec перечисляет предпосылки, без которых PromiseContent не активируется.

### 3.5 Commitment — кто отвечает

**Определение.** `Commitment` — это деонтическая связь: кто и чем обязан. Обещает не «API» и не «сервис», а конкретная роль или организация.

**Пояснение.** Commitment делит ответственность между провайдером и потребителем. Он может быть нарушен. Commitment не заменяет Work: обещание должно быть исполнено Work и подтверждено Evidence.

**Пример из животноводства.**

> `CM-MAST-001`: Провайдер «DairySense Oy` обязуется обновлять индикатор в течение 1 ч после каждой дойки. Ферма обязуется не использовать индикатор как единственное основание для антибиотикотерапии.

**Ключевой признак.** Commitment называет committer'а (роль/организацию) и конкретные обязательства с обеих сторон.

### 3.6 Work — что делается

**Определение.** `Work` — это датированное событие, в котором выполняется действие, необходимое для исполнения PromiseContent. PromiseContent не выполняется сам по себе; его выполняет Work под Commitment.

**Пояснение.** Work содержит performer, time window, ресурсы и связь с MethodDescription. Для сервиса Work — это вычисление, запись, передача. Для лаборатории — анализ пробы.

**Пример из животноводства.**

> `W-MAST-2026-0618-1423`: 18.06.2026 14:23, сервис рассчитал индикатор для коровы 1847 на основе данных робота; использован алгоритм v2.1, latency 23 с.

**Ключевой признак.** Work — датированное событие; его можно связать с Evidence и RoleAssignment.

### 3.7 Evidence — как проверить

**Определение.** `Evidence` — это доказательства, подтверждающие исполнение PromiseContent: исходные данные, журналы Work, калибровки, ссылки на валидационные исследования.

**Пояснение.** Evidence используется для adjudication — решения, выполнено ли обещание. Evidence привязано к Work и не заменяет PromiseContent.

**Пример из животноводства.**

> `EV-MAST-001`: Исходные данные проводимости/удоя за 3 дойки + идентификатор алгоритма v2.1 + ссылка на валидационное исследование `DairySense_Validation_2025` + журнал калибровки прибора.

**Ключевой признак.** Evidence позволяет независимо проверить, что PromiseContent было исполнено.

## 4. Почему смешивать / игнорировать — значит рисковать

Рассмотрим типичное смешанное утверждение:

> *«Наш API молочной лаборатории гарантирует точный анализ.»*

**Разложение по F.18:**

| Часть утверждения | Что это в FPF | Почему рискованно |
|---|---|---|
| «API» | access point / system facet | API — не субъект, он не может «гарантировать» |
| «гарантирует» | `Commitment` | Не назван committer (роль/организация) |
| «точный анализ» | `PromiseContent + acceptance criteria` | Нет шкалы точности и метода измерения |
| (недостаёт) | `Work + Evidence` | Нет требования предоставить датированную работу |

**Основные риски смешивания:**

1. **Interface-as-agent.** «API обещает» — категориальная ошибка; обещают роли/организации.
2. **API = Service.** Точка доступа подменяет потребительское обещание.
3. **Read = Evidence.** Простой вызов API считается доказательством истины.
4. **No admissibility check.** Забывают проверить, были ли соблюдены предпосылки входных данных.
5. **Responsibility gap.** При инциденте никто не может назвать committer.

## 5. Как это выглядит на ферме: правильное применение

**Ситуация:** облачный мониторинг риска мастита.

**Было (нечёткое):**

> «Сервис mastitis даёт нам риск для каждой коровы. Мы действуем по его рекомендациям.»

**Стало (разложенное по F.18):**

**PromiseContent:**
> `PC-MAST-001`: Сервис предоставляет индикатор риска мастита в ближайшие 48 ч (scale: Low / Medium / High) на основе проводимости и удоя, но **не** диагноз.

**accessSpec:**
> `AS-MAST-001`: Входные данные — проводимость и удой за последние 3 дойки, идентификация коровы, калибровка прибора не старше 30 дней.

**Commitment:**
> `CM-MAST-001`: Провайдер «DairySense Oy» обязуется обновлять индикатор в течение 1 ч после каждой дойки. Ферма обязуется не использовать индикатор как единственное основание для антибиотикотерапии.

**Work:**
> `W-MAST-2026-0618-1423`: 18.06.2026 14:23, сервис рассчитал индикатор для коровы 1847 на основе данных робота.

**Evidence:**
> `EV-MAST-001`: Исходные данные проводимости/удоя + алгоритм версии 2.1 + ссылка на валидационное исследование.

**Результат:**
- Ветеринар знает: индикатор — не диагноз, а триггер для осмотра.
- Ферма знает: обязательства провайдера ограничены скоростью обновления.
- Аудитор видит: PromiseContent, accessSpec, Commitment, Work, Evidence разделены.
- Интегратор знает: какие данные нужны и в каком виде.

## 6. Практическое применение: с чего начать

**Шаг 1.** Выберите один внешний интерфейс на ферме: API лаборатории, сервис ветучёта, доильный робот, кормовой калькулятор.

**Шаг 2.** Ответьте на пять вопросов:
- **PromiseContent:** что именно обещано и с какими критериями?
- **accessSpec:** какие входные данные и предпосылки нужны?
- **Commitment:** кто и чем обязан?
- **Work:** какая работа выполняется для исполнения?
- **Evidence:** как проверить, что обещание выполнено?

**Шаг 3.** Для каждого вызова интерфейса проверьте:
- назван ли committer?
- заявлена ли шкала/точность результата?
- указаны ли предпосылки входных данных?
- есть ли ссылка на Work и Evidence?

**Шаг 4.** Перепишите SLA/API-документацию в терминах F.18: не «API возвращает X», а «при соблюдении accessSpec сервис возвращает PromiseContent, за который отвечает Commitment, подтверждаемое Work и Evidence».

**Шаг 5.** Проверьте cross-references: PromiseContent → Commitment → Work → Evidence → Bridge (если данные переносятся в другой контекст).

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| «API/сервис обещает» без указания организации/роли? | Interface-as-agent |
| Точка доступа (endpoint) называется «сервисом»? | API = Service |
| Результат API принимается как доказательство без валидации? | Read = Evidence |
| Нет описания, какие входные данные нужны? | Missing accessSpec |
| При инциденте непонятно, кто несёт ответственность? | Commitment gap |
| PromiseContent не содержит явных исключений («не диагноз»)? | Overclaim risk |

## 9. Что запомнить

1. Интерфейс — это семантическая граница, а не просто endpoint.
2. Каждый интерфейс разлагается на PromiseContent, accessSpec, Commitment, Work, Evidence.
3. Обещает не API, а конкретная роль/организация.
4. Вызов API — не Evidence; Evidence проверяет исполнение PromiseContent.
5. Без accessSpec PromiseContent не активируется.
