# F.17 — Human-Facing Work Sheet: показывай работу, а не сырые данные

> **Цель capture:** объяснить, как в FPF превращают разрозненные записи о работе в один человекочитаемый лист, по которому можно принять решение.

---

## 1. Зачем это читать

Если ветеринар присылает SMS: «BHB 1,4 у коровы 1847», зоотехник не знает: это было вчера или позавчера? Из крови или молока? При каком пороге? Какое действие последовало? Цифра без контекста — просто цифра. F.17 учит собирать из сырых данных один лист, где каждая строка несёт смысловой каркас: объект, цель, метод, результат, исполнителя и триггер следующего шага.

> **FPF-тезис:** *«Человеку нужен не raw data, а осмысленный лист с контекстом: что делалось, почему, по какому методу, с каким результатом.»*

**Фермерский пример:**

> На ферме ежемесячно проводят копытную обработку. Через квартал аудитор спрашивает: «Что делалось 15 мая?» — ответ: «Есть список коров». «Почему корова 2312 пропущена?» — «В таблице нет отметки». «Какой метод?» — «Обычный». «Каков результат?» — «Все ходят». Без Work Sheet невозможно принять или проверить решение.

---

## 2. История одной ошибки

Ферма внедрила скрининг BHB. В таблице были только три колонки: дата, номер коровы, значение BHB.

| Дата | Корова | BHB |
|---|---|---|
| 18.06 | 1847 | 1,4 |
| 18.06 | 1851 | 0,9 |

Через две недели пришёл консультант и спросил: «Это из крови или молока? По какому SOP? Кто брал пробу? Какие коровы вошли в группу? Что было сделано с коровами ≥1,2?» Ответы были разрозненны: часть данных в голове у ветеринара, часть — в журнале кормовщика, часть — вообще не записана. В итоге отчёт оказался бесполезным для принятия решения.

F.17 предлагает заменить такую таблицу на **Work Sheet** — один лист, где каждая строка содержит контекст, достаточный для действия.

## 3. Unified Term Sheet / Human-Facing Work Sheet — полное описание

### 3.1 Определение

**Human-Facing Work Sheet (F.17)** — это человекочитаемая публикация записей `U.Work`, представленная в виде компактной таблицы. Каждая строка связывает объект работы, её цель, использованный MethodDescription, результат/наблюдение, исполнителя и роль, а также следующий шаг или статус.

### 3.2 Почему это важно

Raw data не позволяют проверить compliance, восстановить аудиторский след, связать расход с SOP или передать решение другому специалисту. Work Sheet делает Work видимым как датированное событие с контекстом, а не как набор цифр.

### 3.3 Объект работы — EntityOfConcern

**Определение.** Объект работы — это то, над чем выполнялась работа: конкретная корова, группа, секция, партия корма или оборудование. В FPF это `EntityOfConcern` Work.

**Пояснение.** Объект должен быть достаточно конкретным, чтобы можно было вернуться к исходным данным. «Группа 7» — недостаточно; нужно «группа Fresh, 52 коровы, DIM 3–7».

**Пример из животноводства.**

> `W-BHB-2026-0618-001` выполнялась над группой `FarmA.FreshCows (DIM 3–7, n=52)`. Для повторного замера — над конкретными коровами 1847, 1855, 1861, 1866.

**Ключевой признак.** Объект назван так, что по нему можно найти исходные данные без дополнительных вопросов.

### 3.4 Цель и контекст — Problem / DecisionSubject

**Определение.** Цель работы — это ответ на вопрос «зачем?». В FPF это `Problem` или `DecisionSubject`, который отделяет рутинную работу от внеплановой и позволяет проверить обоснованность.

**Пояснение.** Без явной цели строка превращается в «мы что-то делали». Цель помогает связать Work с последующим решением: лечение, коррекция рациона, ремонт.

**Пример из животноводства.**

> Цель `W-BHB-2026-0618-001`: профилактика субклинического кетоза в раннем периоде лактации. Цель повторного замера `W-BHB-2026-0620-002`: контроль эффективности лечения.

**Ключевой признак.** Каждая строка содержит одну конкретную цель, понятную без пояснений.

### 3.5 MethodDescription — по какому рецепту

**Определение.** `MethodDescription` — это ссылка на утверждённый рецепт или SOP, по которому выполнялась работа. В Work Sheet это не «как обычно делаем», а точная ссылка.

**Пояснение.** Ссылка на MethodDescription позволяет аудитору проверить, что работа была выполнена по утверждённой процедуре, а также понять, были ли отклонения.

**Пример из животноводства.**

> `SOP-BHB-01 v2`: отбор крови из хвостовой вены, прибор Xpress Vet, порог ≥1,2 ммоль/л. Для копытной обработки — `SOP-HoofTrim-04 v1`.

**Ключевой признак.** Каждая строка содержит идентификатор и версию SOP/MethodDescription.

### 3.6 Результат и статус — Observation / Outcome

**Определение.** Результат работы — это измеримое Observation, привязанное к шкале, методу измерения и временному окну (`Window`). Статус — это интерпретация результата относительно цели.

**Пояснение.** Результат должен быть пригоден для принятия решения: BHB = 1,4 ммоль/л с порогом ≥1,2, шкала ratio, метод Xpress Vet. Статус — «в группе риска», «лечение назначено», «повтор через 48 ч».

**Пример из животноводства.**

> Результат `W-BHB-2026-0618-001`: 8 из 52 коров с BHB ≥1,2 ммоль/л. Следующий шаг: пропиленгликоль 300 мл орально, повторный замер через 48 ч. Результат `W-BHB-2026-0620-002`: все повторные значения <1,0 ммоль/л. Статус: случай закрыт.

**Ключевой признак.** Результат содержит число, шкалу, метод, порог и триггер следующего действия.

### 3.7 Исполнитель и роль — RoleAssignment / Work

**Определение.** Каждая строка Work Sheet указывает, кто выполнял работу, под какой ролью, в какое время и с какими ресурсами. В FPF это связка `U.Work` + `U.RoleAssignment`.

**Пояснение.** Исполнитель и роль нужны для accountability и для понимания компетенции. «Ветеринар Петров» — holder; `VetFreshRole` — роль. Ресурсы: время, расходники, инструменты.

**Пример из животноводства.**

> `W-BHB-2026-0618-001`: 18.06.2026, 05:45–07:10, исполнитель Петров / `VetFreshRole`, использовано 54 тест-полоски, 2 набора реагентов.

**Ключевой признак.** В строке есть дата/время, имя/роль исполнителя и ключевые ресурсы.

## 4. Почему смешивать / игнорировать — значит рисковать

Рассмотрим типичную фермерскую таблицу:

> *«15.06: группа 7, MgO 8,4 кг, коров 120»*

**Разложение по F.17:**

| Часть утверждения | Что это в FPF | Почему недостаточно |
|---|---|---|
| «15.06» | `Work window` | Нет времени начала/окончания |
| «группа 7» | `EntityOfConcern` | Неясно, какие коровы входят |
| «MgO 8,4 кг» | `Resource consumption` | Неясно, доза на голову или общая масса |
| «коров 120» | `Scope count` | Нет MethodDescription и результата |

**Основные риски смешивания:**

1. **Loss of context.** Через месяц никто не вспомнит, почему была эта работа.
2. **False compliance.** Наличие цифры выдаётся за доказательство правильности.
3. **No audit trail.** Невозможно связать расход с конкретным SOP и результатом.
4. **Decision paralysis.** Менеджер видит данные, но не может принять решение.

## 5. Как это выглядит на ферме: правильное применение

**Ситуация:** скрининг BHB у свежих коров.

**Было (raw data):**

| Дата | Корова | BHB |
|---|---|---|
| 18.06 | 1847 | 1,4 |
| 18.06 | 1851 | 0,9 |

**Стало (Human-Facing Work Sheet):**

| # | Work ID | Дата/время | Объект | Зачем | MethodDescription | Исполнитель / роль | Результат | Следующий шаг |
|---|---|---|---|---|---|---|---|---|
| 1 | W-BHB-1806-001 | 18.06 05:45 | Группа Fresh (52 коровы, DIM 3–7) | Профилактика кетоза | SOP-BHB-01 v2, кровь хвостовой вены, Xpress Vet | Петров / VetFreshRole | 8 коров ≥1,2 ммоль/л | Пропиленгликоль 300 мл, повтор через 48 ч |
| 2 | W-BHB-2006-002 | 20.06 06:00 | Коровы 1847, 1855, 1861, 1866 | Контроль лечения | SOP-BHB-01 v2 | Петров / VetFreshRole | Все <1,0 ммоль/л | Закрыть случай |

**Результат:**
- Ветеринар видит полную картину: цель, метод, результат, триггер.
- Зоотехник может связать Work с кормовым рационом.
- Аудитор проверяет: MethodDescription утверждён, Work задокументирован, результат измерен.
- Консультант может сравнить с SoTA через Bridge.

## 6. Практическое применение: с чего начать

**Шаг 1.** Выберите один повторяющийся вид работы: скрининг, обработка копыт, вакцинация, кормление.

**Шаг 2.** Определите минимальный набор столбцов Work Sheet:
- Work ID
- Дата/время
- Объект (`EntityOfConcern`)
- Цель (`Problem`/`DecisionSubject`)
- MethodDescription (ссылка на SOP)
- Исполнитель + роль
- Результат (`Observation`)
- Следующий шаг / статус

**Шаг 3.** Для каждой строки проверьте: можно ли по ней принять решение без дополнительных вопросов? Если нет — добавьте недостающий контекст.

**Шаг 4.** Свяжите Work Sheet с источниками: MethodDescription, RoleAssignment, Observation.

**Шаг 5.** Публикуйте Work Sheet как **один лист**, который может удержать внимательный ум — не более 7–8 столбцов, не более одной страницы на тип работы.

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| В таблице только цифры без ссылки на SOP? | MethodDescription потерян |
| Непонятно, план это или факт? | DesignRunTag blur |
| Нет имени исполнителя и роли? | Accountability gap |
| Результат записан без шкалы/метода? | Observation не воспроизводима |
| Таблица не помещается на один экран? | Cognitive overload |
| Нет триггера следующего шага? | Decision gap |

## 9. Что запомнить

1. Work Sheet — это не raw data, а контекстная публикация Work.
2. Каждая строка должна содержать объект, цель, MethodDescription, результат, исполнителя и следующий шаг.
3. Результат — это Observation со шкалой, методом и порогом.
4. Work Sheet не заменяет MethodDescription или Evidence Graph, а делает их доступными.
5. Если по строке нельзя принять решение — в ней не хватает контекста.
