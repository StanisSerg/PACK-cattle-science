Роль: Автор метода, действующий в роли технолога по кормлению КРС
Интересующая сущность: CS.METHOD.007 — FPF-система решения типовых проблем технолога по кормлению
Ограниченный контекст: Cattle Science / кормление КРС / диагностика и решение рационных проблем на ферме

## Обоснование

Задача сейчас — не составить рацион и не вылечить корову, а создать **повторяемый способ действий** для технолога, который сталкивается с типовыми проблемами кормления. Это работа автора метода. При этом автор не абстрактный теоретик: он говорит из практики кормления, понимает разницу между «рассчитали в программе» и «скормили в кормушке», и проектирует метод так, чтобы им мог пользоваться именно технолог.

**Интересующая сущность** — не отдельная корова, не конкретный рацион и не болезнь. Это сам документ-метод: `CS.METHOD.007`. Его нужно сделать достаточно чётким, чтобы технолог мог взять любую проблему, пройти по шагам и получить проверяемое решение. Метод — это объект, над которым ведётся работа.

**Ограниченный контекст** — область «кормление КРС + рационные проблемы + фермерская практика». За его границами остаются ветеринарная терапия отдельных животных, генетика, экономика хозяйства, репродукция. Внутри — всё, что связано с рационом, съёмом корма, метаболическими нарушениями, переходным периодом и взаимодействием технолога с нутрициологом.

## Подходящая FPF-карточка README

**README card:** `2. Write rules, methods, and work-process documents`

Эта карточка выбрана, потому что текущая работа — именно создание метода (U.Method) и рабочего процесса для технолога. Содержание CS.METHOD.007 — это не научная статья и не отчёт, а нормативно-практический документ: «если видишь такую ситуацию — делай так».

## Как это влияет на CS.METHOD.007

1. **Роль зафиксирована.** Документ пишется не для «всех», а для технолога, который может не быть нутрициологом, но должен уметь диагностировать проблему и подготовить качественный запрос.
2. **Границы ясны.** Метод не лезет в лечение отдельных коров и не заменяет ветеринарный протокол. Он работает на уровне группы и рациона.
3. **Структура оправдана.** 8 шагов алгоритма — это не произвольный набор, а реализация паттернов ProblemCard, ADI-cycle, Evidence Graph и Checkpoint Return.
4. **Capture обязателен.** Без записи решённых случаев метод не будет накапливать знания — это следует из того, что EntityOfConcern — метод как развивающийся артефакт, а не разовая задача.
