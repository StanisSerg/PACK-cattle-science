# A.15.4 — Восстановление управляющего источника: dashboard — не основание для решения

> **Цель capture:** объяснить, как перед принятием решения найти управляющий источник, и почему dashboard, копии и сгенерированные объяснения не могут служить основанием для действий.

---

## 1. Зачем это читать

Современные фермы наполнены панелями управления, отчётами, скриншотами и сгенерированными объяснениями. Эти объекты выглядят как источники знания, но часто являются упрощёнными отображениями, копиями или интерпретациями. Если оператор действует по зелёной плитке dashboard, не восстановив управляющий источник, он может принять неверное решение. Паттерн A.15.4 учит различать кандидата в источники и управляющий источник, а также выбирать уровень проверки в зависимости от последствий.

> **FPF-тезис:** *«Dashboard, копия и объяснение — не управляющий источник; перед решением найди governing source.»*

**Фермерский пример:**

> Оператор увидел в панели управления зелёную плитку «Кетоз: низкий риск» и не стал брать кровь на анализ. Через два дня несколько коров заболели. Оказалось, что плитка строилась по упрощённой модели, а управляющим источником были лабораторные пробы; панель не была утверждена для принятия решений. Действие на основе dashboard привело к пропуску реального риска.

## 2. История одной ошибки

В хозяйстве внедрили систему мониторинга температуры молока в танкере. Водитель получил скриншот из мессенджера: «Температура в норме». По дороге к переработчику танкер отказал, и температура поднялась. При приёмке молоко было забраковано. Скриншот оказался копией старого показания; управляющим источником был сертификат температурного режима с датчиков, но водитель действовал по вторичному изображению.

## 3. Work-Relevant Source Restoration — полное описание

### 3.1 Определение

Паттерн A.15.4 утверждает: если на рабочем рубеже встречается кандидат в источники — dashboard, чужая выписка, сгенерированный текст, копия разрешения — нужно найти управляющий источник (`governing source`), то есть тот, на который можно полагаться для данной работы. В зависимости от последствий используются разные режимы: `orientation` (ориентировочно), `source-finding` (поиск источника), `routine` (рутинная проверка) или `high-impact` (строгая верификация).

### 3.2 Почему это важно

Dashboard, копии документов и сгенерированные объяснения часто принимают за управляющие источники, на основании которых можно действовать. Однако плитка dashboard может быть построена по упрощённой модели, копия документа — устареть, а сгенерированный текст — не иметь официального статуса. Если перед решением не восстановить управляющий источник, возникает риск необоснованных действий.

### 3.3 Encountered source candidate

**Определение.** Кандидат в источники (`encountered source candidate`) — это любой носитель, который выглядит как источник, но его статус неясен: плитка dashboard, скриншот, PDF-копия, LLM-объяснение, устное сообщение.

**Пояснение.** Кандидат даёт сигнал, но не может автоматически служить основанием для действия. Его нужно проверить и либо отвергнуть, либо связать с управляющим источником.

**Пример из животноводства.** Зелёная плитка «Кетоз: низкий риск» в панели управления — это кандидат в источники. Сообщение в мессенджере «ветеринар разрешил» без ссылки на приказ — тоже кандидат.

**Ключевой признак.** Возникает сомнение, можно ли на это полагаться для конкретного действия.

### 3.4 Governing source

**Определение.** Управляющий источник (`governing source`) — это источник, который уполномочен и достаточен для данного решения: приказ, утверждённый SOP, лабораторный протокол, официальная запись.

**Пояснение.** Управляющий источник имеет идентификатор, версию, контекст применения и уполномоченную роль. Он отличается от кандидата тем, что на него можно явно ссылаться при аудите.

**Пример из животноводства.** Для диагностики кетоза управляющим источником является лабораторный анализ крови по методу, утверждённому ветслужбой. Для отгрузки молока — сертификат качества, а не скриншот таблицы.

**Ключевой признак.** Можно назвать ID документа и версию, которая действует в данном контексте.

### 3.5 Режимы восстановления

**Определение.** Режимы восстановления источника — это уровни проверки в зависимости от последствий ошибки: `orientation`, `source-finding`, `routine`, `high-impact`.

**Пояснение.** Не все решения требуют одинаковой строгости. Ориентировочный режим допустим, когда последствия малы. High-impact режим требуется, когда решение влияет на здоровье животных, безопасность продукции или значительные экономические риски.

**Пример из животноводства.** Плитка dashboard — `orientation`; лабораторный анализ перед лечением — `high-impact`. Скриншот температуры в холодильнике — `routine`; решение о ветсертификате — `high-impact`.

**Ключевой признак.** Можно объяснить, почему именно такой уровень проверки выбран для данного решения.

## 4. Почему смешивать / игнорировать — значит рисковать

Рассмотрим типичное смешанное утверждение:

> *«Кетоз низкий, плитка зелёная — не берём кровь.»*

**Разложение по A.15.4:**

| Часть утверждения | Что это в FPF | Почему важно разделять |
|---|---|---|
| «Кетоз низкий» | претензия на знание | Требуется управляющий источник |
| «плитка зелёная» | `orientation`-кандидат | Dashboard не является основанием для решения |
| «не берём кровь» | решение | Должно основываться на governing source |

**Основные риски смешивания:**

1. **Dashboard = approval.** Цвет плитки воспринимается как разрешение действовать.
2. **Копия = оригинал.** PDF-копия приказа считается действующим утверждением.
3. **Сгенерированное объяснение = авторитет.** Текст LLM используется как SOP.
4. **Зелёный статус = доказательство.** Индикатор без источника данных.
5. **Пропуск source-finding.** Кандидат взят за источник без проверки.

## 5. Как это выглядит на ферме: правильное применение

**Ситуация:** оператор действовал по зелёной плитке dashboard; лабораторный источник не был найден.

**Было (смешанное / нечёткое):**
> «Кетоз: низкий риск — плитка зелёная, кровь не берём.»

**Стало (разложенное / ясное):**

**Encountered source candidate:**
> SC-001: плитка dashboard «Кетоз: низкий риск», статус — `orientation`

**Governing source:**
> GS-001: лабораторный анализ крови по протоколу KetoneLab-v2, утверждённому ветслужбой

**Режим:**
> RM-001: `high-impact` — решение влияет на здоровье коров

**Work:**
> W-001: `U.Work` взятие крови и анализ BHB, результат зафиксирован и привязан к доказательствам

**Результат:**
- Dashboard помечен как ориентировочный кандидат.
- Перед лечением выполнен high-impact поиск управляющего источника.
- Решение основано на лабораторном анализе, а не на цвете плитки.

## 6. Практическое применение: с чего начать

**Шаг 1.** Определите, какой источник вы видите: dashboard, копия, объяснение, запись.

**Шаг 2.** Оцените последствия действия на основе этого источника.

**Шаг 3.** Выберите режим: `orientation`, `source-finding`, `routine`, `high-impact`.

**Шаг 4.** Найдите управляющий источник: ID, версия, контекст, уполномоченная роль.

**Шаг 5.** Свяжите работу с источником через A.10.

**Шаг 6.** Если источник не найден — отложите действие и зафиксируйте риск.

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| Действие основано на dashboard/плитке/индикаторе? | Требуется проверить governing source. |
| Используется копия без проверки версии? | Нужно найти оригинал. |
| LLM/сгенерированный текст заменяет SOP? | Требуется восстановить официальное описание метода. |
| Решение влияет на здоровье/безопасность/экономику? | Необходимо использовать high-impact режим. |
| Источник не имеет ID и версии? | Это кандидат, а не governing source. |

## 9. Что запомнить

1. Dashboard, копии и объяснения — не управляющие источники по умолчанию.
2. Для каждого рабочего решения нужно найти governing source.
3. Уровень проверки зависит от последствий ошибки.
4. Управляющий источник имеет ID, версию и контекст применения.
5. Если источник не найден, действие следует отложить и зафиксировать риск.
