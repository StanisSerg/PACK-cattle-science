# K.2 — Migration Debt from A.2.6 (USM): долг миграции на Unified Scope Mechanism

> **Цель capture:** объяснить, какие правки требуются в FPF для выравнивания с новым Unified Scope Mechanism (USM) из A.2.6, и как управлять техническим долгом миграции.

---

## 1. Зачем это читать

Паттерн A.2.6 ввёл Unified Scope Mechanism (USM) — единый механизм определения области действия утверждений, наблюдений и правил. До USM области действия задавались по-разному в разных частях FPF: одни паттерны использовали `s.ScopeBoundary`, другие — локальные ограничения, третьи — неявные предположения. Это создало технический долг: многие паттерны нуждаются в правках, чтобы соответствовать новому механизму. Migration Debt from USM фиксирует этот долг и описывает необходимые изменения.

> **FPF-тезис:** *«Новый механизм области действия обесценивает старые способы задания границ.»*

**Фермерский пример:**

> В старом протоколе BHB-скрининга область действия порога была неявной: «BHB ≥ 1.2» действовало для всех коров. После внедрения USM стало ясно, что порог зависит от породы, стадии лактации и метода измерения. Протокол пришлось переписать, чтобы явно задать scope для каждого случая.

---

## 2. История одной ошибки

В хозяйстве использовали порог BHB ≥ 1.2 для всех групп коров. После внедрения USM выяснилось, что этот порог был валиден только для Holstein в первые 21 день лактации при фотометрическом методе. Для Jersey и для тест-полосок порог должен быть другим. Поскольку scope не был явно задан, часть коров получила неправильную классификацию риска.

---

## 3. Migration Debt from A.2.6 (USM) — полное описание

### 3.1 Unified Scope Mechanism

**Определение.** Unified Scope Mechanism (USM) — это единый способ задания области действия утверждений, наблюдений, правил и паттернов в FPF.

**Пояснение.** USM заменяет разрозненные подходы к ограничению применимости. Он требует, чтобы каждое утверждение явно указывало, к каким сущностям, в каких условиях и в каком временном интервале оно относится.

**Пример из животноводства.** Утверждение «BHB ≥ 1.2 указывает на риск кетоза» в рамках USM должно включать scope: «для коров породы Holstein, дни лактации 1–21, измерение фотометрическим методом».

**Ключевой признак.** Если утверждение звучит универсально, но применимо только в ограниченных условиях — это долг USM.

### 3.2 Технический долг миграции

**Определение.** Технический долг миграции — это совокупность изменений, которые необходимо внести в существующие документы, базы данных и протоколы для соответствия USM.

**Пояснение.** Долг включает не только замену терминов, но и переформулировку утверждений, добавление scope-ограничений, обновление relation и пересчёт показателей.

**Пример из животноводства.** Долг может включать: замену неявных порогов на scoped-thresholds, добавление `appliesTo` relation, уточнение population scope в отчётах, обновление SOP.

**Ключевой признак.** Долг миграции растёт, если старые протоколы продолжают использоваться без приведения к USM.

### 3.3 Required edits across FPF

**Определение.** Required edits — это конкретные правки, которые необходимо внести в различные части FPF для полного выравнивания с USM.

**Пояснение.** Правки затрагивают паттерны, где ранее scope задавался неявно или нестандартно. Каждая правка фиксируется с указанием паттерна, раздела и характера изменения.

**Пример из животноводства.**

| Паттерн | Раздел | Правка |
|---|---|---|
| C.2 | threshold | Добавить scope: порода, стадия лактации, метод |
| C.26 | boundary | Переформулировать process boundary через USM |
| A.6.C | scope boundary | Привести определение к единому механизму |
| B.3 | evidence | Указать scope validity для claim |

**Ключевой признак.** Если паттерн содержит неявные или нестандартные ограничения применимости, он требует правки.

---

## 4. Почему смешивать / игнорировать — значит рисковать

Рассмотрим типичное смешанное утверждение:

> *«Порог BHB ≥ 1.2 работает у нас.»*

**Разложение по K.2:**

| Часть утверждения | Что это в FPF | Почему важно разделять |
|---|---|---|
| «Порог BHB ≥ 1.2» | threshold | Требует scope |
| «работает у нас» | неявный scope | Не совместим с USM |
| Без указания условий | — | Риск переноса в неподходящий контекст |

---

## 5. Как это выглядит на ферме: правильное применение

**Было (смешанное / нечёткое):**

> В протоколе: «Если BHB ≥ 1.2 — риск кетоза». Это применялось ко всем коровам без различия породы и метода.

**Стало (разложенное / ясное):**

> В протоколе: «BHB ≥ 1.2 → risk class «субклинический кетоз». Scope: Holstein, дни лактации 1–21, фотометрический метод. Для других пород/методов — см. таблицу scoped-thresholds.»

---

## 6. Практическое применение: с чего начать

1. Проведите инвентаризацию всех утверждений в протоколах, где используются пороги, правила или классификации.
2. Для каждого утверждения определите неявный scope и сделайте его явным.
3. Сопоставьте scope с USM из A.2.6.
4. Обновите документы, добавив relation `appliesTo`, `validFor` или аналогичные.
5. Проверьте, не изменились ли результаты классификации при явном задании scope.

---

## 7. Проверь себя

| Вопрос | Если ответ «да» — проблема |
|---|---|
| В протоколах есть универсальные пороги без указания условий? | Неявный scope, долг USM |
| Scope задаётся по-разному в разных документах? | Отсутствие единого механизма |
| Новые утверждения не проверяются на совместимость с USM? | Долг накапливается |
| Старые отчёты нельзя сопоставить с новыми из-за разных scope? | Интеграция разрушена |

## 9. Что запомнить

1. USM из A.2.6 унифицирует задание области действия в FPF.
2. Многие паттерны имеют технический долг из-за неявных scope.
3. Каждое утверждение должно явно указывать, к чему оно применяется.
4. Пороги и классификации без scope — основной источник ошибок.
5. Migration debt нужно учитывать при приоритизации правок.
