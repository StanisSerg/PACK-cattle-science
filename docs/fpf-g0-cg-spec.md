# G.0 — CG-Spec: законные сравнения и пороги доказательности

## 1. Зачем это читать
Если вы когда-нибудь сравнивали два исследования по кормлению и думали: «Ну, примерно похоже, можно свести в табличку» — вы стояли на краю пропасти. CG-Spec — это то, что отличает **научное сравнение** от **интеллектуального мошенничества**.

Этот паттерн отвечает на вопрос: при каких условиях сравнение двух (или двадцати) исследований **законно**? Что должно быть зафиксировано заранее, чтобы никто не мог «подогнать» правила под желаемый результат? Какие доказательства минимально необходимы, чтобы утверждать: «Корм A лучше корма B»?

G.0 — это **design-time legality gate**. Он работает до того, как вы начнёте считать. Он говорит: можно ли здесь считать вообще.

## 2. История одной ошибки
**Консультант Анна, 2022 год.** Ферма просит сравнить эффективность двух премиксов для сухостойных: Premix-X (Германия) и Premix-Y (локальный). Анна находит 4 исследования по X и 3 по Y.

Исследования по X измеряли NEFA в крови в ммоль/л на 7-й день после отёла. Исследования по Y измеряли BHB в моче в мг/дл на 14-й день. Разные метаболиты, разные биоматериалы, разные сроки, разные единицы.

Анна делает «умную» табличку: переводит всё в «условные баллы здоровья» (своя шкала), усредняет по дням и выдаёт ферме: «X эффективнее на 12%».

**Что сломалось:**
1. Сравнение было **незаконным** — ComparatorSet не был объявлен; Анна изобрела его постфактум.
2. **Масштаб/единицы** не соответствовали SCP (Scale Compliance Profile) — скрытая скаляризация.
3. **Минимальные доказательства** не были декларированы — ферма не знала, что выборка в одном из исследований была 8 коров.
4. **Crossing** был неявным — европейские данные применены к российскому стойлу без декларации потерь.

G.0 говорит: если бы был CG-Spec, сравнение либо было бы объявлено нелегальным заранее, либо проведено с явными допущениями и штрафами.

## 3. CG-Spec: законные сравнения и пороги доказательности — полное описание
### 3.1 ComparatorSet — кого можно сравнивать
**Что это:** Явно заявленный, конечный набор допустимых форм сравнения для данного CG-Frame.

**Аналогия:** На выставке быков нельзя сравнивать быка и корову в одном классе. Есть классы: «молодые быки до 1 года», «племенные быки-производители». ComparatorSet — это список классов. Если ваши данные из разных классов — вы не сравниваете, вы делаете ошибку.

**На ферме:** Для CG-Frame «переходный период» ComparatorSet может включать:
- Сравнение внутри одной метаболитной шкалы (NEFA с NEFA, BHB с BHB).
- Сравнение по одной породе или с явным Bridge для породных различий.
- Агрегация по медиане для порядковых шкал (если доказана legality).

**Что запрещено в ComparatorSet** (если не вынесено в explicit Bridge):
- Сравнение NEFA и BHB напрямую.
- Смешение пастбищного и стойлового содержания без декларации.
- Усреднение данных с разных сроков лактации как «одно число».

### 3.2 SCP — Scale Compliance Profile
**Что это:** Профиль, который говорит: какие операции законны для каждой характеристики (характеристики CHR), с учётом шкалы, единиц измерения и полярности.

**Аналогия:** В аптеке есть правила:
- Температуру можно сравнивать (37° < 39°).
- Боли нельзя складывать («боль 3 + боль 5 ≠ боль 8»), можно только сравнивать порядок.
- Названия препаратов нельзя усреднять.

SCP — это именно такие правила для каждого измерения в вашем Pack.

**На ферме:** Для характеристики «Уровень BHB»:
- Шкала: непрерывная, ммоль/л.
- Законные операции: разность, медиана, сравнение.
- Незаконные: умножение BHB на BHB, «рейтинговая шкала» (1-5 звёзд), присвоение порядковых рангов без explicit policy.

### 3.3 MinimalEvidence — минимальные доказательства
**Что это:** Для каждой характеристики явно декларируется: каких носителей данных достаточно, какова «свежесть» данных, что делать при нехватке.

**Аналогия:** Ветеринарный протокол говорит: «Диагноз мастита требует бактериологического посева ИЛИ клинической картины + теста Калифорнии. Данные старше 14 дней считаются устаревшими».

**На ферме:** Для характеристики «эффективность профилактики кетоза»:
- Требуемые lanes: RCT или когортное исследование с контролем.
- Carriers: публикация в рецензируемом журнале ИЛИ внутренний аудит с N>50.
- Freshness window: данные старше 10 лет — degrade.
- Failure behavior: если данных недостаточно — abstain (не pass, не «ну как-нибудь»).

### 3.4 Γ-fold и penalty routing
**Что это:** Правила сворачивания штрафов и доверия. Если данные имеют недостатки, как это влияет на итоговое доверие к рекомендации?

**Аналогия:** На конкурсе удойности каждая корова получает оценку. Но если данные по корове собраны без ветеринарного паспорта — штраф. Γ-fold говорит: как именно штрафы складываются и куда идут. Важно: штрафы идут только в «оценку достоверности» (R_eff), а не в «сколько молока» (F).

**На ферме:** Исследование с выборкой N=15 получает штраф за статистическую мощь. Этот штраф уменьшает доверие к выводу «премикс работает» — но не изменяет само значение снижения BHB, полученное в этом исследовании.

### 3.5 AcceptanceStubs — шаблоны приёмки
**Что это:** Design-time шаблоны порогов приёмки. Сами пороги управляются CAL (G.4), но форма порога (какие величины сравниваем, в каких единицах) фиксируется в CG-Spec.

**Аналогия:** В протоколе осеменения написано: «Приёмка быка-производителя: плодовитость > 70%, процент выживаемости потомства > 90%». Это не само решение (оно принимается позже), это **форма** решения. CAL скажет: «порог 70% или 75%?», а CG-Spec говорит: «плодовитость измеряем в процентах, сравниваем с контролём».

## 4. Почему смешивать / игнорировать — значит рисковать
| Anti-pattern | Проявление на ферме | Почему опасно |
|---|---|---|
| **Post-hoc ComparatorSet** | Сначала собрали данные, потом придумали, как их сравнивать | Confirmation bias; можно «подогнать» правила под желаемый результат |
| **Unit Mismatch** | Сравнили NEFA (ммоль/л) с BHB (мг/дл) без explicit Bridge | Сравнение бессмысленно; рекомендация ложна |
| **Implicit Totalisation** | Смешали данные с 7-го и 14-го дня, усреднив как «первая неделя» | Разные физиологические окна; эффект маскируется или инвертируется |
| **Missing Evidence Minimum** | В Pack попало исследование N=3 без декларации | Статистическая мощь недостаточна; решение принимается на шуме |
| **Shadow Normalization** | Перевели всё в «условные баллы» без explicit policy и proof refs | Потеря информации; скрытая скаляризация; невозможен аудит |
| **Freshness Blindness** | Используют данные 1990-х как равнозначные современным | Управление, генетика, корма изменились; результаты неприменимы |
| **Failure Behavior = Auto-Pass** | При нехватке данных система берёт «что есть» и делает pass | unknown coerces to pass — прямое нарушение G.Core |

## 5. Как это выглядит на ферме: полный сценарий
**Задача:** Сравнить два протокола профилактики ацидоза при откорме быков на мясо.

**CG-Frame:** `Acidosis_prevention_feedlot_bulls_2026`

**M1 — Charter:**
- Context: «Фермы РФ, стойловое откормочное содержание быков мясных пород, 200-400 кг»
- describedEntity: `Acidosis_prevention`
- ReferenceMap: термины привязаны к UTS (SARA, pH рубца, NDF, концентрат/сено ratio)

**M2 — ComparatorSet:**
- Допустимые сравнения:
  1. pH рубца (непрерывная, один метод измерения) между протоколами.
  2. Частота клинических случаев SARA (порядковая/частотная) между протоколами.
  3. Недопустимо: сравнение pH рубца и «оценки ветеринара по шкале 1-5» без explicit Bridge.

**M3 — SCP:**
- pH рубца: операции — разность, медиана, сравнение. Запрещено: умножение, рейтинговая трансформация.
- Частота SARA: операции — сравнение частот, относительный риск. Запрещено: среднее арифметическое (бинарные события).

**M4 — MinimalEvidence:**
- Для pH рубца: требуется ≥2 измерения на животное, метод фистуляции или оральный зонд, данные <5 лет.
- Для частоты SARA: требуется когортное исследование N≥30 или RCT, данные <10 лет.
- Failure behavior: если данных меньше — abstain (протокол не включается в сравнение).

**M5 — Γ-fold и penalties:**
- Default: `DefaultId.GammaFoldForR_eff` (управляющая дефиниция: CC-G5.4).
- Исследование без контрольной группы: degrade, penalty → R_eff.
- Исследование из пастбищной системы: explicit Bridge с declared losses; если losses не декларированы — abstain.

**M6 — Публикация:**
- `CG-Spec@UTS` с edition-pinned segments.
- Name Cards для каждой характеристики.
- RSCR tests: проверка на illegal_op_refusals, unit mismatches, partial-order scalarisation.

**Результат:** Любой аудитор может за 10 минут проверить: какие сравнения законны, какие доказательства требовались, куда пошли штрафы, что делать при обновлении спеков.

## 6. Практическое применение: с чего начать
**Шаг 1. Выбери один CG-Frame.**
Не пытайся охватить всё. Один вопрос: «Сравнение протоколов профилактики кетоза» или «Оценка заменителей молока для телят».

**Шаг 2. Опиши ComparatorSet.**
Перечисли: какие сравнения законны в этом Frame. Что можно сравнивать с чем. Что — нельзя. Запиши явно.

**Шаг 3. Привяжи CHR-характеристики.**
Каждая сравниваемая величина должна быть CHR Characteristic с ID. Не придумывай «свои» метрики — используй UTS-типизированные.

**Шаг 4. Задай SCP для каждой характеристики.**
Какие операции законны? Какие единицы? Какая полярность? Что запрещено?

**Шаг 5. Объяви MinimalEvidence.**
Для каждой характеристики: lanes, carriers, freshness, failure behavior. Если не можешь обосновать — ставь abstain.

**Шаг 6. Зафиксируй policy IDs.**
Γ-fold, Φ, penalty policies — всё должно иметь policy-id. Нет «по умолчанию мы так делаем». Есть «DefaultId.X, governing definition CC-Y.Z».

**Шаг 7. Опубликуй и пришпиль.**
CG-Spec публикуется в UTS с edition pins. Каждый сегмент адресуется. Изменение edition = RSCR trigger.

## 7. Проверь себя
| Вопрос | Если ответ «не знаю» — проблема |
|---|---|
| Можете ли вы показать явный ComparatorSet для вашего главного CG-Frame? | Post-hoc сравнение — bias |
| Для каждой метрики известна ли шкала и законные операции? | SCP не задан — скрытая скаляризация |
| Объявлены ли минимальные требования к доказательствам? | Missing evidence minimum — решения на шуме |
| Есть ли explicit policy IDs для штрафов и доверия? | Penalty routing непрозрачен |
| Данные какого возраста считаются допустимыми? | Freshness blindness — устаревшие данные |
| Что система делает при нехватке данных? | Failure behavior не определён — риск auto-pass |
| Опубликован ли CG-Spec в UTS с edition pins? | Невоспроизводимость; shadow spec |
