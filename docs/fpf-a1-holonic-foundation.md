# A.1 — Holonic Foundation: сущность, холон, система, эпистема

## 1. Зачем это читать
Если вы называете всё подряд «системой» — документы, теории, команды, коров, датчики — вы рискуете приписать одни и те же свойства разным вещам. Документ не может действовать. Корова — не теория. А список людей — ещё не команда.

A.1 даёт корневую онтологию FPF: какие виды «вещей» бывают и какие свойства к каждому можно применять.

**FPF-тезис:** *«Не всякая вещь — система. Не всякое знание — актор. Различай сущность, холон, систему и эпистему.»*

## 2. История одной ошибки
Фермер Игорь считал, что у него «система управления кормлением», потому что был составлен протокол и куплена программа. На деле протокол хранился в голове одного зоотехника, а программа считала только фактическое потребление, не связывая его с нормой. Когда зоотехник ушёл, «система» исчезла: никто не понимал, где сущность, где холон, где эпистема. Принимать решения стало невозможно, и удой просел на 10 %.

## 3. Holonic Foundation: сущность, холон, система, эпистема — полное описание

### 3.1 Четыре уровня
```text
U.Entity
  U.Holon
    U.System
    U.Episteme
```

#### 3.1.1 U.Entity — «нечто, на что можно указать»
Минимальный уровень. Число, имя, партия корма, строка в документе. Без предположений о частях и целом.

#### 3.1.2 U.Holon — «целое с частями и часть в целом»
Holon — это Entity, которое одновременно:
- воспринимается как целое,
- может быть частью большего целого.

У холона есть:
- `holonIdentity` — идентичность,
- `boundedContextRef` — контекст,
- `boundaryRef` — граница,
- `partRelationSet` — отношения часть-целое,
- `interactionSet` — то, что пересекает границу.

**На ферме:** стадо — холон. Корова — холон. Доильный зал — холон. Рацион — холон.

#### 3.1.3 U.System — «холон, который может действовать»
Только системы могут:
- играть роли,
- исполнять методы,
- выполнять работу (Work),
- производить эффекты.

**На ферме:** корова, ветеринар, доильный аппарат, команда зоотехников (если оформлена как collective system) — системы.

#### 3.1.4 U.Episteme — «холон, который несёт claims, но не действует»
Episteme — это знание: документ, модель, протокол, теория, claim. Оно не может решать, исполнять или изменяться само по себе. Изменяется через носители (SCR) системой.

**На ферме:** протокол лечения кетоза — эпистема. Он не лечит. Лечит ветеринар, который читает протокол.

### 3.2 Тест на членство в холоне
Когда спорно, является ли что-то частью холона, проверь:

1. **Зависимость:** если убрать кандидата, нарушится ли ключевой инвариант целого?
2. **Внутреннее взаимодействие:** участвует ли кандидат во внутренних interactions холона?
3. **Эмерджентность:** вносит ли кандидат вклад в свойство целого, которого нет у частей по отдельности?

Если ни один тест не проходит — кандидат вне холона для данного claim.

### 3.3 Граница и взаимодействие
`U.Boundary` отделяет холон от окружения. `U.Interaction` — то, что пересекает границу: масса, энергия, информация, сигнал управления.

**Важно:** не называйте всякую границу «интерфейсом». Интерфейс — это когда есть явный механизм, сигнатура или архитектурный паттерн.

### 3.4 Коллекция vs коллектив
| Коллекция (set) | Коллектив (system) |
|---|---|
| `{Алиса, Боб, 3.14}` | Команда зоотехников |
| Нет поведения | Есть граница, координационный Method, Work |
| Не может действовать | Может играть AgentialRole |

Если группа должна действовать — моделируйте её как `U.System` с границей, ролями и методом. Если нет — оставляйте множеством.

### 3.5 Правильные и неправильные фразы
| Вместо | Пиши | Почему |
|---|---|---|
| «Протокол требует» | «В контексте X протокол P играет ConstraintSourceRole; система S исполняет Work по его применению» | Episteme не действует |
| «Стадо решило» | «Система-стадо как холон не принимает решений; решение принял оператор/ветеринар в роли R» | Только системы действуют |
| «Команда = список сотрудников» | «Команда — collective system с boundary + coordination Method» | Список не действует |
| «Эта диаграмма — архитектура фермы» | «Диаграмма — publication face; архитектура — selected structure холона» | Публикация ≠ холон |

## 4. Почему смешивать / игнорировать — значит рисковать
Если пренебречь A.1, типичные риски на ферме:

| Что происходит | Почему это опасно |
|---|---|
| Решения принимаются на основе смешанных или устаревших данных | Невозможно отследить, что именно повлияло на результат |
| Роли и методы не разделены | Один и тот же человек или документ отвечает за несовместимые обязательства |
| Перенос знаний между контекстами без явного bridge | Рекомендации из одной традиции применяют к другой без учёта потерь |
| Отсутствие контрольного списка | Ошибки обнаруживаются слишком поздно, когда они уже стоят денег |

**Каскад:** нечёткое утверждение → неправильная интерпретация → некорректное решение → экономический убыток и снижение доверия к системе.

## 5. Как это выглядит на ферме: правильное применение
**Задача:** классифицировать объекты фермы по уровням: Entity, Holon, System, Episteme.

**Было:**
> «У нас есть система кормления.» — под системой понимался и рацион, и отдел, и программа.

**Стало:**
> `Entity`: корова, силос, премикс.
> `Holon`: бригада доения, лактирующая группа (часть стада и одновременно целое).
> `System`: цикл «рацион → раздача → фактическое поедание → коррекция» со связями и эмерджентностью.
> `Episteme`: NASEM_2021, внутренние протоколы, измерительные методики.

Каждый уровень имеет свои invariant'ы; прыгать между уровнями можно только через явный bridge.

## 6. Практическое применение: с чего начать
**Шаг 1.** Перечислите 10 объектов вашей фермы.

**Шаг 2.** Для каждого объекта определите: он скорее Entity, Holon, System или Episteme?

**Шаг 3.** Проверьте, не называете ли вы System'ой просто документ или отдельного человека.

**Шаг 4.** Для каждого System укажите входящие Holon'ы, цель и критерий успеха.

**Шаг 5.** Зафиксируйте, какие Episteme легитимны для каждого System.

## 7. Проверь себя
| Вопрос | Если ответ «да» — проблема |
|---|---|
| `CC-A1-1`: объект типизирован как Entity / Holon / System / Episteme до того, как к нему применены part-whole, role, work или architecture claims. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-2`: part-whole и агрегация — только для холонов. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-3`: каждый холон используется с named bounded context и boundary. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-4`: спорное членство проверяется тестами dependency / internal interaction / emergence. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-5`: acting roles, method enactment, work — только к System. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-6`: множество не считается действующим коллективом без boundary и role assignments. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-7`: Episteme не является агентом. | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-8`: slot position не меняет kind (system в роли остаётся system; episteme как EntityOfConcern остаётся episteme). | Контрольный критерий не выполнен — возможна структурная ошибка. |
| `CC-A1-9`: publication forms отделены от описываемых холонов. | Контрольный критерий не выполнен — возможна структурная ошибка. |
