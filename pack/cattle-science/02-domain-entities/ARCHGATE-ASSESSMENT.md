## 1. Эволюционируемость (8/10)

### Что сломается при изменении:

**Стабильное:**
- Иерархия P0/P1/P2 — чёткая классификация по приоритету
- YAML frontmatter — стандартизированная структура метаданных
- ID-нумерация (CS.ENTITY.XXX) — неизменна
- Связи related_entities — типизированы

**Риски:**
- Несогласованность YAML между P0/P1/P2 (разные ключи: `related_sota` vs `source-sota`)
- Отсутствие единой схемы валидации
- Ручное обновление инвентаря при изменениях

### SPF-анализ:

| Принцип | Реализация | Оценка |
|---------|-----------|--------|
| Separation | P0/P1/P2 — логическое разделение | ✅ |
| Single Responsibility | Каждая сущность — одно понятие | ✅ |
| Stable Dependencies | YAML/markdown — стабильны | ✅ |
| Open/Closed | Расширение через новые сущности | ✅ |

**Рекомендации:**
1. Унифицировать YAML схему (единые ключи для всех приоритетов)
2. Добавить JSON Schema для валидации
3. Автоматизировать обновление инвентаря

## 2. Масштабируемость (7/10)

### Текущая структура:

```
02-domain-entities/
├── P0/          (58 files)
├── P1/          (92 files)
├── P2/          (37 files)
└── 03-interpretations/  (11 files)
```

### Что будет при 10x (1000+ сущностей):

**Проблемы:**
- Папки P1/P2 разрастутся (>100 файлов)
- Поиск замедлится
- Инвентарь станет огромным
- Дублирование сущностей возможно

**Решения:**
- Подпапки по area (feeding/, health/, reproduction/)
- Автоматическая генерация индекса
- Полнотекстовый поиск (ripgrep)
- Дедупликация при создании

### Лимиты:

| Уровень | Текущее | Комфортный лимит | Максимум |
|---------|---------|------------------|----------|
| P0 | 58 | 100 | 150 |
| P1 | 92 | 200 | 300 |
| P2 | 37 | 500 | 1000 |

**Оценка:** Текущая структура выдержит 300-400 сущностей. Для 1000+ нужна реорганизация.
