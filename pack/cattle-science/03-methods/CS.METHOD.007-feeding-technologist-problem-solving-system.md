## [CS.METHOD.007] Система решения типовых проблем технолога по кормлению через FPF

### Что это такое

Метод структурированного решения рабочих проблем, с которыми сталкивается технолог по кормлению молочного стада. Он переводит нечёткую ситуацию («что-то не так с рационом») в последовательность шагов, управляемых паттернами **First Principles Framework (FPF)**.

> **FPF** = First Principles Framework = фреймворк первых принципов (паттерн-язык для структурирования знаний и решений).
> **ADI** = Abduction-Deduction-Induction-Audit-Decide = цикл абдукции-дедукции-индукции-аудита-решения.
> **DCAD** = dietary cation-anion difference = диетическая разница катионов и анионов.
> **TMR** = total mixed ration = полнорационная кормосмесь.

Метод не заменяет нутрициологическую экспертизу, но делает её воспроизводимой: любой технолог может взять карточку проблемы, пройти по шагам и либо решить вопрос самостоятельно, либо подготовить качественный запрос консультанту.

### FPF-фрейминг задачи

Роль: Автор метода, действующий в роли технолога по кормлению КРС  
Интересующая сущность: CS.METHOD.007 — FPF-система решения типовых проблем технолога по кормлению  
Ограниченный контекст: Cattle Science / кормление КРС / диагностика и решение рационных проблем на ферме

**README card:** `2. Write rules, methods, and work-process documents`

Эта карточка выбрана, потому что текущая работа — создание метода (U.Method) и рабочего процесса для технолога. Содержание CS.METHOD.007 — не научная статья и не отчёт, а нормативно-практический документ: «если видишь такую ситуацию — делай так». Автор пишет из практики кормления, понимает разницу между «рассчитали в программе» и «скормили в кормушке», и проектирует метод так, чтобы им мог пользоваться именно технолог.

**Интересующая сущность** — не отдельная корова, не конкретный рацион и не болезнь. Это сам документ-метод. Его нужно сделать достаточно чётким, чтобы технолог мог взять любую проблему, пройти по шагам и получить проверяемое решение.

**Ограниченный контекст** — область «кормление КРС + рационные проблемы + фермерская практика». За границами остаются ветеринарная терапия отдельных животных, генетика, экономика хозяйства, репродукция. Внутри — всё, что связано с рационом, съёмом корма, метаболическими нарушениями, переходным периодом и взаимодействием технолога с нутрициологом.

### Зачем нужен

- Снизить зависимость решений от интуиции и опыта отдельного человека.
- Ускорить диагностику: быстрее отличать симптом от причины.
- Связывать практические ситуации с принципами из CS.DPF.001 и связанных сущностей.
- Создавать накопительный эффект: каждая решённая проблема обогащает систему.
- Дать технологу язык для разговора с нутрициологом, ветеринаром и руководством.

### Ключевые различия

- **Не пошаговая инструкция, а диагностическая система.** Метод задаёт вопросы, а не навязывает готовые дозировки.
- **Различает модель и реальность.** Расчётный рацион — это claim, который нужно верифицировать.
- **Ориентирован на repeatable capture.** Каждое решение должно быть записано, иначе система не учится.
- **Работает на уровне группы, а не отдельной коровы.** Для индивидуальных случаев используются ветеринарные протоколы.

## Алгоритм: FPF-диагностика проблемы кормления

### Стиль Б — Нумерованный алгоритм

Метод построен как восемь шагов, которые образуют мини-ADI цикл.

---

### 1. Зафиксировать проблему (Problem Card)

**Цель:** превратить жалобу в чёткое описание ситуации.

#### Обязательный скрининг:

- [ ] Что наблюдаем? (симптом)
- [ ] Что ожидали? (норма / цель)
- [ ] Когда заметили?
- [ ] Что изменилось за 7–14 дней до этого?
- [ ] К какой группе относится проблема?
- [ ] Кто ещё заметил?

#### Шаблон Problem Card:

```markdown
**ID:** CS.PROB.XXX
**Симптом:** [краткое описание]
**Ожидаемый результат:** [что должно быть]
**Фактический результат:** [что есть]
**Группа:** [название группы]
**Срок:** [когда началось]
**Изменения:** [что менялось]
**Срочность:** [низкая / средняя / высокая]
```

> **Примечание:** формат Problem Card отсылает к паттерну C.22.2 — стабилизировать дешёвую запись проблемы до того, как вкладываться в диагностику.

---

### 2. Собрать evidence

**Цель:** отделить факты от предположений.

#### 2.1 Проверить model claim

| Вопрос | Источник |
|--------|----------|
| Какая модель / программа используется? | AMTS, NASEM-2021, CNCPS, Excel |
| Какие входные данные взяты из библиотеки? | Feed library |
| Какие — из лаборатории? | Анализы кормов |
| Когда последний раз обновлялись данные? | Журнал анализов |
| Какие допущения заложены в модели? | Настройки программы |

#### 2.2 Проверить фактическое исполнение

- [ ] Рацион заложен по расчёту?
- [ ] Коровы съедают весь рацион?
- [ ] Есть сортировка?
- [ ] Вода доступна и чиста?
- [ ] Условия содержания изменились?

#### 2.3 Проверить измерения

- [ ] Как измеряли симптом?
- [ ] Прибор откалиброван?
- [ ] Время и способ отбора проб правильные?
- [ ] Сколько проб взяли? (не одна)

> **Примечание:** этот шаг использует паттерны P5 Assay Trust, C.16 Measurement Evidence, B.3 F-G-R Trust.

---

### 3. Найти FPF-паттерны

**Цель:** соотнести проблему с принципами и паттернами из домена.

#### Типовые мосты между симптомом и паттерном:

| Симптом | Вероятные паттерны |
|---------|-------------------|
| pH мочи не снижается при анионной диете | P4 DCAD, P5 Assay Trust, P1 Feed Fractionation |
| Низкий удой при «правильном» рационе | P2 Ration Balancing, P3 DMI Forecast, P6 Model Calibration |
| Провалы жира в молоке | P1 Feed Fractionation, P3 Carbohydrate Fractionation |
| Плохое оплодотворение | P7 Transition Physiology, P2 Energy Balance |
| Коровы не едят | P3 DMI, P8 New Feed Admission, P5 Assay Trust |
| Проблемы после ввода нового корма | P8 New Feed Admission, P1 Feed Fractionation |

> **Примечание:** для каждого паттерна задать вопрос: «Какое утверждение этого паттерна могло нарушиться?»

---

### 4. Построить гипотезы

**Цель:** сгенерировать 2–4 объяснения и ранжировать их.

#### Формат гипотезы:

```markdown
**Гипотеза 1:** [утверждение]
**Паттерн:** [какой принцип нарушен]
**Evidence за:** [факты, которые подтверждают]
**Evidence против:** [факты, которые опровергают]
**Эксперимент:** [как проверить быстро и дёшево]
**Вероятность:** [высокая / средняя / низкая]
```

#### Правила ранжирования:

1. Сначала проверяются гипотезы, которые можно опровергнуть быстро.
2. Сначала ищутся ошибки в данных (P5), а не в модели.
3. Сначала смотрят на то, что менялось недавно.

---

### 5. Выбрать минимальный эксперимент

**Цель:** получить ответ с минимальными затратами и рисками.

#### Критерии хорошего эксперимента:

- [ ] Изменяем только одну переменную.
- [ ] Можно измерить результат за 3–7 дней.
- [ ] Риск для животных минимален.
- [ ] Результат однозначно подтвердит или опровергнет гипотезу.

#### Примеры экспериментов:

| Гипотеза | Эксперимент |
|----------|-------------|
| Высокий K в сенаже | Заменить часть сенажа на солому, замерить pH мочи |
| Низкая поедаемость из-за вкуса | Снизить дозу добавки на 20%, оценить остатки |
| Проблема в измерении | Взять 10 проб в одно время, откалибровать pH-метр |
| Буферы нейтрализуют анионы | Убрать MgO/CaCO₃ на 5 дней, замерить отклик |

---

### 6. Выполнить и измерить

**Цель:** получить фактический отклик.

#### Что фиксировать:

- [ ] Что именно изменили.
- [ ] Дата и время изменения.
- [ ] Какие показатели мерили.
- [ ] Метод измерения.
- [ ] Кто выполнял.
- [ ] Погодные и управленческие условия.

> **Примечание:** если эксперимент не дал ответа — вернуться к шагу 4 и пересмотреть гипотезы. Это паттерн C.24 Checkpoint Return.

---

### 7. Интерпретировать и решить

**Цель:** на основании evidence выбрать действие.

#### Возможные исходы:

| Исход | Действие |
|-------|----------|
| Гипотеза подтвердилась | Внедрить решение, зафиксировать capture |
| Гипотеза опровергнута | Вернуться к шагу 4 |
| Эффект есть, но недостаточный | Уточнить дозу / условия, повторить эксперимент |
| Эффекта нет, гипотезы исчерпаны | Эскалировать нутрициологу / ветеринару |

#### Критерии принятия решения:

- Решение не противоречит CS.DPF.001.
- Решение можно объяснить через FPF-паттерн.
- Решение безопасно для группы.
- Есть план повторной оценки через 7–14 дней.

---

### 8. Зафиксировать capture

**Цель:** превратить опыт в повторяемый принцип.

#### Формат capture:

```markdown
**Situation:** [Problem Card]
**Action:** [что сделали]
**Result:** [что получили]
**Principle:** [обобщённый урок]
**Kind:** FieldValidatedHeuristic@Context
**Applies to:** [паттерны]
```

> **Примечание:** capture — не формальность. Без него метод не работает как накопительная система.

## Пример применения: pH мочи не закисляется

### Problem Card

```markdown
**ID:** CS.PROB.001
**Симптом:** pH мочи сухостойных коров 7.3, не снижается ниже 7.0
**Ожидаемый результат:** pH 6.0–6.5
**Фактический результат:** pH 7.3 (среднее по группе)
**Группа:** Сухостой II периода
**Срок:** 5 дней после ввода анионных солей
**Изменения:** Введены анионные соли, рацион — солома, сенаж, силос, комбикорм
**Срочность:** Средняя
```

### Evidence

- AMTS показывает DCAD −100 мЭкв/кг СВ.
- pH-метр откалиброван.
- Замер через 3 часа после кормления.
- Коровы съедают рацион.

### Гипотезы

1. **Высокий K в сенаже.** Паттерн P1 + P5. Эксперимент: уменьшить сенаж, добавить солому.
2. **Щелочные буферы в МВД.** Паттерн P2. Эксперимент: проверить состав минеральной смеси.
3. **Преимущественно серный источник анионов.** Паттерн P4. Эксперимент: проверить Cl:S в добавке.

### Эксперимент

Уменьшили сенаж, увеличили солому.

### Результат

pH мочи снизился до 6.0.

### Capture

```markdown
**Situation:** Сухостой II периода, анионные соли, DCAD −100 в AMTS, pH мочи 7.3.
**Action:** Уменьшили сенаж, увеличили солому.
**Result:** pH мочи снизился до 6.0.
**Principle:** Реальный калий сенажа/силоса может нейтрализовать расчётный DCAD.
**Kind:** FieldValidatedHeuristic@Context
**Applies to:** P4.DCAD, P1.FeedFractionation, P5.AssayTrust
```
