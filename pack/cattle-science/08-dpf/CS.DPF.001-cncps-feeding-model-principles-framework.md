# Readme — First Practical Entries

CNCPS-DPF нужен, когда рациональщик переходит от «составить рацион по таблице НР» к расчёту на основе фракционирования углеводов и белков в рубце. Пакет помогает избежать типичных ошибок: ввод оценочных значений как лабораторных, использование одной Kd для всех кормов, игнорирование баланса аминокислот, доверие к выходу программы без сверки с фактом.

## Вход 1 — Составить рацион для существующей группы

1. `CNCPS.5` — описать животных и прогноз DMI.
2. `CNCPS.1` — разметить имеющиеся корма (анализ или помеченные оценочные значения).
3. `CNCPS.2` — выбрать Kd под условия рубца.
4. `CNCPS.3` — рассчитать MCP и баланс азота.
5. `CNCPS.4` — проверить MP, Lys, Met.
6. `CNCPS.6` — после 2–4 недель сверить прогнозы с фактом.

Типичный результат: один `CNCPS.RationCase@Context` с явными допущениями и метками неопределённости.

## Вход 2 — Понять, почему модель расходится с фактом

1. `CNCPS.6` — зафиксировать отклонения (DMI, жир, белок, MUN).
2. `CNCPS.1` — проверить качество входных данных.
3. `CNCPS.2` — проверить Kd и условия ферментации.
4. `CNCPS.5` — проверить описание животных.
5. `CNCPS.3`/`CNCPS.4` — проверить расчёт MCP и аминокислот.

Типичный результат: `CNCPS.CalibrationNote@Context` с гипотезой отклонения и планом коррекции.

## Вход 3 — Добавить нестандартный корм

1. `CNCPS.8` — собрать локальные анализы и обосновать фракционирование.
2. `CNCPS.1` — разметить значения как measured/literature/estimated.
3. `CNCPS.6` — откалибровать по ферме, где корм уже используется.

Типичный результат: `CNCPS.FeedExtensionRecord@Context`.

## Вход 4 — Обучить технолога

1. `Preface` — объяснить, зачем фракционирование лучше «среднего состава».
2. `CNCPS.1`–`CNCPS.5` — пройти по шагам на одном примере.
3. `CNCPS.6` — показать сверку с фактом.
4. `CNCPS.7` — объяснить, как фиксировать версию инструмента.

Типичный результат: `CNCPS.LearningRoute@Context` + набор упражнений в отдельных файлах.

# Preface — Cross-Cutting Ideas And Principles

CNCPS — не «более сложная таблица НР». Это модель, которая заменяет категории «сырой протеин» и «переваримая энергия» на фракции, деградирующие в рубце с разной скоростью. Цель DPF — сделать так, чтобы практик пользовался моделью осознанно, а не как чёрным ящиком.

## Центральное различие: фракция ≠ средний состав

В традиционных системах корм корневой оценивается по среднему содержанию CP, NDF, ADF. CNCPS разбивает те же вещества на фракции по скорости деградации в рубце. Это меняет всё:

- Один и тот же сырой протеин (CP) может дать разный MCP в зависимости от распределения PA–PB1–PB2–PB3–PC.
- Один и тот же NFC может по-разному влиять на энергию рубца, если CA/CB1/CB2 отличаются.
- Значения Kd для одного корма не универсальны: они зависят от условий рубца (pH, температуры, скорости прохождения).

Поэтому первый дисциплинарный шаг — разметить происхождение каждого числа.

## Вторая дисциплина: модель — это не реальность

CNCPS прогнозирует DMI, MCP, MP, аминокислоты, компоненты молока. Прогнозы имеют погрешность. Хороший рациональщик не «доверяет» модели, а использует её как гипотезу, которую проверяет по факту. Для этого нужны:

- записанные входные параметры;
- фиксированная версия модели/инструмента;
- сравнимые фактические данные;
- правило, при каком отклонении пересчитывать.

## Третья дисциплина: аминокислотный баланс — не опция

Одно из главных преимуществ CNCPS — возможность рассчитывать баланс метаболизируемого протеина и отдельных аминокислот (Lys, Met, иногда His). Если рациональный расчёт останавливается только на MP, теряется смысл модели.

## Четвёртая дисциплина: версии и инструменты — часть модели

CNCPS v6.5 и более ранние версии отличаются уравнениями и константами. Разные программы (CPM Dairy, AMTS, Spartan, BRaNDS) реализуют модель с разными допущениями. Нельзя переносить цифры из одного инструмента в другой, не фиксируя версию.

## Пятая дисциплина: генеративность через ограничения

CNCPS-DPF должен быть обучаемым (новый технолог может освоить рационирование по шагам) и генеративным (позволяет строить новые рационы, добавлять корма, адаптировать под ферму). Генеративность достигается не за счёт «творческой свободы», а за счёт чётких ограничений: каждый паттерн говорит, что можно делать, а что — нет.

# Framework Context — Boundary And Owner Routing

## Граница применения

Этот DPF управляет применением модели CNCPS в рационировании молочного скота. Он не переписывает FPF Core и не заменяет научные источники.

- **In scope:** сбор данных кормов, фракционирование CHO/CP, выбор Kd, расчёт MCP, баланс MP и аминокислот, прогноз DMI, калибровка по факту, версионирование инструментов, добавление новых кормов.
- **Out of scope:** лечебная ветеринария, генетика, экономика фермы в целом, правовые нормы, разработка программного обеспечения.

## Owner routing

- **Научная истина/доказательства:** SoTA-записи (`CS.SOTA.*`), рецензируемые источники, `A.10`.
- **Достоверность прогнозов:** `B.3`, полевая калибровка, `CNCPS.6`.
- **Язык и точность терминов:** `C.2.LS`, `C.28`; если термин угрожает стать новым видом (kind) — precision restoration.
- **Структурная информация:** `C.33`, `C.34` при переходе от модели к архитектуре инструмента.
- **Публикация/обучение:** `E.17`, `E.17.EFP`; учебные материалы хранятся отдельно.
- **Источники и обновления:** `G.2`, `G.11`.

## Intended readers

- Практики-рациональщики молочных ферм.
- Технологи, переходящие с традиционных систем нормирования на CNCPS.
- Преподаватели курса по кормлению КРС.
- Разработчики/консультанты, внедряющие инструменты расчёта рационов.

## Non-use boundary

Не используйте этот DPF:
- как единственное основание для ветеринарных решений;
- для замены лабораторного анализа кормов;
- для переноса чисел между разными версиями модели без проверки;
- как доказательство того, что CNCPS «лучше» других систем вне заявленного применения.

## CNCPS.1 — Feed Fraction Intake and Labeled-Value Discipline

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.FeedFractionIntake@Context`, DPF-local relation record для одного корма.

### CNCPS.1:1 — Problem frame

Используйте этот паттерн, когда в расчёт попадает таблица кормов, и неясно, какие числа — из лабораторного анализа, какие — из справочника, какие — оценочные, и как они разбиты на фракции CNCPS.

Первое полезное действие: создать `CNCPS.FeedFractionIntake@Context` с указанием источника каждого значения, фракциями CHO (CA, CB1, CB2, CC) и CP (PA, PB1, PB2, PB3, PC), неопределённостью и блоком неправомерного использования.

Что идёт не так, если пропустить: рациональщик использует «средние по региону» значения как точные, получает красивый баланс MP, но модель расходится с фактом. Проблема не в модели — в данных.

Что даёт: каждое число становится либо надёжным входом, либо явно помеченным допущением.

### CNCPS.1:2 — Problem

CNCPS требует не средний состав корма, а фракции. Но на практике часто бывает так:

- Лаборатория выдаёт только CP, NDF, ADF, WSC, крахмал.
- Из этих значений рациональщик «выводит» фракции по формулам или берёт из таблицы.
- Разные таблицы дают разные фракции для одного корма.
- Не размечается, какие значения measured, literature, estimated, inherited.

Без разметки модель кажется неточной, хотя на самом деле неточны входные данные.

### CNCPS.1:3 — Forces

| Force | Tension |
|-------|---------|
| Доступность анализов vs требования модели | Лаборатория не всегда даёт нужные фракции. |
| Скорость расчёта vs дисциплина | Разметка источников замедляет, но предотвращает ошибки. |
| Унификация vs локальная вариабельность | Табличные значения усреднены; фермерский корм может отличаться. |
| Простота обучения vs полнота | Новичку хочется сразу считать; опытный сначала размечает. |

### CNCPS.1:4 — Solution

Создайте для каждого корма запись:

```text
CNCPS.FeedFractionIntake@Context:
  feedName:
  feedCode?:
  originTag: measured | literature | estimated | inherited | mixed
  labReportRef?:
  literatureSourceRef?:
  estimationMethodRef?:
  moistureBasis: as-fed | DM
  dryMatterValue: # %
  chofractions:
    CA: # % CHO or % DM (consistent with basis)
    CB1:
    CB2:
    CC:
  cpfractions:
    PA:
    PB1:
    PB2:
    PB3:
    PC:
  kdValues:
    CB1: # %/h
    CB2:
    PB1:
    PB2:
    PB3:
  uncertaintyNotes:
  nonUseBoundary: # what this intake may not justify
  sourceReturnCondition:
```

Затем выполните четыре проверки:

| Проверка | Вопрос | Если нет — действие |
|----------|--------|---------------------|
| Базис | Все значения в одном базисе (as-fed или DM)? | Перевести в DM или as-fed и зафиксировать. |
| Сумма | Суммы CA+CB1+CB2+CC и PA+PB1+PB2+PB3+PC близки к 100% фракции? | Проверить расчёт или лабораторный отчёт. |
| Происхождение | Каждое значение имеет originTag? | Пометить estimated/inherited, пока нет анализа. |
| Локальность | Литературные/наследованные значения применимы к данному корму? | Добавить note о риске. |

### CNCPS.1:5 — Archetypal Grounding

#### Зрелый пример: кукурузный силос

```text
CNCPS.FeedFractionIntake@CornSilageFarmA:
  feedName: Кукурузный силос, Farm A
  originTag: measured
  labReportRef: LabA-2026-06-15
  moistureBasis: DM
  dryMatterValue: 32.0
  chofractions:
    CA: 5.0
    CB1: 28.0
    CB2: 18.0
    CC: 42.0
  cpfractions:
    PA: 2.0
    PB1: 8.0
    PB2: 35.0
    PB3: 35.0
    PC: 5.0
  kdValues:
    CB1: 18.0
    CB2: 6.0
    PB1: 120.0
    PB2: 8.0
    PB3: 2.0
  uncertaintyNotes: NDF лаборатории не разделялся на CB2/CC; CC получен по NDFD; CB1 по крахмалу + WSC
  nonUseBoundary: Не использовать для другой кукурузный силос без переразметки
  sourceReturnCondition: LabA-2026-06-15; при отклонении MUN/DMI пересдать анализ
```

#### До и после ремонта

До:
> «Силос — стандартный, CP 7,5%, NDF 45%. Берём средние фракции из таблицы.»

После:
> «Силос Farm A: DM 32%, CP 7,5% DM, NDF 45% DM. CA/CB1/CB2/CC рассчитаны по лабораторным WSC, крахмалу и NDFD с пометкой estimated для CB2↔CC разделения. Используется только для Farm A.»

### CNCPS.1:6 — Bias-Annotation

Этот паттерн блокирует **табличный дрифт**: табличные или средние значения используются как замерянные. Ремонт: явно пометить originTag и добавить sourceReturnCondition. Scope: локально для CNCPS-DPF; не отменяет требования лабораторного анализа.

### CNCPS.1:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS1-1` | Для каждого корма указан originTag. |
| `CC-CNCPS1-2` | Фракции CHO и CP суммируются корректно в выбранном базисе. |
| `CC-CNCPS1-3` | Kd имеют источник или пометку estimated. |
| `CC-CNCPS1-4` | Указано, для какой фермы/партии корма применима запись. |
| `CC-CNCPS1-5` | Явно записан nonUseBoundary для estimated/inherited значений. |

### CNCPS.1:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| Таблица вместо анализа | Модель считает по чужим данным. | Пометить originTag=literature и откалибровать. |
| Средний корм | Усреднение нескольких партий в одно значение. | Разделить партии или явно указать диапазон. |
| Базисная путаница | Значения as-fed и DM смешаны. | Перевести всё в DM, фиксировать базис. |

### CNCPS.1:9 — Consequences

Польза: входные данные становятся прозрачными и проверяемыми. Цена: дополнительное время на разметку.

### CNCPS.1:10 — Rationale

CNCPS — garbage-in-garbage-out модель. Точность прогноза ограничена качеством входных данных. Разметка origin — минимальная цена за воспроизводимость.

### CNCPS.1:11 — SoTA-Echoing

Fox et al. (2004) и Van Amburgh et al. (2015) подчёркивают, что точность CNCPS зависит от качества описания кормов. Версии v6.5 уточняют фракционирование белков и углеводов; без лабораторных данных модель теряет преимущество перед более простыми системами.

### CNCPS.1:12 — Relations

Использует `CS.ENTITY.191`, `CS.ENTITY.193`, `CS.ENTITY.188`, `CS.ENTITY.200`; `A.10` для claims о точности; `G.2` для источников; `C.2.LS` для разметки базиса и терминов; `CNCPS.2` при переходе к Kd; `CNCPS.6` для калибровки.

## CNCPS.2 — Rumen Fermentation Kinetics and Rate Selection

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.RateSelectionRecord@Context`, DPF-local record для одного корма/группы.

### CNCPS.2:1 — Problem frame

Используйте этот паттерн, когда нужно выбрать скорость деградации (Kd) для фракций CB1, CB2, PB1, PB2, PB3. Частая ошибка — взять Kd из «типовой таблицы» без учёта условий рубца.

Первое полезное действие: зафиксировать параметры рубца (pH, температуру, скорость прохождения) и обосновать выбор Kd.

Что идёт не так: одно и то же зерно в рубце с высоким pH и быстрым прохождением даст другой Kd, чем в рубце с низким pH и медленным прохождением. Игнорирование этого приводит к ошибкам в расчёте MCP и VFA.

### CNCPS.2:2 — Problem

Kd — не константа корма. Она зависит от:

- pH рубца;
- температуры рубца;
- уровня кормления и DMI;
- соотношения концентрата и грубых кормов;
- обработки зерна (валковая/молотковая дробилка, экструдирование, термическая обработка);
- присутствия протозой и их активности.

В таблицах обычно даётся Kd при «стандартных условиях». На ферме условия могут отличаться.

### CNCPS.2:3 — Forces

| Force | Tension |
|-------|---------|
| Простота vs точность | Табличный Kd быстрее, но менее точен. |
| Измеряемость vs доступность | Прямое измерение Kd сложно; чаще используют indirect estimates. |
| Стабильность vs изменчивость | Одна и та же ферма даёт разные Kd в разные периоды лактации. |
| Обучаемость vs глубина | Новичку нужен простой правило; опытный может корректировать. |

### CNCPS.2:4 — Solution

Создайте запись выбора скоростей:

```text
CNCPS.RateSelectionRecord@Context:
  feedName:
  fraction:
  baseKd: # %/h from source
  baseKdSource: table | in-situ | in-vitro | model-default | farm-calibrated
  rumenConditions:
    pH:
    temperature: # °C
    passageRate: # %/h
    concentrate:forageRatio:
  adjustmentRationale:
  adjustedKd:
  uncertainty:
  sourceReturnCondition:
```

Правила корректировки (примеры, не заменяют источник):

| Условие | Влияние на Kd | Пример корректировки |
|---------|---------------|----------------------|
| pH < 6.0 | Снижение ферментации крахмала и белка | Уменьшить Kd CB1/PB1 на 10–20% |
| Высокий уровень кормления (DMI > 3% BW) | Повышение passage rate | Уменьшить эффективную деградацию |
| Мелкое дробление зерна | Повышение Kd CB1 | Увеличить Kd CB1 |
| Экструдирование сои | Повышение RUP | Снизить PB1/PB2, повысить PB3/PC |
| Высокое содержание протозой | Повышение эффективной деградации | Корректировать через модель, если поддерживается |

### CNCPS.2:5 — Archetypal Grounding

#### Пример: ячмень при разных pH

```text
CNCPS.RateSelectionRecord@BarleyHighConcentrate:
  feedName: Ячмень дроблёный
  fraction: CB1
  baseKd: 45.0
  baseKdSource: table
  rumenConditions:
    pH: 5.9
    temperature: 39.0
    passageRate: 6.0
    concentrate:forageRatio: 70:30
  adjustmentRationale: Низкий pH и высокий passage rate снижают эффективную ферментацию крахмала
  adjustedKd: 36.0
  uncertainty: ±15%
  sourceReturnCondition: Вернуться к измерениям pH и DMI; при изменении рациона пересчитать
```

### CNCPS.2:6 — Bias-Annotation

Блокирует **табличный Kd-фетишизм**: табличное значение воспринимается как физическая константа. Ремонт: явно записать условия рубца и обоснование корректировки.

### CNCPS.2:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS2-1` | Для каждого Kd указан источник. |
| `CC-CNCPS2-2` | Условия рубца зафиксированы или явно помечены как «стандартные». |
| `CC-CNCPS2-3` | Корректировки обоснованы ссылкой на условие или литературу. |
| `CC-CNCPS2-4` | Неопределённость Kd указана. |

### CNCPS.2:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| Kd из таблицы без контекста | Модель считает не те условия. | Добавить rumenConditions. |
| Игнорирование pH | Крахмал ферментируется иначе. | Измерить/оценить pH, скорректировать Kd. |
| Перенос Kd между версиями | v6.5 и v7 используют разные уравнения. | Фиксировать версию модели. |

### CNCPS.2:9 — Consequences

Польза: расчёт MCP и энергии рубца становится адаптированным к ферме. Цена: требуется знание условий рубца.

### CNCPS.2:10 — Rationale

Кинетика ферментации — сердце CNCPS. Без учёта условий рубца модель деградирует до статической таблицы.

### CNCPS.2:11 — SoTA-Echoing

CNCPS v6.5 и v7 включают динамические уравнения для pH, passage rate, влияния протозой. Van Amburgh et al. (2015) показывают, что учёт этих факторов улучшает прогноз MCP и MP.

### CNCPS.2:12 — Relations

Использует `CS.ENTITY.189`, `CS.ENTITY.191`, `CS.ENTITY.004`; `CNCPS.1` для входных фракций; `CNCPS.3` для расчёта MCP; `CNCPS.6` для калибровки; `A.10` для claims о физиологии; `G.2`/`G.11` для источников Kd.

### CNCPS.2:End

## CNCPS.3 — Microbial Protein Synthesis and Nitrogen Balance

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.MicrobialProteinBalance@Context`, DPF-local record для одного рациона.

### CNCPS.3:1 — Problem frame

Используйте этот паттерн, когда нужно связать доступную энергию рубца, деградацию протеина и синтез микробного протеина (MCP). Частая ошибка — считать, что «больше протеина = больше MCP».

Первое полезное действие: рассчитать баланс между доступной энергией (FCHO × Kd) и деградируемым протеином (FPROT × Kd), оценить эффективность синтеза MCP.

### CNCPS.3:2 — Problem

Синтез MCP в CNCPS описывается примерно так:

```text
MCP ∝ (FCHO × Kd_CHO + FPROT × Kd_PROT) × эффективность
```

Но на практике:

- Избыток RDP без достаточной энергии приводит к потере азота (высокий MUN).
- Недостаток RDP при высокой энергии ограничивает рост микробов.
- Соотношение NFC:N и источники NFC влияют на состав микробной биомассы.
- Протозои рециркулируют бактерии, снижая выход MCP.

### CNCPS.3:3 — Forces

| Force | Tension |
|-------|---------|
| Энергия рубца vs доступный азот | MCP требует и того, и другого. |
| Быстрая энергия (CA/CB1) vs медленная (CB2/CC) | Разные источники дают разный профиль VFA. |
| RDP vs RUP | Часть протеина должна пройти в кишечник. |
| Экономия vs точность | Дешёвые источники NPN могут исказить баланс. |

### CNCPS.3:4 — Solution

Запишите баланс:

```text
CNCPS.MicrobialProteinBalance@Context:
  rationRef:
  fchoAvailable: # ферментируемые углеводы × Kd
  fprotAvailable: # деградируемый протеин × Kd
  mcpSynthesized: # г/день
  mcpEfficiencySource: # версия уравнения, константа
  rumenNBalance: deficit | adequate | excess
  munPrediction:
  optimizationMoves:
    - if rumenNBalance == excess: reduce RDP or increase fermentable CHO
    - if rumenNBalance == deficit: add soluble true protein or NPN
  sourceReturnCondition:
```

### CNCPS.3:5 — Archetypal Grounding

#### Пример: рацион с высоким соевым шротом

До:
> «Соевый шрот — хороший источник протеина, добавим 3 кг.»

После:
> «Соевый шрот даёт 1,2 кг RDP, но энергия рубца ограничена (FCHO доступно 4,5 кг). Прогноз: RDP избыточен, MUN 18 мг/дл. Оптимизация: снизить шрот на 0,5 кг или добавить крахмалистый концентрат.»

### CNCPS.3:6 — Bias-Annotation

Блокирует **протеиновый фетишизм**: считать, что любой протеин автоматически улучшает питание. Ремонт: связывать протеин с энергией рубца и синтезом MCP.

### CNCPS.3:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS3-1` | Доступная энергия и доступный азот рассчитаны. |
| `CC-CNCPS3-2` | Эффективность синтеза MCP указана и соответствует версии модели. |
| `CC-CNCPS3-3` | Rumen N balance классифицирован. |
| `CC-CNCPS3-4` | Прогноз MUN или аммиака зафиксирован для сверки. |

### CNCPS.3:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| Добавить протеин «на всякий случай» | Избыток RDP, высокий MUN, потери азота. | Связать с FCHO. |
| Использовать NPN без энергии | Неэффективное использование, риск отравления. | Проверить энергию рубца и pH. |
| Игнорировать протозой | Завышенный прогноз MCP. | Использовать версию модели с коррекцией, если доступно. |

### CNCPS.3:9 — Consequences

Польза: рацион оптимизируется по азоту, снижаются потери и загрязнение. Цена: требуется считать не только CP, но и фракции + энергию.

### CNCPS.3:10 — Rationale

MCP — основной источник MP для коровы. Его синтез зависит от баланса энергии и азота. Без этого баланса модель не работает.

### CNCPS.3:11 — SoTA-Echoing

CNCPS v6.5 уточняет расчёт MCP с учётом фракций углеводов и белков, pH, passage rate. Van Amburgh et al. (2015) показывают, что прогноз MP улучшается при точном учёте RDP и энергии рубца.

### CNCPS.3:12 — Relations

Использует `CS.ENTITY.197`, `CS.ENTITY.201`, `CS.ENTITY.191`; `CNCPS.1`, `CNCPS.2`; `CNCPS.4` для перехода к MP; `A.10` для claims; `G.2`/`G.11`.

### CNCPS.3:End

## CNCPS.4 — Metabolizable Protein and Amino-Acid Adequacy

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.ProteinAminoAcidAdequacy@Context`, DPF-local record для одной группы.

### CNCPS.4:1 — Problem frame

Используйте этот паттерн, когда нужно проверить, достаточно ли метаболизируемого протеина (MP) и ключевых аминокислот (Lys, Met, His, иногда Leu, Ile).

Первое полезное действие: рассчитать MP = MCP из рубца + RUP из кормов − потери в кишечнике, затем проверить Lys и Met как % MP и соотношение Lys:Met.

### CNCPS.4:2 — Problem

Традиционные системы останавливаются на сыром протеине. CNCPS позволяет идти глубже — до аминокислот. Но:

- MP зависит от точности MCP и RUP.
- Требования по Lys/Met различаются в зависимости от удоя, стадии лактации, массы.
- Соотношение Lys:Met важнее абсолютных значений.
- Источники RUP (например, мясокостная мука, кукурузный глютен) сильно отличаются по аминокислотному профилю.

### CNCPS.4:3 — Forces

| Force | Tension |
|-------|---------|
| Достаточность MP vs баланс АК | MP может быть достаточно, но Lys — нет. |
| Стоимость vs эффективность | Защищённые аминокислоты дороги; нужен расчёт окупаемости. |
| Удой vs стадия лактации | Требования меняются. |
| Простота vs точность | Многие инструменты скрывают расчёт АК. |

### CNCPS.4:4 — Solution

Запишите:

```text
CNCPS.ProteinAminoAcidAdequacy@Context:
  group:
  mpRequirement: # г/день
  mpSupply: # г/день
  mpBalance: # %
  lysRequirement: # % MP
  lysSupply:
  metRequirement: # % MP
  metSupply:
  lysToMetRatio:
  hisRequirement?:
  hisSupply?:
  limitingAminoAcid:
  correctionOptions:
  sourceReturnCondition:
```

Целевые диапазоны (типовые, зависят от версии модели и группы):

| Параметр | Целевой диапазон | Интерпретация |
|----------|------------------|---------------|
| MP баланс | 100–115% потребности | <100% — дефицит; >120% — возможно избыточно |
| Lys | 6,5–7,2% MP | Первый лимитирующий АК |
| Met | 2,0–2,5% MP | Второй лимитирующий АК |
| Lys:Met | 2,8–3,1:1 | <2,7 или >3,3 — дисбаланс |

### CNCPS.4:5 — Archetypal Grounding

#### Пример: высокоудойная группа

```text
CNCPS.ProteinAminoAcidAdequacy@HighYielders:
  group: Дойные 35–150 DIM, удой 40 кг
  mpRequirement: 3050
  mpSupply: 3120
  mpBalance: 102%
  lysRequirement: 6.8
  lysSupply: 6.4
  metRequirement: 2.2
  metSupply: 2.1
  lysToMetRatio: 3.05
  limitingAminoAcid: Lys
  correctionOptions:
    - add protected Lys source
    - increase soybean meal or replace part with blood meal
  sourceReturnCondition: Пересчитать при изменении удоя >2 кг или замене кормов
```

### CNCPS.4:6 — Bias-Annotation

Блокирует **MP-единственность**: остановка только на MP без проверки аминокислот. Ремонт: всегда проверять Lys, Met, Lys:Met.

### CNCPS.4:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS4-1` | MP рассчитан из MCP + RUP − потери. |
| `CC-CNCPS4-2` | Lys и Met проверены как % MP. |
| `CC-CNCPS4-3` | Lys:Met в целевом диапазоне или помечен как риск. |
| `CC-CNCPS4-4` | Лимитирующий АК назван. |

### CNCPS.4:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| Только MP | Аминокислотный дисбаланс маскируется. | Добавить Lys/Met. |
| Игнорирование защиты | Незащищённые добавки разрушаются в рубце. | Учитывать intestinal availability. |
| Фиксированные нормы | Требования меняются с лактацией. | Привязать к группе/стадии. |

### CNCPS.4:9 — Consequences

Польза: рацион точнее соответствует потребностям; возможна точная коррекция. Цена: требуется больше данных и расчётов.

### CNCPS.4:10 — Rationale

Одно из главных преимуществ CNCPS — аминокислотный баланс. Без него теряется смысл перехода от CP к MP.

### CNCPS.4:11 — SoTA-Echoing

CNCPS v6.5 включает требования по Lys, Met, His. Исследования показывают, что оптимизация Lys:Met улучшает содержание белка молока и эффективность использования кормового протеина.

### CNCPS.4:12 — Relations

Использует `CS.ENTITY.197`, `CS.ENTITY.193`, `CS.ENTITY.191`; `CNCPS.3`; `CNCPS.6` для калибровки; `A.10` для claims; `G.2`/`G.11`.

## CNCPS.5 — Animal Description and Intake Prediction

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.AnimalIntakeDescription@Context`, DPF-local record для одной группы.

### CNCPS.5:1 — Problem frame

Используйте этот паттерн, когда нужно описать животных и спрогнозировать сухое вещество (DMI). CNCPS чувствительна к ошибкам в описании животного: масса, удой, стадия лактации, номер лактации, BCS.

### CNCPS.5:2 — Problem

DMI влияет на:

- passage rate (а значит, на Kd);
- общее потребление питательных веществ;
- разбавление рациона в рубце;
- точность прогноза продуктивности.

Если DMI занижен, модель предскажет дефицит питательных веществ; если завышен — избыток.

### CNCPS.5:3 — Forces

| Force | Tension |
|-------|---------|
| Индивидуальная вариабельность vs групповой расчёт | CNCPS обычно применяется к группе, а не к отдельной корове. |
| Фактический DMI vs прогнозный | Факт редко измеряется точно. |
| Простота vs полнота | Можно использовать «среднюю корову», но теряется точность. |
| Стадия лактации | DMI и требования сильно меняются. |

### CNCPS.5:4 — Solution

Запишите описание группы:

```text
CNCPS.AnimalIntakeDescription@Context:
  group:
  averageBW: # кг
  averageMilkYield: # кг/день
  milkFat: # %
  milkProtein: # %
  daysInMilk:
  parity: # 1, 2, 3+
  bcs: # body condition score
  pregnancyDays?:
  environmentalConditions:
    temperature: # °C
    humidity: # %
    heatStressIndex:
  dmiMethod: measured | predicted | historical
  dmiValue: # кг/день
  dmiPredictionEquation: # NRC, CNCPS, farm-specific
  sourceReturnCondition:
```

Если DMI не измерен, укажите метод прогноза и его неопределённость. После внедрения рациона измеряйте остатки и корректируйте.

### CNCPS.5:5 — Archetypal Grounding

#### Пример: группа ранней лактации

```text
CNCPS.AnimalIntakeDescription@EarlyLactation:
  group: Дойные 14–60 DIM, 2-я+ лактация
  averageBW: 650
  averageMilkYield: 42
  milkFat: 4.0
  milkProtein: 3.3
  daysInMilk: 35
  parity: 2+
  bcs: 3.0
  environmentalConditions:
    temperature: 22
    humidity: 70
    heatStressIndex: moderate

  dmiValue: 24.5
  dmiPredictionEquation: CNCPS v6.5
  sourceReturnCondition: Измерить остатки через 7 дней; при отклонении >5% скорректировать
```

### CNCPS.5:6 — Bias-Annotation

Блокирует **«среднюю корову»**: использование одного усреднённого описания без учёта группы и стадии. Ремонт: явно задокументировать группу и метод DMI.

### CNCPS.5:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS5-1` | Масса, удой, жир, белок, DIM, лактация указаны. |
| `CC-CNCPS5-2` | DMI — measured, predicted или historical. |
| `CC-CNCPS5-3` | Если predicted — указано уравнение. |
| `CC-CNCPS5-4` | Условия окружающей среды учтены или помечены как умеренные. |

### CNCPS.5:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| Одна «средняя корова» | Потеря вариабельности группы. | Разделить на подгруппы. |
| DMI из головы | Прогноз не сверяется с остатками. | Измерить остатки. |
| Игнорирование жара | Снижение DMI не учтено. | Добавить heat stress correction. |

### CNCPS.5:9 — Consequences

Польза: вход животного становится точным; прогнозы MP и энергии надёжнее. Цена: требуется актуальная информация по группе.

### CNCPS.5:10 — Rationale

CNCPS — модель животного + корма + рубца. Без точного описания животного корректная часть модели не спасёт.

### CNCPS.5:11 — SoTA-Echoing

CNCPS v6.5 включает уравнения для DMI с учётом массы, удоя, стадии лактации, BCS, окружающей среды. Фактический DMI остаётся одним из главных источников неточности.

### CNCPS.5:12 — Relations

Использует `CS.ENTITY.191`; `CNCPS.1`–`CNCPS.4`; `CNCPS.6` для калибровки; `A.10`; `G.2`/`G.11`.

### CNCPS.5:End

## CNCPS.6 — Model-to-Reality Calibration

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.CalibrationRecord@Context`, DPF-local record для одной фермы/группы.

### CNCPS.6:1 — Problem frame

Используйте этот паттерн, когда прогнозы CNCPS расходятся с фактической продуктивностью. Цель — не «доказать, что модель плоха», а найти, какой входной параметр или допущение ведёт к отклонению.

### CNCPS.6:2 — Problem

Даже при правильных входных данных модель не идеальна. Отклонения могут быть из-за:

- ошибок в описании кормов;
- неверных Kd;
- неточного DMI;
- индивидуальной вариабельности животных;
- ошибок измерения на ферме (молоко, масса, остатки);
- различий между версиями модели.

### CNCPS.6:3 — Forces

| Force | Tension |
|-------|---------|
| Модель vs факт | Модель даёт гипотезу, факт — источник правды. |
| Точность vs стоимость измерений | Идеальные измерения дороги; нужен баланс. |
| Быстрая коррекция vs системная | Можно подгонять числа, но лучше найти причину. |
| Группа vs индивид | Групповой расчёт не ловит отдельных коров. |

### CNCPS.6:4 — Solution

Создайте запись калибровки:

```text
CNCPS.CalibrationRecord@Context:
  farm/group:
  period:
  modelVersion:
  toolVersion:
  predictions:
    dmi:
    milkYield:
    milkFat:
    milkProtein:
    mun:
  actuals:
    dmi:
    milkYield:
    milkFat:
    milkProtein:
    mun:
  deviations:
  diagnosticHypotheses:
    - hypothesis:
      likelyInputCause:
      evidence:
      correction:
  calibrationAction:
  nextReviewDate:
  sourceReturnCondition:
```

Допустимые отклонения (ориентировочно):

| Показатель | Прогноз | Допустимое отклонение | Действие при превышении |
|------------|---------|----------------------|-------------------------|
| DMI | кг/день | ±5% | Проверить корма, остатки, DMI. |
| Молочный жир | % | ±0.3% | Проверить NDF, CB2/CC, pH. |
| Молочный белок | % | ±0.15% | Проверить MP, Lys, Met. |
| MUN | мг/дл | 12–16 | Проверить RDP/энергию. |

### CNCPS.6:5 — Archetypal Grounding

#### Пример: расхождение по белку молока

```text
CNCPS.CalibrationRecord@FarmB:
  farm/group: Farm B, высокоудойные
  period: 2026-05-01 — 2026-05-31
  modelVersion: CNCPS v6.5
  toolVersion: AMTS.Cattle.Professional 4.7
  predictions:
    milkProtein: 3.25
  actuals:
    milkProtein: 3.05
  deviations: -0.20% (превышает порог)
  diagnosticHypotheses:
    - hypothesis: Занижен Lys
      likelyInputCause: Соевый шрот переценён по PB2
      evidence: MUN норма, MP 104%, Lys 6.1%
      correction: Переразметить соевый шрот, рассмотреть защищённый Lys
  calibrationAction: Пересчитать рацион с обновлёнными фракциями сои
  nextReviewDate: 2026-06-15
  sourceReturnCondition: Сверить с новым анализом соевого шрота
```

### CNCPS.6:6 — Bias-Annotation

Блокирует **модельную враждебность**: склонность обвинять модель вместо проверки входов. Ремонт: систематически проверять входные параметры, прежде чем менять модель.

### CNCPS.6:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS6-1` | Зафиксированы версия модели и инструмента. |
| `CC-CNCPS6-2` | Есть прогнозы и факт по сопоставимым показателям. |
| `CC-CNCPS6-3` | Отклонения оценены относительно порогов. |
| `CC-CNCPS6-4` | Выдвинута гипотеза о причине отклонения. |
| `CC-CNCPS6-5` | Запланирована повторная проверка. |

### CNCPS.6:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| «Модель не работает» | Отказ от системного анализа. | Проверить входы по CNCPS.1–CNCPS.5. |
| Подгонка DMI | Меняют DMI до совпадения, без объяснения. | Найти причину (остатки, корм, группа). |
| Одна сверка | Разовая сверка не даёт тренда. | Встроить регулярный review. |

### CNCPS.6:9 — Consequences

Польза: модель становится надёжным инструментом именно для данной фермы. Цена: требуется регулярный сбор фактических данных.

### CNCPS.6:10 — Rationale

CNCPS — не оракул. Её ценность в том, что она даёт структурированную гипотезу, которую можно проверить и улучшить.

### CNCPS.6:11 — SoTA-Echoing

Валидационные исследования CNCPS показывают R² 0.85–0.95 для прогноза продуктивности при хороших входных данных. При плохих данных точность падает.

### CNCPS.6:12 — Relations

Использует `CS.ENTITY.191`; `CNCPS.1`–`CNCPS.5`; `A.10`, `B.3`; `G.2`/`G.11`.

### CNCPS.6:End

## CNCPS.7 — Tool Implementation and Version Control

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.ToolVersionRecord@Context`, DPF-local record для одного инструмента/проекта.

### CNCPS.7:1 — Problem frame

Используйте этот паттерн, когда выбираете или меняете программу для расчёта по CNCPS. Разные инструменты (CPM Dairy, AMTS, Spartan, BRaNDS) могут давать разные результаты из-за разных реализаций модели, библиотек кормов и умолчаний.

### CNCPS.7:2 — Problem

- Версия модели (v5, v6.1, v6.5, v7) влияет на результаты.
- Библиотека кормов в программе может отличаться от ваших анализов.
- Настройки по умолчанию могут быть неприменимы к вашей ферме.
- Обновления ПО меняют расчёты, но пользователь может не заметить.

### CNCPS.7:3 — Forces

| Force | Tension |
|-------|---------|
| Удобство vs контроль | Готовые библиотеки экономят время, но скрывают допущения. |
| Стоимость vs функциональность | Дорогие программы дают больше настроек. |
| Обновления vs стабильность | Новая версия может сломать сравнимость. |
| Обучаемость vs мощность | Сложные инструменты сложнее осваивать. |

### CNCPS.7:4 — Solution

Запишите:

```text
CNCPS.ToolVersionRecord@Context:
  toolName:
  toolVersion:
  cncpsModelVersion:
  feedLibrarySource:
  defaultSettings:
    animalDefaults:
    feedDefaults:
    environmentDefaults:
  customizations:
  exportFormat:
  backupPolicy:
  updatePolicy:
  sourceReturnCondition:
```

### CNCPS.7:5 — Archetypal Grounding

#### Пример: миграция между инструментами

До:
> «У нас был CPM Dairy, перешли на AMTS — теперь цифры другие, но это нормально.»

После:
> «CNCPS v6.5, AMTS 4.7. Библиотека кормов заменена на фермерскую. Параметры животных перенесены 1:1. Первый месяц — параллельный расчёт в CPM Dairy 3.0.8 для сверки отклонений.»

### CNCPS.7:6 — Bias-Annotation

Блокирует **«программа = модель»**: отождествление выхода конкретного инструмента с истиной. Ремонт: фиксировать версию и настройки.

### CNCPS.7:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS7-1` | Инструмент и версия модели зафиксированы. |
| `CC-CNCPS7-2` | Источник библиотеки кормов указан. |
| `CC-CNCPS7-3` | Настройки по умолчанию проверены и скорректированы. |
| `CC-CNCPS7-4` | Политика обновлений и резервного копирования задана. |

### CNCPS.7:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| Разные версии без пометок | Результаты несопоставимы. | Фиксировать версию в каждом расчёте. |
| Библиотека вместо анализа | «Средний» корм заменяет фермерский. | Перезаписывать библиотечные значения. |
| Автообновление без проверки | Изменяются результаты рационов. | Тестировать после обновления. |

### CNCPS.7:9 — Consequences

Польза: результаты воспроизводимы и сопоставимы. Цена: требуется дисциплина версионирования.

### CNCPS.7:10 — Rationale

Модель — это уравнения и константы. Инструмент — их интерпретация. Нельзя путать одно с другим.

### CNCPS.7:11 — SoTA-Echoing

CNCPS v6.5 — последняя широко распространённая версия; v7 развивается в исследовательских кругах. Программные реализации могут отставать или адаптировать модель под рынок.

### CNCPS.7:12 — Relations

Использует `CS.ENTITY.191`; `CNCPS.1`–`CNCPS.6`; `C.33`, `C.34` при архитектуре инструмента; `E.17` для публикации; `G.2`/`G.11`.

### CNCPS.7:End

## CNCPS.8 — Extension to New Feeds and Local Conditions

> **Type:** DPF pattern body  
> **Primary EntityOfConcern:** `CNCPS.FeedExtensionRecord@Context`, DPF-local record для одного нового корма или условия.

### CNCPS.8:1 — Problem frame

Используйте этот паттерн, когда в рацион нужно включить корм, отсутствующий в стандартной библиотеке, или когда условия фермы сильно отличаются от «стандартных» (климат, хранение, обработка).

### CNCPS.8:2 — Problem

Новые/локальные корма часто добавляют по аналогии: «это как соевый шрот» или «это как кукурузный силос». Но фракционирование может отличаться из-за:

- сорта/гибрида;
- климата выращивания;
- стадии уборки;
- способа хранения и обработки;
- загрязнения/примесей.

### CNCPS.8:3 — Forces

| Force | Tension |
|-------|---------|
| Доступность корма vs точность | Новый корм дешевле, но данных мало. |
| Скорость внедрения vs безопасность | Хочется быстро включить; риск ошибки. |
| Локальные знания vs общие таблицы | Фермерские данные точнее усреднений. |
| Обучаемость vs экспертность | Новичок не может оценить новый корм. |

### CNCPS.8:4 — Solution

Запишите:

```text
CNCPS.FeedExtensionRecord@Context:
  feedName:
  localConditions:
    region:
    climate:
    harvestStage:
    storage:
    processing:
  availableAnalyses:
    - parameter:
      value:
      origin: measured | literature | estimated
  fractionEstimationMethod:
  validationPlan:
    - farm:
      inclusionRate:
      monitoringMetrics:
      duration:
  riskNotes:
  sourceReturnCondition:
```

### CNCPS.8:5 — Archetypal Grounding

#### Пример: местный жмых подсолнечника

```text
CNCPS.FeedExtensionRecord@LocalSunflowerMeal:
  feedName: Подсолнечный жмох, местный
  localConditions:
    region: Центральный регион
    climate: умеренно-континентальный
    harvestStage: полная спелость
    storage: силосный ров
    processing: механическое отжим
  availableAnalyses:
    - parameter: CP
      value: 38.0
      origin: measured
    - parameter: NDF
      value: 35.0
      origin: measured
    - parameter: PB1/PB2/PB3
      value: "10/35/35"
      origin: estimated from literature
  fractionEstimationMethod: NDF + CP + literature proportions
  validationPlan:
    - farm: Farm C
      inclusionRate: 8% DM
      monitoringMetrics: [DMI, milk yield, milk protein, MUN]
      duration: 21 days
  riskNotes: Высокое содержание ADIN/PC возможно при перегреве
  sourceReturnCondition: После валидации обновить originTag на measured
```

### CNCPS.8:6 — Bias-Annotation

Блокирует **аналоговый дрифт**: новый корм классифицируется по сходству, а не по анализу. Ремонт: собрать минимальный набор анализов и зафиксировать план валидации.

### CNCPS.8:7 — Conformance Checklist

| Check | Passing condition |
|-------|-------------------|
| `CC-CNCPS8-1` | Новый корм не классифицирован только по аналогии. |
| `CC-CNCPS8-2` | Доступные анализы и их происхождение указаны. |
| `CC-CNCPS8-3` | Метод оценки фракций задокументирован. |
| `CC-CNCPS8-4` | Есть план валидации на ферме. |

### CNCPS.8:8 — Common Anti-Patterns

| Anti-pattern | Что ломается | Ремонт |
|--------------|--------------|--------|
| «Это как соя» | Неверное фракционирование. | Сделать анализ или взять литературу по конкретному корму. |
| Без плана мониторинга | Риск незамечен. | Задать метрики и срок. |
| Игнорирование обработки | Термическая обработка меняет RUP. | Учесть processing. |

### CNCPS.8:9 — Consequences

Польза: модель расширяется на локальные условия без потери точности. Цена: требуется валидация.

### CNCPS.8:10 — Rationale

CNCPS применима не только к стандартным кормам. Но расширение требует дисциплины, а не интуиции.

### CNCPS.8:11 — SoTA-Echoing

CNCPS развивается в сторону включения местных кормов и динамических условий. Валидация новых кормов — стандартная практика в исследованиях.

### CNCPS.8:12 — Relations

Использует `CS.ENTITY.191`; `CNCPS.1`–`CNCPS.7`; `A.10`; `G.2`/`G.11`.

### CNCPS.8:End

# Acceptance Cases

Acceptance-кейсы проверяют, что набор паттернов работает на разных типах ферм, кормов и задач. Они не являются полевыми испытаниями; они — сценарии, по которым можно оценить применимость DPF.

## Case 1 — Типовая молочная ферма с зерновыми и силосом

**Контекст:** Ферма 500 голов, рацион: кукурузный силос, люцерновое сено, кукуруза, соевый шрот, минеральная добавка.  
**Вход:** `CNCPS.5` → `CNCPS.1` → `CNCPS.2` → `CNCPS.3` → `CNCPS.4` → `CNCPS.6`.  
**Проходной критерий:** Прогноз DMI отклоняется от факта не более чем на 5%; MP баланс 100–115%; Lys:Met в диапазоне 2,7–3,3; MUN 12–16.  
**Что проверяет:** Базовая последовательность паттернов даёт рабочий рацион.

## Case 2 — Ферма с местным жмыхом подсолнечника

**Контекст:** Ферма использует местный подсолнечный жмох, отсутствующий в библиотеке.  
**Вход:** `CNCPS.8` → `CNCPS.1` → `CNCPS.2` → `CNCPS.6`.  
**Проходной критерий:** Новый корм размечен с originTag; фракции обоснованы; есть план валидации на 21 день.  
**Что проверяет:** DPF позволяет расширять модель на новые корма.

## Case 3 — Высокоудойная группа с дефицитом Lys

**Контекст:** Удой 45 кг, MP 104%, Lys 6,1%, Met 2,1%, Lys:Met 2,9.  
**Вход:** `CNCPS.4` → `CNCPS.1` (перепроверить сою) → `CNCPS.3` → `CNCPS.6`.  
**Проходной критерий:** Выявлен лимитирующий АК; предложена коррекция; прогноз изменения белка молока зафиксирован.  
**Что проверяет:** DPF ловит аминокислотные дисбалансы.

## Case 4 — Обучение технолога с нуля

**Контекст:** Технолог знает традиционные нормы, но не знает CNCPS.  
**Вход:** `Preface` → `CNCPS.1` → `CNCPS.2` → `CNCPS.3` → `CNCPS.4` → `CNCPS.5` → упражнение.  
**Проходной критерий:** Технолог может самостоятельно разметить один корм, выбрать Kd с обоснованием, объяснить разницу между CP и MP.  
**Что проверяет:** Обучаемость (критичная характеристика ArchGate).

## Case 5 — Миграция с CPM Dairy на AMTS

**Контекст:** Ферма меняет программу расчёта.  
**Вход:** `CNCPS.7` → `CNCPS.1` (проверка библиотеки) → `CNCPS.6` (параллельный расчёт).  
**Проходной критерий:** Версии модели и инструментов зафиксированы; отклонения между программами объяснены; первый месяц — параллельный контроль.  
**Что проверяет:** DPF обеспечивает версионирование и сопоставимость.

## Case 6 — Составление рациона для новой группы

**Контекст:** На ферме появляется группа ранней лактации; нужен новый рацион.  
**Вход:** `CNCPS.5` → `CNCPS.1` → `CNCPS.2` → `CNCPS.3` → `CNCPS.4` → `CNCPS.6`.  
**Проходной критерий:** Рацион создан с явными допущениями; прогнозы записаны; план сверки через 14 дней.  
**Что проверяет:** Генеративность (критичная характеристика ArchGate).

Этот DPF специализирует Core-отношения:

- `A.6.3.NAR` — структура модели может быть изложена в виде обучающего маршрута (учебные материалы — в отдельных файлах);
- `A.10` — claims о точности, физиологии, требованиях по аминокислотам;
- `B.3` — достоверность прогнозов после калибровки;
- `C.2.LS` — языковая дисциплина терминов (CP, RDP, RUP, MP, MCP и т.д.);
- `C.28` — причинные заявления о влиянии кормов на продуктивность;
- `C.33`/`C.34` — если DPF используется для проектирования инструмента;
- `E.17`/`E.17.EFP` — публикация/обучение;
- `G.2`/`G.11` — источники и refresh.

## Source reuse

- CS.ENTITY.191 — базовая сущность CNCPS; DPF добавляет практические паттерны.
- CS.ENTITY.004, CS.ENTITY.188, CS.ENTITY.189, CS.ENTITY.193, CS.ENTITY.197, CS.ENTITY.199, CS.ENTITY.200, CS.ENTITY.201 — связанные сущности, упомянутые в паттернах.
- CS.METHOD.006 — метод оценки кормления на ферме; DPF дополняет его моделью CNCPS.
- CS.SOTA.* — evidence base для claims.

## Generated-carrier note

Этот DPF — generated/structured publication carrier. Он не является исходным научным текстом, но опирается на него. При публикации или обучении фиксируется edition ref и dependencies.

## Teaching publication-carrier relation

Учебные материалы (слайды, упражнения, семинары) должны храниться отдельно и ссылаться на этот DPF. В DPF остаются только паттерны, relation-записи и acceptance-кейсы.

## Ethics routing

CNCPS-DPF не даёт ветеринарных рекомендаций. Если рацион влияет на здоровье животных (кетоз, ацидоз, гипокальциемия), маршрутизация идёт к ветеринарным методам и специалистам (`CS.METHOD.001`, `CS.METHOD.002` и др.).

# Refresh Route

## Что обновляется

| Источник изменения | Куда возвращаться | Что делать |
|--------------------|-------------------|------------|
| Новая версия CNCPS | `G.2`, `G.11`, CS.ENTITY.191 | Проверить, какие уравнения изменились; пересчитать critical кейсы. |
| Новые данные кормов | `CNCPS.1`, CS.ENTITY.191 | Обновить fraction intake records; пересчитать рационы. |
| Изменение условий фермы (жара, новый корм) | `CNCPS.2`, `CNCPS.5`, `CNCPS.6` | Пересмотреть Kd, DMI, calibration records. |
| Новые исследования по Lys/Met | `CNCPS.4`, `CS.SOTA.*` | Обновить целевые диапазоны, если изменились. |
| Обновление программы | `CNCPS.7` | Проверить совместимость, провести parallel run. |
| Пилот/обратная связь от практиков | `Acceptance cases` | Добавить новый кейс или уточнить существующий. |

## Минимальный refresh-цикл

1. **Ежемесячно:** проверить актуальность лабораторных анализов (`CNCPS.1`).
2. **Ежеквартально:** сверить прогнозы с фактом (`CNCPS.6`).
3. **При изменении версии модели/инструмента:** parallel run и аудит отклонений (`CNCPS.7`).
4. **При появлении нового корма:** заполнить `CNCPS.8` и провести валидацию.
5. **Ежегодно:** пересмотреть acceptance cases и обновить edition ref DPF.

## Source-return condition

Всякий раз, когда число из CNCPS-DPF становится основанием для покупки корма, изменения рациона или ветеринарного решения, возвращайтесь к:

- исходному лабораторному анализу (`CNCPS.1`);
- версии модели и инструмента (`CNCPS.7`);
- записи калибровки (`CNCPS.6`);
- соответствующей SoTA-записи или научному источнику.

# Integration Gate Record

> **Status:** пройден

> **Owner Role:** практик-рациональщик / технолог молочной фермы; методологическая поддержка — PACK-cattle-science + DS-cattle-course

## 1. Service Clause (обещание)

CNCPS-DPF обещает практикам-рациональщикам структурированный, повторяемый и проверяемый способ применения модели CNCPS при формировании и корректировке рационов молочного скота.

- **Триггер:** нужно составить/проверить рацион на базе CNCPS, калибровать модель по факту, обучить технолога или добавить новый корм.
- **Входы:** анализы кормов, описание группы животных, условия фермы, версия инструмента.
- **Выходы:** размеченный рационный кейс, calibration note, learning route или feed extension record.
- **Время отклика:** один рацион — от 30 минут до 4 часов в зависимости от качества данных.
- **Инвариант:** каждое число имеет originTag; каждый прогноз — проверяемый по факту.
- **Режим отказа:** если данных недостаточно, DPF требует явной пометки estimated и плана валидации, а не даёт «красивый» результат.

## 2. Usage Scenarios (сценарии использования)

### Scenario A — Рациональщик составляет рацион

**Кто:** технолог/зоотехник фермы.  
**Когда:** плановая смена рациона или новая группа.  
**Зачем:** получить сбалансированный рацион с учётом MP и аминокислот.  
**Что делает с результатом:** передаёт в производство, планирует закупку кормов, ставит задачу сверки через 14 дней.

### Scenario B — Консультант калибрует модель по факту

**Кто:** внешний консультант по кормлению.  
**Когда:** прогнозы модели расходятся с фактической продуктивностью.  
**Зачем:** найти, какой входной параметр врёт.  
**Что делает с результатом:** корректирует разметку кормов/Kd/DMI, обновляет рацион, докладывает пилоту.

### Scenario C — Преподаватель обучает технолога

**Кто:** преподаватель курса DS-cattle-course.  
**Когда:** модуль по CNCPS.  
**Зачем:** научить переходить от CP/NDF к фракциям и аминокислотам.  
**Что делает с результатом:** использует паттерны как скелет занятия; учебные материалы хранятся отдельно.

## 3. Role (роль в системе)

**Kind:** Domain Principle Framework (`U.Method` в терминах FPF).  
**Owner Role:** PACK-cattle-science как хранитель методологии; DS-cattle-course как downstream-потребитель для обучения; практик — исполнитель.

**Обязанности DPF:**
- давать повторяющиеся паттерны для типовых ситуаций;
- фиксировать границы применения и owner routing;

**Полномочия DPF:**
- требовать разметку originTag для данных;

**Связи:**
